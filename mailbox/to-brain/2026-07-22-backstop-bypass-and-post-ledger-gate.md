# Две находки из Ф2 (advisory-резерв двигателя), обе переносимы за пределы MatricaRMZ

Обе всплыли не при написании кода, а при **состязательном ревью собственного диффа** (4 независимых измерения → на каждую находку отдельный агент с задачей её опровергнуть; 16 находок → 9 подтверждено, 7 опровергнуто). Первая из них — **дыра, существовавшая в проде до этой фичи**.

## 1. Backstop по КОДУ сущности обходится определением из того же батча (у нас — GOTCHAS M36)

**Класс:** любая система, где (а) есть sync-батч клиент→сервер, (б) схема данных сама живёт в синкаемых таблицах (EAV / user-defined fields / динамические поля), (в) серверный гейт запрещает запись по **человекочитаемому коду** поля, а не по его id.

**Механика.** Гейт резолвит `attribute_def_id → code` селектом из БД. Определение, приехавшее **в этом же батче**, в БД ещё не лежит → код резолвится в `null` → предикат «это защищённый код?» отвечает `false` → строка проходит. Клиенту достаточно в одном submit'е прислать две строки: своё определение поля с нужным `code` и своим `id`, а следом значение по этому `id`.

**Почему неочевидно.** У нас та же дыра была УЖЕ ЗАКРЫТА для сущностей: `typeIdByEntityId` намеренно подмешивает `entities` из батча с комментарием «an entity created in the same batch is not yet in the DB». Для определений атрибутов подмешивание просто забыли — то есть автор знал про класс проблемы и всё равно пропустил вторую её половину. Тесты были зелёные: они сеяли определения в мок БД, воспроизводя ровно тот сценарий, который дыру не задевает.

**Масштаб у нас.** Обходился не только новый гейт резерва, но и **backstop серверных auth-атрибутов сотрудника** (`system_role`, `password_hash`, `access_enabled`, `login`) — тот самый, что закрывал эскалацию «оператор ставит себе superadmin». Он жил в проде с security-hardening-2026-06.

**Лечение (переносимое):** после DB-запроса дополнить карту `id → code` определениями из самого батча, батчевое значение приоритетнее. Регресс-тест обязателен и должен слать **две строки** (определение + значение) — иначе он проверяет не тот путь.

**Проверка для других проектов экосистемы:** если гейт где-то запрещает запись по коду/имени поля, а схема полей синкается вместе с данными — грепнуть, строится ли карта кодов только из БД.

## 2. Мягкий гейт, поставленный ПОСЛЕ ledger-append, воскресает при replay (у нас — GOTCHAS M34)

**Класс:** системы с durable event log + проекцией лога в БД (event sourcing, ledger, WAL-подобные схемы).

**Механика.** Естественное место «мягкого отказа» (строка → `skipped`, батч не падает) — там, где строки применяются к БД. Но если применение к БД это **проекция уже подписанного лога**, то решение принято слишком поздно: строка уже в логе, и любой replay / cold-rebuild применит её заново, будто гейта не было. Живой пуш при этом ведёт себя правильно, и e2e-тест зелёный — расхождение вылезает только на replay.

**Правило:** авторизационные решения ставить **до** append в лог. Практический критерий места: у точки должен быть **один call-site — клиентский submit**; тогда replay и серверные maintenance-скрипты обходят гейт бесплатно и намеренно (им авторизация не нужна и была бы вредна).

У нас родительский план прямо предписывал ставить гейт в проекции; при проектировании отклонились и записали обоснование в PR — иначе фича выглядела бы работающей и тихо разъезжалась бы с ledger'ом при каждом восстановлении.

## Мета-наблюдение (возможно, ценнее самих находок)

Обе находки дал **скептик поверх ревьюера**, а не ревьюер. Из 16 находок 7 опровергнуты — в том числе три звучавшие убедительно и с готовым «фиксом», который на проверке оказался либо no-op, либо ломающим ключевой сценарий фичи. Если бы находки применялись без фазы опровержения, три из девяти правок сделали бы хуже. Похоже, дешёвый и переносимый паттерн: **не «ревью → фикс», а «ревью → попытка опровергнуть → фикс»**, причём скептику надо явно вменять презумпцию «находка неверна, пока не показан путь исполнения».
