## TL;DR

Прогнал полноценный аудит безопасности приложения **одним Workflow-скриптом**, который сам себя проверяет на ложные срабатывания: карта поверхностей → 11 параллельных ревьюеров (по одному на вектор атаки + red-team-линза) → **состязательная проверка КАЖДОЙ находки отдельным скептиком, которому велено опровергнуть** → синтез в план по критичности. Результат: 51 сырая находка → 46 подтверждённых (0 опровергнуто, но критичность/эксплуатируемость скорректированы верификаторами), включая 2 critical, которые реально открывали всю БД. Из 46 за сессию закрыл и **раскатал на прод** 2 critical + 4 high + сессионную гигиену (3 PR + объединённый релиз).

## Как устроено у меня

- **Pipeline, не barrier:** `pipeline(DIMENSIONS, review, verify)` — каждая находка уходит на верификацию, как только её измерение готово (dimension «bugs» верифицируется, пока «perf» ещё ревьюится). Нулевой простой.
- **Измерения = векторы атаки**, не файлы: electron-hardening/IPC, auth/session, authz-IDOR, injection/traversal/SSRF, sync-ledger-crypto, secrets, local-data-at-rest (stolen-laptop), updater-integrity, network/TLS/DB-reachability, exfil/audit/web-admin, + отдельная red-team-линза «как украсть данные end-to-end, игнорируя категории».
- **Состязательный верификатор по умолчанию скептик:** «попробуй ОПРОВЕРГНУТЬ; подтверждай только с цитатой кода; если есть смягчающий контроль / путь недостижим / authz уже режет — refuted/partial». Это и отсеивает шум, и даёт готовый `recommendedFix` + скорректированную severity.
- **Статический режим:** агентам явно запрещено бить прод/запускать эксплойты — только чтение кода/конфигов. Threat models (a–f) переданы в преамбулу, каждая находка маппится на конкретную модель.
- **Грабля харнесса:** последние verify-агенты упали на **session-limit**; их verdict=null → выпали и из confirmed, и из refuted (51→46, недостача 5 = ровно один не-доверифицированный dimension). Добрал руками. Урок: при фан-ауте закладывать, что хвост агентов может умереть на лимите — возвращать и «сырые» находки, не только confirmed, чтобы ничего не потерять.

## Почему переносимо

Любой проект с аутентификацией/синхронизацией/десктоп-клиентом (GONBA, setka — если есть auth/API) может прогнать тот же скрипт, заменив только список путей в преамбуле и dimension-focus. Паттерн «fan-out по векторам → adversarial refute-verify → synthesize» — не про MatricaRMZ, а про любой security/quality-аудит, где ценнее всего отсечь false positives. Это конкретное приложение quality-паттернов из описания Workflow (adversarial verify + perspective-diverse + red-team lens) к безопасности.
