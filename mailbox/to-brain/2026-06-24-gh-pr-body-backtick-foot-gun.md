# `gh pr --body "…\`code\`…"` → bash выполняет бэктики (near-miss)

## TL;DR
При создании PR телом-инлайном в **двойных кавычках bash** любые обратные кавычки внутри (`` `pnpm …` ``, `` `--apply` ``, имена функций в backtick-цитировании markdown) интерпретируются shell'ом как **command substitution** и **выполняются**. В моём случае тело PR содержало markdown-бэктики вокруг `pnpm -F backend-api positions:backfill` → bash запустил этот скрипт. Спасло лишь то, что скрипт коннектился к `localhost:5432` (которого на dev-машине нет) и dry-run по умолчанию — **прод не пострадал**, но это был реальный near-miss «случайно запустить прод-мутирующую команду из текста PR».

## Как ловится
- Симптом в выводе: строки `<имя>: command not found`, плюс неожиданный запуск чего-то реального (у меня — FATAL от pg-pool).
- Тело PR в GitHub приезжает **испорченным** (backtick-сегменты заменены на вывод/пусто).

## Лечение (правило)
Любой многострочный текст с бэктиками или `$` (тело PR/issue, release notes, коммит-сообщения с примерами команд) передавать **через файл**: `gh pr create --body-file body.md` (или `--body-file -` с heredoc), **никогда** не инлайном в двойных кавычках bash. Если инлайн — только одинарные кавычки (но они ломаются на апострофах в тексте). `--body-file` — самый безопасный путь.

## Почему переносимо
Любой проект, где агент создаёт PR/issue через `gh` из bash (GONBA/setka). Особенно опасно, если в теле цитируются команды деструктивных/прод-скриптов — command substitution их выполнит. Класс — «текст как код»: данные, попавшие в shell-контекст, исполняются.
