## TL;DR

Служебный долгоживущий admin-JWT (у нас — `MATRICA_LEDGER_RELEASE_TOKEN` в прод `.env`, TTL 30 дней) протухает по расписанию, и его ротация числилась ручной процедурой «владелец заходит в web-admin → выпускает токен → вписывает в .env». Наблюдение: **сам хранимый токен — это валидный Bearer-credential для эндпойнта, который выпускает такие токены** (`POST /auth/release-token`, requireAuth admin). Пока старый жив, новый минтится им же одной curl-командой — без пароля, без UI, без участия владельца.

## Как устроено у нас

```bash
OLD=$(grep '^MATRICA_LEDGER_RELEASE_TOKEN=' .env | cut -d= -f2-)
NEW=$(curl -fsk -X POST https://127.0.0.1/auth/release-token \
  -H "Authorization: Bearer $OLD" -d '{"ttlHours":720}' | jq -r .accessToken)
cp .env .env.bak-$(date +%Y%m%d)
sed -i "s|^MATRICA_LEDGER_RELEASE_TOKEN=.*|MATRICA_LEDGER_RELEASE_TOKEN=$NEW|" .env
```

Рестарт сервисов не нужен (скрипт-потребитель читает `.env` при запуске). Проверка — декод payload (`exp`). Рецепт вписан в PENDING рядом с датой следующего протухания.

## Почему переносимо

Любой проект со служебным JWT/API-токеном с TTL (deploy-токены, CI-токены, machine-to-machine ключи), у которого есть аутентифицированный эндпойнт выпуска токенов: ротация из «ручного ритуала с паролем» превращается в идемпотентную 2-минутную ops-команду (или cron до протухания). Неочевидность в том, что токен психологически воспринимается как «узкоспециальный секрет для скрипта X», а технически это полноценный credential своего же issuer'а.

## Оговорка (security trade-off)

Это же свойство — расширение blast-radius утечки: украденный «release-токен» может бесконечно продлевать сам себя. У нас принято осознанно (токен живёт только в прод `.env` с правами 0640 вне репо); если модель угроз строже — issuing-эндпойнт стоит гейтить отдельным правом, а не общим `role=admin`, и тогда паттерн не применять.
