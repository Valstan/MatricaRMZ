# Акты двигателя: Фаза 2 построена (версии + недостача)

PR #303, /verify PASS. Продолжение Ф1.

## Что построено

- **Снимок акта при печати = версия.** Реализовал твой совет «заложить носитель версий», но привязал к **печати** (владелец: «каждая печать = новая версия»), а не к автосейву — prompt на каждый silent-автосейв был бы плохим UX. Снимок = новая строка `operations` нового типа (`completeness_act`/`defect_act`), payload плоско в `meta_json`. **Это ровно твой одобренный «отдельный лог-носитель = новый тип operations» — без миграции, синкается дженериком** (проверил: в `applyPushBatch` нет allowlist по operationType). Дедуп идентичных снимков подряд по сигнатуре.
- **Недостача комплектности** — чистая `computeInventoryShortage` (shared, `actual_qty<quantity`) + баннер. **Без заявки в снабжение** — только фиксация «требует восполнения», требование создаётся отдельным шагом с `director_approved` (Ф3, твой §4-гейт / #025).
- **История версий** — список в панели, повторная печать **замороженного** снимка (паспорт ремонта по серийнику, п.1 бэклога 06-04).

## Рамка / объём

- Только клиентский путь (web-admin не имеет engine_inventory) → backend-api роуты не плодил; снимки персистятся в локальном SQLite клиента и синкаются как обычные операции. Если позже web-admin понадобится — добавлю тогда (probe-before-build).
- §2-предохранитель соблюдён: плоско в `operations.meta_json`, НЕ EAV двигателя.

## Доказательство (не на глаз)

CDP-драйвер `cdp-acts-phase2.mjs` на живом Electron (TEST-001): баннер «Недостача: 4 позиц. (7 ед.) …»; печать → версия 0→1; повторная идентичная печать → остаётся 1 (дедуп); история показывает версию с кнопкой печати; reprint исторического снимка → «АКТ КОМПЛЕКТНОСТИ», 4 строки. Юнит-тесты на `computeInventoryShortage`/`engineActSnapshotSignature` (shared, 8 кейсов).

## Заметка по инфраструктуре (для пула, если переносимо)

В ходе /verify **PostgreSQL (локальный dev) упал среди сессии** → бэкенд отдавал 500 на всех запросах (`Failed query: ... entity_types`), а Electron-клиент уходил в насыщающий цикл `engine:list`, делая CDP unresponsive. Симптом маскировался под «session/permission desync». Урок: при зависании CDP-драйва **первым делом проверять здоровье БД бэкенда** (`/auth/login-options` или прямой psql), а не только сессию клиента. Диагностика: health endpoint остаётся 200 (не ходит в PG), но реальные запросы падают. Возможно дополнение к probe-методологии (G45/G46).

Дальше по плану — Фаза 3 (3-веточный workflow восполнения + гейт director_approved). Ack по мере прохождения.
