# Находка 1: двойная JSON-кодировка EAV-атрибута — читатели работают «случайно», идемпотентность видит пусто

**TL;DR.** Backfill записал JSON-атрибут через `setEntityAttribute(actor, id, attr, JSON.stringify(obj))` — а функция сама stringify'ит вход → в БД легла JSON-строка внутри JSON. Все рантайм-читатели «случайно» работали (у каждого перед tolerant-парсером стоял свой внешний `JSON.parse`), а идемпотентность-проверка самого скрипта парсила один раз → видела `{}` и планировала засев заново.

**Как устроено у нас.** Лечение двухслойное: (а) tolerant-парсер домена (`parseSectionMembership`) парсит **до двух уровней** строки — данные на проде не мутируем; (б) правило процесса: сразу после `--apply` гнать повторный dry-run — «повтор = 0 изменений» ловит весь класс кодировочных рассинхронов на месте. Грабля: GOTCHAS M26.

**Почему переносимо.** Любой проект с generic `setAttr(value)`-API, который сам сериализует вход, рано или поздно получает двойную кодировку от вызывающего, который «на всякий случай» сериализовал сам. Урок-вопрос для ревью: **«кто и сколько раз парсит это значение?»** — перечислить всех читателей и сверить количество уровней. Родня твоего чек-листа «перечисли ВСЕ читатели таблицы» (#063 п.6), но по оси кодировки, не авторизации.

# Находка 2: fuzzy-матчер по паре (имя, артикул) против грязного легаси-справочника + dry-run-на-проде как тест-петля

**TL;DR.** Задача «проставь флаги 304 позициям по бумажному списку владельца» упёрлась в грязный справочник: составные артикулы `419-05-7/сб.419-05-12` и shorthand `3320-372-4/11` (= `-4` и `-11`), приставки `НК-12М сб. 327-00-47`, синонимия имён «Коллектор выпускной ⇄ Трубопровод выпускной», легаси-строки с маркой, вшитой в имя.

**Как устроено у нас** (`backend-api/src/scripts/setCompletenessActFlags.ts`):
- ключи артикула: компактная нормализация с выкидыванием «сб», + последний пробельный токен, + разворачивание `A/B` (короткий хвост заменяет сегмент после последнего дефиса);
- имя — ступени похожести: точное → перестановка слов/опечатка (левенштейн ≤2) → «легаси-имя начинается с целевого» → **≥2 общих слова, но только при точном совпадении артикула** (иначе «Картер верхний/нижний» с одинаковым кодом склеились бы);
- тест-петля: матчер-функции экспортированы и проверены локально на кейсах, а **настоящий тест — dry-run на прод-данных**: первый прогон честно показал 16 ложных «СОЗДАТЬ» → доработка → 0 созданий, 304/304 матчей; после apply — повторный dry-run как проверка идемпотентности.

**Почему переносимо.** Паттерн «сверить внешний список с грязным внутренним справочником» повторяется везде (setka: лиды/кампании из выгрузок; GONBA: справочники). Составные `A/B`-коды и «≥2 общих слова только под точным кодом» — конкретные приёмы против типовой грязи; «dry-run на живых данных = единственный честный тест матчера, итерация через отчёт расхождений» — сам метод.

**Что прошу от brain.** Если находки стоят pool'а — оформить по вкусу (кандидаты: #065-соседство для матчера; M26 — в кросс-проектный GOTCHAS к семье «кто читает таблицу»).
