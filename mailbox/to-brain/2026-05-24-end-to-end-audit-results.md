# Результат финального end-to-end аудита MatricaRMZ

## TL;DR

**Блокеров запуска не найдено.** Все 8 стадий жизненного цикла двигателя (контракт → приёмка → дефектация → разборка → ремонт → сборка → испытания → отгрузка) **функционально присутствуют и работают**. Code-аудит первоначально дал 18 находок, из которых ~70% отменились / переоценились после live-обхода UI и обсуждения бизнес-модели с пользователем. Реально живая система значительно полнее, чем выглядит из чтения кода.

5 точечных UX-полировок зафиксированы в [`docs/PENDING_FOLLOWUPS.md`](../docs/PENDING_FOLLOWUPS.md) раздел «End-to-end audit 2026-05-24» — можно делать постепенно по фидбеку операторов после запуска, эксплуатацию не блокируют.

## Методология

Гибрид. По соображениям token-economy (ADR-0003) полный live-прогон через computer-use 8 стадий × 5 пунктов A-E оценился в 80+ скриншотов и большое контекстное окно. Применил два-фазный подход:

**Фаза 1 — code-аудит, 4 субагента в параллель** (стадии 1+2 / 3+4 / 5+6 / 7+8). Каждый прочитал соответствующие сервисы, формы, схему БД и сформировал заметки по чек-листу A-E (создание / связи / учёт / отчёты / BOM).

**Фаза 2 — live-обход через установленный клиент v1.22.0 на проде.** Read-only navigation по карточке существующего двигателя 2Ш10АТ1680 (АО «Уралтрансмаш», В-84, 1 из 1598 в системе). Открыл все секции карточки + раздел «Контроль и аналитика → Отчёты». **Фейкового двигателя `End-to-end audit 2026-05` намеренно не создавал** — чтобы не мусорить прод-данные, и потому что навигация по существующей записи выявила нужные UI-факты не хуже.

**Фаза 3 — обсуждение находок с пользователем.** Три критические находки субагентов отменены после объяснения бизнес-модели (см. ниже).

## Найденные баги / недоделки (критично — мешают запуску)

**Пусто.** Все первоначально-критические находки переоценены и переведены в техдолг либо отменены.

## Найденные улучшения (опционально, без бюрократии)

5 пунктов в [`docs/PENDING_FOLLOWUPS.md`](../docs/PENDING_FOLLOWUPS.md):

1. Запретить удаление контракта при привязанных активных двигателях (UX, soft-delete оставляет двигатели с «Контракт: не найден»)
2. Audit-log для смены status-флагов двигателя (сейчас пишется только в `attribute_values`)
3. UI-валидация обязательного двигателя при создании Assembly наряда (вместо ошибки при закрытии)
4. Выбор `variantGroup` BOM в Assembly наряде (если у марки несколько вариантов)
5. Третий вариант решения по дефектной детали — «заменить новой» отдельно от «утиль» (требует фидбека завода)

Все 5 — нон-блокеры, делаются по фидбеку операторов после запуска.

## Подтверждённые стадии (работают как задумано)

1. ✅ **Контракт** — entity `contract` (EAV) с полями `number`, `date`, `due_date`, `customer_id`, `contract_sections` (primary + addons). Связь с двигателем через `contract_id` атрибут на двигателе. Прогресс контракта вычисляется из связанных двигателей с цветовой подсветкой (синий 100% / градиент / красный просрочен).
2. ✅ **Приёмка** — поле `arrival_date` в карточке двигателя. На всех 1598 реальных двигателях заполнено. Контракт+клиент автоподтягиваются при выборе контракта.
3. ✅ **Дефектация** — встроенная подпанель «Лист дефектовки» в карточке двигателя. Шаблон `defect_default` (MVP). Поля: акт полной дефектовки (номер), марка*, № двигателя*, паспорт, дата начала/окончания, итоги. Таблица деталей с qty / ремонтно-пригодная / утиль. **3 поля подписей** (составил / согласовали / ремонт производить) с автоподстановкой ФИО+должности. **Вложения** (фото/PDF) к листу дефектовки.
4. ✅ **Разборка** — кнопка «Разобрать двигатель» в «Модуле движения деталей» карточки двигателя. Создаёт `engine_dismantling` warehouse-документ с привязкой engine_id и шаблоном строк из BOM. Проводка пишет движения в `erp_reg_stock_movements` (DismantleIn → ремфонд / DismantleScrapIn → утиль) и обновляет `erp_reg_stock_balance`.
5. ✅ **Ремонт** — наряд типа `Repair`, поля work-lines (двигатель / марка / вид работ / изделие / № изделия / кол-во / ед. / цена / сумма). Закрытие автогенерирует `repair_recovery` warehouse-документ с posting в склад. Идемпотентно через `linkedDocumentId`.
6. ✅ **Сборка** — наряд типа `Assembly`, БОМ как шаблон в `EngineAssemblyBomPage`. Закрытие списывает то, что бригада реально сделала, через `assembly_consumption`. **Несколько нарядов от разных бригад в сумме покрывают BOM** (см. «Архитектурные находки»).
7. ✅ **Испытания** — секция в карточке двигателя: «Дата прохождения испытаний», «Дата и № акта исследований», «Оригинал акта исследований получен», «Проверка комплектности (отметка мастера ОТК)». Отдельная подпанель «Акт комплектности двигателя» с таблицей деталей: Наименование / Обозначение / Количество (план) / Наличие (чекбокс) / **Фактическое количество**. Отдельная таблица «Несоответствующие детали и их №».
8. ✅ **Отгрузка** — поля карточки двигателя: «Дата упаковки», «Место хранения упакованного двигателя», «Дата отгрузки», status-флаги «Отправлен заказчику»/«Принято заказчиком». Прогресс контракта пересчитывается автоматом.

## Открытые вопросы для пользователя

**Закрыты в ходе аудита.** Три ключевых вопроса пользователь подтвердил:
1. BOM на уровне двигателя, не наряда — несколько бригад делают части → каждый наряд списывает только своё, в сумме покрывают BOM (это **bay design**, не баг).
2. Удаление контракта — реальный риск только UX (плохая навигация при soft-deleted), не катастрофа.
3. Двигатель не складская позиция — проводка при отгрузке не нужна, статусы в карточке закрывают учёт.

## Архитектурные находки (вне scope — для cross-project ADR)

Не блокеры, не задачи. Кандидаты на обсуждение / pool idea brain'а если окажутся переносимыми.

### 1. Стадии моделируются тремя параллельными механизмами

- **Operations** (`OperationTypeCode`: Acceptance / Defect / Disassembly / Repair / Test / Completeness / Otk / Shipment / CustomerDelivery / Kitting / Packaging / ToolMovement / SupplyRequest / + stock-операции)
- **Work Orders** (`WorkOrderKind`: Regular / Repair / Assembly / Manufacturing)
- **Warehouse Documents** (`engine_dismantling` / `repair_recovery` / `assembly_consumption` / `assembly_return` / `production_release` / + общие stock-документы)
- **Status Flags** (атрибуты двигателя: `status_repaired`, `status_customer_sent`, `status_customer_accepted`, `status_rejected`, `status_storage_received`)

Нет единого поля `engine.stage` ENUM. Текущая стадия двигателя **implicit** — вычисляется из наличия записей в operations/work_orders/warehouse_documents + проверки status-флагов. Работает, но усложняет диагностику и не даёт явного state-machine на двигателе.

**Не баг**, может никогда и не понадобится менять. Зафиксировано на случай если в будущем потребуется явный workflow-engine.

### 2. Закрытие контракта — implicit

Когда `aggregateContractExecutionProgress` посчитал `progressPct ≥ 100` (все двигатели имеют `status_customer_accepted`) — контракт считается «закрытым». Но **нет явной операции** `contract_closed` в `operations`. При расследовании «когда именно закрылся контракт XYZ» — точную дату из операционной истории не получить, только косвенно через даты `status_customer_accepted_date` всех двигателей.

### 3. `shipping_date` не требует предварительного прохождения испытаний

Чекбокс «Дата отгрузки» в карточке двигателя можно поставить минуя секцию «Дата прохождения испытаний» и «Акт комплектности». Скорее всего **by design** — статусы независимы, оператор ставит когда удобно. Но если на заводе будет требование строгой последовательности (нельзя отгрузить пока ОТК не подписал) — потребуется явная валидация. Решение завода, не аудита.

## Мета-наблюдение для brain (важно)

**Code-only audit без UI-обхода неприменим для acceptance-аудита.** Из 18 первичных code-находок 4 субагентов:
- 6+ отменилось при первом же UI-скриншоте (вложения, подписи, кнопки в карточке, готовые отчёты)
- 3 переоценились после обсуждения бизнес-модели с пользователем (BOM на уровне двигателя, soft-delete защита, двигатель не складская позиция)
- 5 осталось как реальный (но не блокирующий) техдолг
- 3 архитектурных замечания вне scope

**False-positive rate ~70%.** Эффективная процедура acceptance-аудита требует гибрид:

1. **Code-аудит** — формирует гипотезы «вот это должно быть проблемой»
2. **Live UI-обход** — проверяет какие гипотезы подтверждаются в реальном UI
3. **Бизнес-фидбек пользователя** — отметает гипотезы основанные на неправильной модели бизнеса
4. **Финальная классификация** — критическое / техдолг / архитектурное / отменено

Это потенциально материал для **cross-project ADR brain'а** или **pool idea** «процедура финального acceptance audit перед запуском в эксплуатацию для ERP-проектов». Если brain сочтёт переносимым (GONBA / setka / будущие проекты) — оформить как pool idea. Если только для MatricaRMZ — можно не оформлять, факт зафиксирован в этом письме и в `DEVELOPMENT_LOG.md`.
