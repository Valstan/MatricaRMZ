## TL;DR

При верификации фичи, где **сервер пишет данные, которые должны дойти до клиентов** (write → persist/sync → read), CDP/e2e-смоук, проверяющий **немедленный ответ** хендлера, может пройти зелёным, хотя данные **никогда не доезжают** до места, откуда их читают. Лечится: верифицировать **персистентную/синканную поверхность** (свежий reload / re-mount + инспекция БД), а не ответ.

## Как всплыло у нас

Ремфонд Ф3 (поэкземплярный учёт деталей): backend-route захватывал записи и **возвращал их в ответе**. UI рисовал список из ответа → CDP-смоук «прошёл». Но route делал голый `db.insert` без проведения через единый sync-путь (`recordSyncChanges`), поэтому строки не получали `last_server_seq` → incremental-pull клиентов их фильтровал → реестр был пуст на свежей загрузке и не доезжал на другие машины. Поймали **через релиз позже** (Ф4), когда свежая загрузка карточки показала пустой реестр. Инспекция прод-БД подтвердила: проблемные строки `last_server_seq=NULL`/`pending` vs исправленные `synced`+seq.

## Почему переносимо

Это не про домен MatricaRMZ — это про **любую систему с разделением «ответ хендлера» vs «персистентный/реплицируемый стор»** (sync-движки, CQRS-проекции, event-sourcing, кэш-инвалидция, репликация). Урок верификации общий: **прохождение ответа доказывает, что хендлер отработал, а не что запись легла туда, откуда её читают читатели.** Смоук-чек happy-path по ответу даёт ложную уверенность.

## Чек-лист (предлагаемый to-do при верификации write-фич)

1. Проверять не ответ, а **то место, откуда реально читают** (свежий reload / re-mount, который перечитывает локальный стор; запрос к реплике/проекции).
2. При наличии — **инспектировать стор напрямую** (seq/cursor/флаг синхронизации, а не только «строка есть»).
3. Если стор асинхронный (sync/репликация) — дождаться/форсировать пропагацию, затем читать.
