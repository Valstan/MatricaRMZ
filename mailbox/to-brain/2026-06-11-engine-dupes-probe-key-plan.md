# Задача 1: probe дублей двигателей + канонический ключ + план

## Probe-цифры (прод, read-only, 2026-06-11)

- Живых двигателей: **1615** (entity type `engine`, EAV); 1612 с `engine_number`, 3 без номера.
- Дубли по точному номеру: **22 группы, 23 лишних записи** (одна группа из трёх — Х06АТ0241).
- Compact-нормализация (`normalizeLookupCompact`: lower, ё→е, пунктуация/пробелы вон) **не добавляет ни одной группы** — вариантов написания типа `240-1` vs `2401` в данных нет. Но ключ всё равно берём нормализованный — защита на будущее, и тот же норматайзер что в поиске #035 (G45 соблюдён: считал по смыслу через JOIN на attribute_defs, не по строкам).
- Паттерн каждой группы: один двигатель из **массового импорта 2026-02-21** (часто с 5 актами) + поздний ручной дубль (2026-02-25…2026-05-27) — операторы заводили заново вместо поиска. Это подтверждает и приоритет login/search UX-ниток.
- **5 групп с расхождением бренда** при одинаковом номере (Т10АТ7387, У02АТ6800, Ф08АТ1817, Э05АТ6263, 5Н10АТ3319-пустой-бренд). Номера АТ-серии глобально уникальны → это та же физическая машина с ошибкой бренда в одной из записей, не разные двигатели.

## Карта ссылок (готова, merge сильно проще parts-chain)

На 23 дубль-id ссылаются **только** `operations.engine_entity_id` — 72 строки, все `operation_type='engine_inventory'` (акты). Проверено: `erp_reg_stock_movements.engine_id` — 0, `erp_engine_instances` — 0, входящих EAV-ссылок в `attribute_values.value_json` — 0 (position-скан по всем defs).

## Канонический ключ

**`normalizeLookupCompact(engine_number)` — без бренда.** Обоснование: 5 brand-mismatch групп — реальные дубли (ошибка бренда), ключ с брендом их пропустил бы. Норматайзер — тот же модуль что поиск (#035 Ф0 lift в `shared/src/domain/tieredSearch.ts`) → дубль-детекция и поиск никогда не разойдутся.

## План (cleanup → constraint → UI)

1. **Dry-run merge-скрипт** (`backend-api/src/scripts/dedupeEngines.ts`, методика parts-chain + G36): survivor = больше актов, тай-брейк — старший; перевешивание 72 operations; EAV-merge атрибутов (survivor держит непустые, пустые добиваются из loser'а; brand-конфликт — в отчёт владельцу, не авто); soft-delete loser'ов. Отчёт row counts — в PR.
2. **Apply на прод** — только owner same-turn confirm (#025) + pg_dump до. Все мутации через `recordSyncChanges` (урок #310) → клиенты получат soft-delete и перевешенные operations инкрементальным pull'ом.
3. **Tombstone-маппинг A→B** (`merged_into`): серверный приём синка переписывает входящие ссылки на влитый id → офлайн-очереди не стопорятся и дубль не воскресает. Реализация — таблица-маппинг + rewrite на sync-accept.
4. **Гейт create + sync-merge (серверный):** проверка канонического ключа при создании двигателя И при приёме синка; конфликт — явная ошибка оператору со ссылкой на существующий двигатель, не молчаливый пропуск. DB-constraint на EAV в лоб не положить (значение в attribute_values) — гейт в сервисном слое + страховочный уникальный индекс на materialized-нормализованный номер рассмотрю при постройке.
5. **UI-предупреждение «похожий двигатель уже есть»** — кандидаты бесплатно из #035-модуля (substring/fuzzy). Поэтому слотинг: гейт-UI логично идёт ПОСЛЕ или ВМЕСТЕ с Ф0 #035 (lift норматайзера).

Сигнатуры актов: merge не трогает payload снимков (engineNumber в header остаётся историческим), перевешивается только `engine_entity_id` — урок Ф4 учтён.

## Задачи 2+3 — ack и слотинг

Строю **сейчас одной мини-фазой «login-UX»**: login-MRU (machine-local `login-mru.json` в userData, только имена, prune по unknown/disabled + сверка с активными; учтён dev/prod userData split 06-08) + автоскрытие дропдауна 3с (один debounce-таймер, сбросы по вводу/стрелкам/hover, ESC/blur — сразу). Бонус-безопасность отмечу владельцу. Слотинг батча: login-UX → письмо это → client-perf probe A → dedup apply по owner-confirm → гейт (возможно вместе с Ф0 #035).
