# Ответ: client-perf (A probe-план + B статус)

## B. Delta-обновления — статус Этапа-2

1. **Статус: не начат** (честно — в очереди). С 05-30 сессии шли по parts-chain Phase 3 (B–E), shop-stats Ф0 и engine-acts Ф1–Ф4 (закрыты, /verify PASS каждая). Движения по Этапу-2 не было — твоё наблюдение по handoff верно.
2. **Приоритет — принимаю сигнал владельца.** Предлагаемый слотинг ниже; «раньше, чем позже» = следующая большая нитка, не дожидаясь Ф5 актов (Ф5 самодостаточна и может подождать).
3. **Атомарность/verify/rollback — подтверждаю, это обязательная часть реализации, не опция:** сборка нового installer из blockmap-кусков во временный файл → сверка sha512 целого файла с `latest.yml` ДО какого-либо применения → при несхождении автоматический fallback на полную закачку (текущий путь, он никуда не девается). Т.е. худший исход delta-сбоя = сегодняшнее поведение (85 МиБ), не «глючная установка». Это войдёт в acceptance-гейт Этапа-2 (#027: фантом-гейт не считается).
4. Подтверждаю и месседж владельцу из ADR: дробить клиент на «платформу/модули» не нужно и опасно (ABI-связка Electron ↔ better-sqlite3); blockmap-diff даёт ровно «обновлять кусочками» безопасно.

## A. Приоритетная начальная синхронизация — probe-план (ДО постройки)

Probe ещё не выполнен; строить не начинаю (#020/G46 принимаю). Что уже знаю из кода (уточняет вопрос 2 твоего чек-листа):

- **Cold start = full pull целиком** (`sync.fullPull` — полная копия серверной БД в клиентский SQLite). **Инкрементальность уже существует, но только ПОСЛЕ холодного старта**: watermark `lastPulledServerSeq` в `sync_state`, дальше тянутся только изменения. Т.е. боль владельца — именно первый full pull после установки/восстановления; «инкрементализировать» его нечем (нет локальной базы для diff), значит кандидаты — приоритизация + progressive fill + сжатие/батчинг, а не watermark.
- Отсюда probe-вопросы сужаются: в первом full pull что доминирует — сеть (объём/round-trips), вставка в SQLite (транзакции/батчи), индексация? Какие таблицы топ по строкам/байтам? Сколько секунд до «оператор видит свою страницу» при наивной приоритизации?

**Probe-план (1 сессия):** телеметрия фаз full pull (per-таблица: строки, байты, ms сети, ms вставки) на dev-снапшоте прод-БД + replay на чистом изолированном userData (стенд verify это уже умеет); вывод письмом с числами → решение priority-queue / progressive fill / batching / комбинация.
