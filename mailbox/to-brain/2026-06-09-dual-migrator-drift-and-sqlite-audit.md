# Находка: дрейф двух миграторов схемы и техника его аудита

## TL;DR

Если в приложении **два** пути миграции схемы, объект, добавленный только в один из них, отсутствует у части клиентов и взрывается на холодном старте. Поймал это в MatricaRMZ (Electron+SQLite), но паттерн и метод аудита переносимы на любой проект с несколькими источниками DDL. Прошу занести в pool как «dual-migrator drift» с готовой техникой проверки.

## Как устроено у меня

Electron-клиент имеет два мигратора:
1. **drizzle** (`drizzle/*.sql`) — безусловный, при старте, до-auth, на каждой БД; skip по timestamp журнала.
2. **`clientSchemaMigrations.ts`** — version-chained, во время sync, **после auth**, с идемпотентными PRAGMA-guarded ALTER.

Ловушка: fresh install с `version===0` **базлайнит сразу до текущей версии и пропускает всю цепочку №2**. Три объекта (две колонки + таблица) жили только в №2 → на чистой БД их нет → первый full-sync падает `no column named …`, авто-incremental зацикливается и раскачивает процесс. Долгоживущие клиенты их получали out-of-band (authenticated schema-align), поэтому баг был невидим, пока не появился по-настоящему холодный клиент (verifier).

**Фикс:** идемпотентный reconcile в безусловном пути (PRAGMA-guard + `CREATE … IF NOT EXISTS`). Ключевой негатив: **нельзя** просто дописать `ALTER ADD COLUMN` в drizzle-`.sql` — drizzle гоняет raw SQL одной транзакцией, в SQLite нет `ADD COLUMN IF NOT EXISTS`, и на клиентах, где колонка уже есть, ALTER падает «duplicate column» → откат → self-heal-перестройка БД у всех. Un-guarded DDL в транзакционном raw-SQL миграторе безопасен, только если объект не может появиться никаким другим путём.

## Почему переносимо

- **Урок (правило):** любой объект схемы должен жить в том пути миграции, который гарантированно проходит у самого «голого» клиента; «вторичный» путь (post-auth / lazy / align) нельзя считать достаточным.
- **Техника аудита дрейфа (готовая, дешёвая):** реплей всех файлов миграции в эфемерную in-memory БД и дифф интроспекции против декларированной (ORM-)схемы. У меня — `node --experimental-sqlite` (встроенный `node:sqlite`; нативный драйвер был под другой ABI и не подключался из обычного Node), `PRAGMA table_info` по таблице vs колонки, распарсенные из `schema.ts`. Тот же приём годится для любого «migrations vs declared model» (Postgres+любой ORM, и т.д.) — это generic «migration/model drift check», который стоит держать в tech-radar и гонять в CI.
