# Профиль окружения на каждый компьютер (docs/machines/<hostname>.md)

## TL;DR
Разработка идёт на нескольких компах. На каждом dev-стенд настроен по-разному (порт PG, порты backend/vite/CDP, наличие psql в PATH, последовательность запуска скилла verify, ABI-нюансы). Без записи каждая новая сессия заново нащупывает «как тут всё запускается» и путается в портах — это сожжённые токены и ложные старты. Завёл **профиль окружения на каждый комп**: `docs/machines/<hostname>.md`.

## Как устроено у меня
- **Один файл на hostname** (`docs/machines/PC40.md`), правит только сам этот комп → межмашинных merge-конфликтов нет (в отличие от одного общего файла с секциями).
- **Коммитится** (не gitignore) → переживает реклон, синхронизируется между машинами (GitHub = источник истины); чужой профиль = справка, не трогаем.
- **`/start` §0.5** (раньше git-операций и запуска инструментов): определить `hostname` → прочитать свой профиль → использовать значения вместо нащупывания; если файла нет — завести по шаблону по мере настройки.
- **Пишется по мере изучения** в течение сессии: выяснил, как запускается X (порт/путь/команда/ABI-грабля) → дописал строкой. Не «копим на потом».
- **Без секретов** (пароли остаются в `*.env*`) — только «как запускать».
- Добавлен в always-read источники правды (рядом с SESSION_HANDOFF / CODEBASE_MAP).
- Реализация: PR #486 (`docs/machines/README.md` конвенция+шаблон + `PC40.md`, правка `.claude/commands/start.md` §0.5 и `CLAUDE.md`).

## Почему переносимо
Любой проект, который ведётся **с нескольких машин** (а у тебя в реестре их несколько — GONBA/setka и т.д.), страдает тем же: cold-start заново выясняет локальные порты/пути/запуск тулинга. Паттерн доменно-нейтрален: «по файлу на hostname, читать свой на старте, писать по мере изучения, без секретов, коммитить». Развивает идею «карта прежде разведки» (ADR-0003) на ось железа/окружения.
