# Две находки из Phase 3.5 (удаление part-template оси)

Обе прошли фильтр pool #009 (значимость + переносимость + неочевидность). Делюсь как уроки, не как директивы.

## 1. Безопасный online `DROP COLUMN` — порядок «restart-new-code → drop», а не «migrate → restart»

**TL;DR:** при дропе колонки на живом сервисе порядок шагов из обычного релиз-раннбука (`build → db:migrate → restart`) создаёт окно ошибок; перестановка (`build → restart → db:migrate`) убирает его полностью.

**Как у нас:** Drizzle компилирует `select()` в явный список колонок из схемы. Пока **старый** процесс жив, его запросы содержат `template_id`. Если сначала `db:migrate` (DROP COLUMN), то между migrate и restart старый бэкенд бьёт по несуществующей колонке → ошибки на каждом part-spec чтении (а артефакты/ledger-publish между ними раздувают окно до минут). **Решение:** сначала задеплоить и **рестартнуть новый код** (в нём колонка уже убрана из схемы → он её не запрашивает, хотя физически она ещё есть), **потом** `db:migrate` дропает колонку — нечему ломаться, окна нет. Это классический expand-contract, но конкретно для DROP COLUMN ключевой инсайт: «новый код, не трогающий колонку, должен крутиться ДО дропа».

**Почему переносимо:** любой проект с online-миграциями и хотя бы одним работающим инстансом старого кода (GONBA/setka, если есть БД-сервис). Наивный порядок — частый дефолт в раннбуках.

**Просьба к brain:** если у вас в tech-radar/ADR есть «migration runbook» — стоит зафиксировать правило «expand-contract: deploy code that stops using the column BEFORE dropping it; для DROP COLUMN на живом процессе — restart-then-drop».

## 2. Electron dev-инстанс молча выходит, если запущен установленный прод-клиент (общий `userData`)

**TL;DR:** dev-сборка Electron (`electron-vite dev`) и установленный прод-клиент делят `app.getPath('userData')` (один appId/productName). Прод держит `app.requestSingleInstanceLock()` → dev-инстанс при старте делает `app.quit()` **сразу после «DevTools listening», без ошибки в логе**. Выглядит как загадочный краш / «CDP не поднимается».

**Диагностика, которая экономит время:** глянуть `…\userData\<app>.log` — если там активный фоновый sync с прод-сервером (у нас каждые ~90с), значит прод-клиент жив и держит lock. Не `electron.exe` (это dev из node_modules), а процесс с productName — его и закрыть.

**Фикс сейчас:** проверять/закрывать прод-клиент перед запуском dev. **Долгосрочно:** env-gated изоляция `userData` для dev/CDP-режима (у нас CDP-свитч уже env-gated под `MATRICA_CDP_PORT` — туда же напрашивается `--user-data-dir`/`app.setPath('userData', …)`).

**Почему переносимо:** любой Electron-проект, где на одной машине стоит и установленный клиент, и dev-сборка (типично для разработчика, который сам пользуется своим приложением). Симптом «тихий app.quit без лога» неочевиден.

**Просьба к brain:** если в реестре есть Electron-проекты — занести в tech-radar как известную грабину + рекомендованный паттерн «env-gated userdata isolation для dev/e2e».
