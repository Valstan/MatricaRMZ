## TL;DR

Прод-инцидент MatricaRMZ 2026-07-07: оба backend-инстанса залипли на ~50% CPU, клиентов подвешивало и выкидывало в релогин. Корень — **presence-хартбиты писались в durable, зашифрованный, раздаваемый всем sync-ledger**. Переносимый анти-паттерн + дешёвая эвристика для любого проекта с ledger/append-log-синхронизацией.

## Паттерн (переносимый)

`user_presence` был полноценной sync-таблицей: каждый онлайн-хартбит (`/presence/me` раз в 60с × каждый клиент + «активность при отправке чата») писался отдельной строкой в append-only зашифрованный ledger. Presence составлял ~⅔ всех записей ledger.

Механизм деградации: каждая запись двигает голову ledger → у **всех** N подключённых клиентов появляется «новое» → каждый поллит `/ledger/state/changes` и **расшифровывает** перекрывающееся окно на главном JS-потоке. Хартбиты × пуллеры = **O(N²)** крипто-работы. При всего 4-5 клиентах — уже ~50% CPU на оба процесса, и **не масштабируется**.

Ключевой признак в диагностике: **запись в ledger мизерная (~11 строк/мин), но CPU высокий** → стоимость на стороне чтения/расшифровки, не записи. В access-логе клиенты повторяют один и тот же `since`-курсор по 5-8 раз.

## Почему это был дубль

Онлайн-статус и так живёт в проекционной таблице `userPresence` (сервер отдаёт его напрямую в ответах chat/notes + self-ping `/presence/me`). Клиент **синканные presence-строки для UI не читает** — фанаут через ledger был чистой лишней работой.

## Фикс

Presence обновляет только проекционную таблицу; в ledger не пишется. Плюс защитный фильтр в единой точке записи (`writeSyncChanges`) — инвариант «presence не попадает в ledger» держится для любого пути (server-internal + client push). Прод-verify: presence-запись в ledger 9/мин → **0**, load 1.5 → 0.6. Фикс #110 (backend-only hot-deploy).

## Эвристика для tech-radar / cross-project GOTCHAS

**Эфемерные высокочастотные данные (presence / heartbeat / «печатает…» / cursor-позиции) НЕ кладём в durable, подписанный, раздаваемый всем sync-ledger.** Им место в отдельном лёгком эфемерном канале или в проекционной таблице, читаемой server-side по требованию. Проверочный вопрос при добавлении новой sync-таблицы: «читает ли клиент эти синканные строки для UI, или он получает то же самое из ответа другого эндпоинта?» Если второе — таблице не место в фанаут-ledger. Возможно, актуально для GONBA/setka, если у них есть похожий append-log/CRDT-синк с онлайн-индикаторами.
