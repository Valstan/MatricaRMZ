# Находка: валидатор ссылок ослеп на мигрировавший тип (проверял по старой таблице)

## Симптом

Гард целостности ссылок (`enforceEntityReferenceIntegrity`, добавлен месяц назад) валидировал **все** изменённые ссылки нарядов, резолвя их id по таблице `entities`. Но один класс сущностей — детали — за это время **мигрировал** из EAV (`entities`+`attribute_values`) в отдельные таблицы `directory_parts`/`erp_nomenclature`. Итог: живой `partId` наряда не находился в `entities` → гард репортил `not_found` и **блокировал сохранение** любого сборочного наряда с деталью. На проде — полный блок ключевого сценария; всплыло только когда оператор применил шаблон сборки (первый раз, когда `partId` реально менялся и попадал в проверку).

## Почему это переносимо и неочевидно

Условия воспроизведения (перечислимы, ни одно не про наш домен):
1. есть валидатор/гард, резолвящий ссылки **по таблице** (а не по абстракции «существует ли сущность X»);
2. один из ссылаемых типов **переезжает** в другую таблицу/хранилище;
3. миграция и валидатор — в **разных слоях/PR**, правятся не в лок-степе;
4. валидатор для этого типа не покрыт тестом на «живая ссылка проходит» (покрыт был только «битая — падает»).

Гард выглядел корректным и прошёл ревью: он делал ровно то, что написано, — просто «источник истины про существование» под ним сместился. Классический **stale-invariant**: механизм не сломан, он валидирует против устаревшей карты мира. Тот же род, что G176 (гейт, выведенный из-под наблюдения конфигом) и наш no-op write-path (G177), но здесь — не конфиг и не путь записи, а **таблица-резолвер**.

## Лечение (у нас) и общее правило-кандидат

Лечение: каталожные типы (part/nomenclature/product/service) резолвим по `erp_nomenclature`+`directory_parts` в дополнение к `entities` — ровно как уже делала соседняя ветка того же гарда для `productId` заявок (значит развилка была известна, но не применена ко второму call-site — это подтип «урок не доехал до соседа», #009).

Правило-кандидат для tech-radar / GOTCHAS:
> **Мигрируешь тип сущности в новое хранилище — грепни все резолверы ссылок на этот тип и правь их в том же PR.** Валидатор, резолвящий по таблице, обязан иметь тест «живая ссылка проходит» для КАЖДОГО ссылаемого типа, а не только «битая падает» — иначе миграция типа тихо превращает валидатор в блокиратор.

Проверять на вашей стороне, есть ли аналогичные «резолвер по таблице + мигрировавший тип» — не обязательно, скорее для пула паттернов.
