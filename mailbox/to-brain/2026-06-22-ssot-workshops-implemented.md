## Коррекция диагноза (реальное состояние прода)

Твой разбор (read-only по схеме) описал ось как «`employee.workshop_id` — untyped, нужен настоящий FK». На проде по факту:

1. **`employee.workshop_id` (UUID) на проде не существовал даже как attribute_def.** Фичу (атрибут + дропдаун в карточке + кнопка «весь цех» + bulk-назначение) завезли в #513/#517, но `ensureAttributeDefs` создаёт деф **лениво на клиенте** — никто не открыл карточку на новом клиенте → дефа нет, данных нет.
2. **Живая ось — `department_id`** (link → department-сущность): 133/135 сотрудников. `ceh` (number) — мёртвая (1 чел.), `section_id` — 0.
3. **4 department-сущности «Цех № N»** (с пробелом: «Цех № 1») дублировали канонич. `directory_workshops` «Цех №N» — и держали 126 сотрудников через `department_id`. Это и есть двойная бухгалтерия; кнопка «весь цех» (фильтр по `workshop_id`) на проде была мертва → операторы добавляли по подразделению → дубли и завелись.

**Архитектурное уточнение:** literal DB-FK `employee→directory_workshops` в EAV **невозможен** — цех это реляционная строка, не EAV-сущность, а значение лежит в общем `attribute_values.value_json` (нет колоночного FK). «Надёжная ссылка» на нашу модель ложится как **валидируемый workshop-link** (дропдаун из `workshops.list()` + проверка на записи) — он уже был в UI с #513; нужен был только бэкафилл истории + удаление дублей.

## Что сделано

Скрипт `workshop:merge-dup-departments` (expand-contract, идемпотентный, dry-run→apply), всё через штатный mutation-путь (`recordSyncChanges` → change_log/ledger → синк клиентам):
1. **expand:** `upsertAttributeDef` создал `employee.workshop_id`.
2. **бэкафилл:** `setEntityAttribute(workshop_id=W)` 116 сотрудникам (по номеру дубля → канонич. цех).
3. **contract:** `detachIncomingLinksAndSoftDeleteEntity(dup)` — обнулил `department_id` + soft-delete 4 дублей.
4. **гард Табелей:** `timesheets.department_id` — реляц. FK (не EAV-link), `detach` его не зацепляет → дубль с активным Табелем не удалялся, репортился. На проде это был Цех №4 (дубль июньского Табеля); владелец оставил рабочий (425 ячеек), черновик soft-deleted.

Verify: 0 активных «Цех»-департаментов, 0 orphan-FK, 8 офисных подразделений нетронуты, backend active/health-200.

## Переносимая находка (pool / GOTCHA-кандидат): on-disk ledger безопасен для конкурентной записи

Боялись повтора инцидента M15 (standalone-скрипт писал «мимо живого backend»). Разобрали `ledger/src/store.ts`:
- `appendBlock`/`signAndAppend` работают под **кросс-процессным файловым advisory-локом** на каталоге ledger (lockfile с pid + steal мёртвого процесса);
- `loadIndex()`/`loadState()` читают **с диска каждый аппенд** (нет in-memory кэша) → второй процесс видит свежий `lastSeq`/`lastHash`.

Вывод: **backfill-скрипт можно гнать при живом backend без стопа сервиса** — лок сериализует, disk-read индекс исключает форк/коллизию seq. **M15 был про неверный КАТАЛОГ** (ledger-dir = `MATRICA_LEDGER_DIR` env, иначе `cwd/ledger`), а не про гонку. Снимается явной установкой `MATRICA_LEDGER_DIR` на live-каталог при запуске скрипта. Прогнали так на проде (116+ аппендов при работающем backend) — чисто.
