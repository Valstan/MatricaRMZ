# Утечка ключа в публичном репо: три переносимых урока

Делал H8 из security-аудита (владелец авторизовал «ротация ключа + переписать историю + force-push»). При выполнении вскрылось, что задача шире и что **мандат на force-push был неоптимальным путём**. Три переносимых урока.

## 1. Утечка оказалась шире «подписного ключа» — проверяй ВСЮ историю, не только HEAD

Аудит нашёл `server-key.json` (подписной Ed25519) в HEAD. Но `git log --all -- <path>` показал, что в историю попадал ещё и **`data-key.json`** (ключ шифрования полей) — добавлен, потом «убран» через `git rm --cached` (untrack ≠ вычистка из истории, достаётся `git show <old-sha>:...`). Рядом закоммичены `state.json` + 119 блоков с шифротекстом. Итог: **ключ шифрования И шифротекст оба публичны → все `enc:v1:` поля расшифровываемы** (реальные ПДн — переписка/атрибуты сотрудников). Урок: при «закоммичен секрет X» всегда грепать `git log --all --diff-filter=A` по соседним секретам и по тому, что ещё лежало в той же папке.

## 2. **make-private ДОМИНИРУЕТ history-rewrite+force-push, когда forks/stars/watchers = 0**

Мандат был «filter-repo + force-push». Перед необратимым шагом собрал blast-radius: репо публичный, но **forks=0, stars=0, watchers=0**, а rewrite затронул бы **1276 коммитов + 579 тегов**, реклон ВСЕХ машин (вкл. прод-checkout), снятие branch-protection, риск для GitHub Releases.

Поднял владельцу развилку. **Сделать репо приватным**: мгновенно закрывает весь публичный доступ к истории, **ноль реклонов/рестартов/разрушений**, релиз-пайплайн не ломается (Actions работают на приватных, клиенты качают с своего сервера). При 0 форков это полностью эффективно. Владелец выбрал приватный.

**Переносимое правило:** для утёкшего-в-публичный-репо секрета порядок такой — (1) **ротировать** ключ (это и есть фикс живого риска; history-rewrite на публичном репо всё равно не «отменяет» утечку — форки/кэш/архивы), (2) **сделать приватным**, если публичность не нужна (проверь forks!), (3) history-rewrite — только опционально и только если репо обязан остаться публичным. force-push 5 месяцев истории + реклон парка — это последнее средство, а не первое. «Owner authorized force-push» ≠ «force-push оптимален» — собери blast-radius и предложи дешевле.

## 3. Капкан «live-данные внутри git-checkout» + CI-последствия приватизации

- **Капкан (→ наш GOTCHAS M19, кандидат в cross-project):** прод-backend резолвил ledger в `cwd/ledger`, а `cwd` = git-checkout → рантайм-данные (ключи+блоки) ОДНОВРЕМЕННО трекались git. Наивный `git rm --cached`/`git pull`/`reset --hard` на проде **удалил бы живой ключ и блоки**. Лечение: до любых git-операций **вынести live-данные из checkout'а** (env-переменная пути, `mv` при остановленных сервисах — rename атомарен) + бэкап. Класс «зелёный git-pipeline, а на проде потеря данных».
- **Приватизация молча ломает CI:** (а) **CodeQL** на приватном репо требует платный GitHub Advanced Security → default-setup просто исчезает (SAST-регрессия). Заменили на **Semgrep OSS** (бесплатен на приватных, в Actions) — он сразу нашёл реальный баг (`createDecipheriv('aes-256-gcm',…)` без `authTagLength` → принимает укороченный auth-тег). (б) **gitleaks-action** на приватном репо упал `403 Resource not accessible by integration` на чтении `/pulls/N/commits` — чинится `permissions: pull-requests: read` в workflow. Урок: смена visibility — это CI-изменение, проверь все workflow на токен-права и GHAS-зависимые фичи.
