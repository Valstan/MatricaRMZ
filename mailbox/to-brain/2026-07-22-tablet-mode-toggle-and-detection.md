# TL;DR

Делал планшетный (touch) режим для Electron-клиента. Две переносимые находки:

1. **«Показывать переключатель только на планшете» ≠ автодетект в рантайме.** Надёжного рантайм-сигнала «это планшет, а не сенсорный ноутбук» **не существует**, и наивный автодетект ломается на реальном кейсе (мышь воткнули в планшет). Решение: ручной переключатель, чью **видимость** гейтит **явная машинно-локальная настройка**; эвристика лишь сеет дефолт **один раз** и замерзает.
2. **Density/touch-режим на ~75% делается одним root-атрибутом + переопределением CSS-переменных-токенов** — без правки компонентов, если размеры контролов вынесены в токены.

# Как устроено у меня

**Находка 1 — детект.** Проверял адверсариально. Провалы автосигналов:
- `navigator.maxTouchPoints > 0`, `any-pointer: coarse` = «тач ЕСТЬ» — сенсорный ноутбук рапортует так же (свойство наличия, не класс устройства).
- `pointer: coarse` (ОСНОВНОЙ указатель) различает планшет-без-мыши от ноутбука, но **ломается ровно на нежелательном кейсе**: воткнули мышь в планшет → основной указатель `fine` → live-гейт спрятал бы кнопку, хотя это планшет. Отсюда: использовать `any-pointer`/`any-hover`, не основной.
- Диагональ в дюймах из `screen.width × DPR` не вычислить (нет EDID) → «маленький экран» врёт в обе стороны.
- Windows-нативные сигналы класса шасси (WMI `Win32_SystemEnclosure.ChassisTypes`, `SM_CONVERTIBLESLATEMODE`, WinRT `DeviceForm`) ближе к истине, но флаки (OEM оставляют SMBIOS generic) и тянут native-зависимость.

Реализация: настройка `deviceIsTablet` (машинно-локально, не синк, per-device — как тема) сеется эвристикой (`maxTouchPoints` + `any-pointer:coarse` + `any-hover:none` + мелкий экран) **один раз** и фиксируется явным значением; дальше правит только галочка в Настройках. Кнопка режима видна ⟺ настройка = планшет; сам режим — ручной тумблер.

**Находка 2 — density одним атрибутом.** Клиент уже держал размеры инпутов/кнопок/строк/паддингов в `:root`-CSS-переменных (`--ui-input-height`, `--ui-button-*-height`, `--list-row-padding-*`, …), а общие Input/Button читают их через `var()`. Планшет-режим = `document.documentElement.dataset.uiMode='tablet'` + один CSS-блок `:root[data-ui-mode='tablet'] { …переопределённые токены… }` (зеркалит существующий механизм темы `data-theme`). Это масштабирует ~75% UI без единой правки компонентов; правки нужны только там, где размеры захардкожены инлайном (степперы, дата-инпут — `!important`/inline-ternary) или где нужна логика (скрытые контролы, numpad). Грабля-нюанс: если часть токенов уже задаётся инлайном из JS (у меня — пользовательские размеры шрифта), CSS-класс их не переборет → те подмешивать через тот же setProperty-канал с `Math.max(userPref, tabletMin)` (уважать больший пользовательский).

# Почему переносимо

Любой проект с оператором на планшете/сенсорном экране (GONBA/setka — если появится полевой/цеховой ввод) столкнётся ровно с этим: (а) соблазн «само определит планшет» и его провал на мыши; (б) желание крупного touch-UI без форка всех экранов. Оба решения доменно-нейтральны.
