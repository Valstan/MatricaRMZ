# Раскол «открытое vs сделанное» в проектной памяти

## TL;DR

Проектные «источники правды» (handoff, PENDING_FOLLOWUPS, plans/) со временем **копят завершённую работу** → два симптома: (1) **«всплытие уже сделанного»** — устаревший «next-step»/«крупный кандидат» предлагается как недоделанный, хотя давно на проде; (2) **раздувание токенов холодного старта** — каждый `/start` платит за кладбище ✅-записей.

У нас `PENDING_FOLLOWUPS.md` дошёл до **465 строк, ~93% — ✅-записи** (129 done-маркеров vs ~10 открытых), дублируя историю, которая и так в `git log`/PR. За один день дважды взялись «выкатывать» Phase 3 и T3-импорт — оба давно на проде; диагноз: handoff/план тащили устаревший статус.

Лечение — **зеркало твоего же Tier-1-индекс + Tier-2-детали + lifecycle**, всё в репо (синк между машинами, #010):

- `PENDING_FOLLOWUPS.md` → **только открытое** (🔴/⏳/🟡/🟢 + `🗓 since:`).
- `COMPLETED.md` (новый) → **тонкий done-индекс**: 1 строка на нитку/релиз, без описаний (granular в git/PR). Это ledger «не гонять повторно».
- `GOTCHAS.md` (новый, проектный) → грабли по симптомам (Tier-1 индекс + записи), грепать перед отладкой. Кросс-проектные — по-прежнему к тебе.
- `plans/_archive/` → закрытые планы (`git mv`, история цела); в `plans/` только активные.
- **Процесс держит чистоту:** `/start` читает лёгкий PENDING (COMPLETED/GOTCHAS по требованию); `/close_session` при отгрузке переносит PENDING→COMPLETED одной строкой, `git mv` плана в `_archive`, и **сверяет next-step против PENDING/COMPLETED** (анти-stale-гард).

Итог у нас: `PENDING` 465→**86 строк** (−81%), 31 план в `_archive`, холодный старт читает только открытое.

## Почему переносимо

Боль структурная, не доменная: **любой** проект с «sticky-note handoff + список задач + папка планов» накопит завершёнку и начнёт ею фонить. GONBA/setka/KARMAN с тем же скелетом источников правды — кандидаты. Паттерн — твой же (Tier-1 INDEX + Tier-2 + статус-маркеры из `cross-project-ideas`/`GOTCHAS`), просто применён к **проектному** контуру для разделения «todo» и «done».

Ключевая мысль, которой не было до этой сессии: **рабочие файлы должны держать только открытое; завершённое — индекс + git, а `/close_session` обязан физически выносить done и сверять next-step.** Иначе «done»-история и устаревшие указатели отравляют каждый старт.
