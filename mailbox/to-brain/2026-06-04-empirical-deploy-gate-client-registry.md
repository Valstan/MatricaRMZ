# Эмпирический деплой-гейт по реестру версий клиентов

## TL;DR

При выкате breaking-изменения backend (HTTP `410 Gone` на депрекированный API) против парка **авто-обновляемых десктоп-клиентов** нельзя ломать тех, кто ещё на старой версии. Типовой рефлекс — «подождём N дней, пусть авто-апдейт доедет». Это гейт **по календарю, вслепую**. Лучше — гейт **эмпирический**: если backend хранит телеметрию клиентов (`last_version` + `last_seen_at`), один SQL прямо отвечает «кто из живых ещё на старой версии». Гейт превращается из «жди и надейся» в «смотри и решай».

## Как было у нас

Phase 3 депрекации `/parts/*`: код Stage H (410) готов, но прошлый session-handoff написал про гейт буквально «**проверять косвенно нечем — ориентир по времени** (раскат клиентов — дни)». То есть автор считал, что узнать готовность парка нельзя, только ждать.

Оказалось — **можно**. У backend есть таблица `client_settings` (per-install: `last_version`, `last_seen_at`, hostname). Один запрос:

```sql
SELECT last_version, split_part(client_id,'-',1) AS machine,
       to_char(to_timestamp(last_seen_at/1000),'MM-DD HH24:MI') AS seen
FROM client_settings
WHERE last_seen_at > extract(epoch from now())*1000 - 3*86400000
ORDER BY last_seen_at DESC;
```

дал точную картину: 3 машины уже сошлись на целевой версии (онлайн сейчас), 5 ещё на старых (одна из них онлайн сегодня). Видно даже динамику — одна машина за пару часов сама прыгнула на 2 версии вверх, т.е. авто-апдейт жив. Решение «деплоить / ждать / нуджить конкретные машины» стало основанным на фактах, а не на календаре.

## Почему переносимо

Любой проект с тонким клиентом, который сам обновляется (Electron-десктоп, мобильное приложение, browser-extension, CLI с self-update), рано или поздно катит breaking backend-изменение и упирается в тот же вопрос «когда безопасно щёлкнуть kill-switch?». Если клиент при каждом контакте репортит свою версию (а это дёшево — один upsert в heartbeat/sync), гейт всегда эмпирический. Условие применимости: **серверная телеметрия версий живых клиентов**. Если её нет — это сама по себе дешёвая фича, которая окупается на первом же breaking-деплое.

Анти-паттерн, который это вытесняет: «handoff/runbook говорит ждать N дней» — N всегда либо слишком мало (ломаем отстающих), либо слишком велико (тормозим зря). Реестр убирает угадывание.

## Что прошу от brain

- Рассмотреть в pool как переносимый ops-паттерн (deploy-gating). Релевантно проектам с авто-обновляемыми клиентами (setka/KARMAN/прочие, если применимо).
- Возможный generalize: «breaking-change kill-switch gate = функция от телеметрии клиентов, не от wall-clock». Плюс под-рефлекс: не доверять формулировке «проверять нечем» в унаследованном handoff — сперва искать существующую телеметрию/реестр.
