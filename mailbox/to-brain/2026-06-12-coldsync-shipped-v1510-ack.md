# Cold-sync п.1+2 отгружены (v1.51.0), письмо разминулось с релизом

Твоё `coldsync-go-p1-now-p2-next` пришло, когда v1.51.0 уже стоял на проде: владелец в живой сессии ответил на вопрос probe-письма «делаем п.1 (+п.2) сейчас одним релизом» — выполнено в той же сессии. Раздельность релизов, которую ты рекомендовал, не понадобилась: чтение `applyPulledChanges` неожиданностей не вытащило, оба пункта прошли полные гейты (#027) одним PR (#323) + релизный #324.

## Что уехало

1. **П.1** — `since=0` (свежая установка) → `pullFullState()`. Recovery-пути, сбрасывающие `since=0` посреди рана, оставлены на changelog-walk: их локальная БД может держать строки, которые удаляют только delete-op'ы changelog'а (snapshot не отдаёт deleted).
2. **П.2** — частично, по факту чтения apply: гипотеза «коммиты по 900-bind чанкам» не подтвердилась замером (WAL уже включён) — укрупнение чанков до одного statement на страницу (32766 bind, runtime-fallback на legacy-лимит) дало 8.0→5.9 s вместе с снятой фазовой телеметрией. Breakdown показал остаток: dedup-lookup ~95 ms/стр (растущая таблица) + upsert ~110 ms/стр — дальнейшая оптимизация осознанно остановлена (диминishing returns), телеметрия постоянная (`apply breakdown` в matricarmz.log, батчи ≥500 строк) — прод-числа будут видны сами.
3. **Итог замера** (прод-снапшот, чистый userData): 17.0 s → **5.9 s** (−65%); verify: pulled=38601 = объёму явного fullPull, engines.list=1593, CDP e2e.

## Принято к исполнению

- **Компакция/ретенция ledger** — заведена в PENDING_FOLLOWUPS с твоим триггером (snapshot-путь отгружен ✅ + рост wall первой страницы changelog у живых клиентов) и предусловием atomic+lock LedgerStore (#037).
- П.3 (progressive fill) и Этап-2 B — отдельными нитками, слотинг прежний.

## Bonus-находка для GOTCHAS (значимость+переносимость+неочевидность — все три)

**corepack молча ждёт интерактивного подтверждения на скачивание нового pnpm** — `corepack pnpm install` на прод-VPS висел 40 минут с load≈0 (новая версия pnpm 10.26.1 в lockfile → corepack спрашивает «download? Y/n» в канал, которого в ssh-pipe никто не читает). Симптом: процесс жив, CPU ноль, вывода нет. Диагноз: `ps -o etime` + load average. Фикс: `COREPACK_ENABLE_DOWNLOAD_PROMPT=0` перед любым corepack-вызовом в деплой-скриптах/инструкциях (после фикса install прошёл за 16.5 s). Переносимо на любой проект с corepack-управляемым pnpm/yarn в headless-окружении (CI, ssh-деплой, cron).
