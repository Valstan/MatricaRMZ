# Находка: stray-файлы в общей рабочей копии при параллельных сессиях

## Симптом
2026-06-24 в MatricaRMZ две Claude/Cowork-сессии работали в **одной** рабочей копии (`D:\GitHubReps\MatricaRMZ`). Сессия A (я, /zavod) положила временный файл для анализа в корень репо (`cp …/голосовые.txt → tmp_golosovye_2026-06-24.txt`). Сессия B (web-admin) в это же время переключала ветки и коммитила через `git add -A` — и **сгребла чужой temp-файл (99 КБ сырого текста) в свой коммит и запушила на origin** (в feature-ветку; до `main` не дошло — B сама убрала перед мержем + добавила `/tmp_*` в `.gitignore`).

Симптомы у A: рабочая копия неожиданно оказалась НЕ на своей ветке (HEAD уехал на ветку B); `git status` показывал «удаление отслеживаемого» temp-файла; `reflog` показал `checkout main → feat/… ; commit` чужими руками.

## Корень
Общий рабочий каталог + чужой `git add -A` = любой stray-файл в дереве рискует попасть в чужой коммит/историю (тем более сырой текст). Плюс переключение веток в общей копии ломает state другой сессии.

## Митигация (рабочая, проверено)
1. Temp-файлы анализа — **вне репо** (системная temp) или работать с контентом из памяти (Read внешнего пути), не `cp` в дерево.
2. Свою работу в общей копии вести через **изолированный `git worktree`** от `origin/main` (`git worktree add -b <branch> <path> origin/main`) — не переключая ветки общей копии; файлы worktree не видны чужому `git add -A`. Так я и завёл свой PR в этой же сессии, не задев активную параллельную.
3. Stray-паттерны сразу в `.gitignore` (`/tmp_*`) — но помнить: `.gitignore` НЕ убирает уже закоммиченный файл.

## Переносимость
Касается любого проекта (GONBA/setka), где допускается параллельная работа нескольких Cowork-сессий в одной рабочей копии. Кандидат в cross-project GOTCHAS.
