## Находка 1 — key-ремоунт с чужими stale-props: «снимок при монтировании» фиксируется по чужой сущности

**TL;DR.** Если компонент детальной карточки ремоунтится через `key={selectedId}`, а данные (`details`) живут в родительском state и загружаются асинхронно, то в переходном рендере карточка НОВОГО id монтируется со СТАРЫМИ details предыдущей сущности. Любое «вычислить один раз при открытии» (ref-guarded snapshot) фиксируется по чужим данным и уже не пересчитывается.

**Как проявилось у нас.** `EngineDetailsPage` при создании двигателя снимает флаг `isNewEngine` (= «номер пуст при открытии») один раз на engineId (ref-guard, чтобы подсказка жила всю create-сессию). App держит `engineDetails` в state; `openEngine(newId)` менял `selectedEngineId` сразу, а details подгружал позже → карточка нового id монтировалась с атрибутами ПРЕДЫДУЩЕГО двигателя (номер непустой) → `isNewEngine=false` → трёхвариантный выбор пути при дубле номера не предлагался. Воспроизводилось только в цепочке «открыл карточку → закрыл → создал новую» — на «чистом» открытии всё работало, из-за чего CDP-смоук первого прогона проходил, а второй сценарий падал.

**Лечение.** В `openEngine` сбрасывать stale-details ДО смены id: `setDetails(prev => prev && prev.id !== id ? null : prev)` — рендер карточки гейтится на `details != null`, переходный кадр с чужими данными исчезает.

**Почему переносимо.** Паттерн «`key={id}` + async-fetch в родительский state + per-mount snapshot (useRef/useState-инициализатор)» — стандартный для любых React/Electron-карточек (GONBA, setka web-UI). Симптом коварный: баг живёт только в переходах между сущностями, юнит-тесты и «открой карточку» его не ловят.

## Находка 2 — осознанный обход инварианта уникальности: flag+intent + порядок записи в батче

**TL;DR.** Когда жёсткий инвариант («номер двигателя уникален») нужно осознанно нарушать в редких легитимных случаях (повторный заезд той же единицы), вместо снятия гейта или отдельного API-параметра работает схема: (1) флаг-атрибут на самой сущности («повторный заезд»/«коллизия номера»), (2) оба гейта (клиентский offline и серверный) пропускают дубль только при флаге на пишущей сущности, (3) UI пишет флаги ПЕРЕД уникальным полем в том же батче — к моменту проверки флаг уже на месте, никакого спец-канала. Случайный дубль (без флага) блокируется как раньше; все механизмы склейки дублей обязаны исключать флагованных (иначе уничтожат разделение) — исключение в трёх местах: авто-pass, analyze-UI, ручной merge (hard reject, защита в глубину).

**Почему переносимо.** Любая offline-first система с синхронизацией и инвариантами уникальности рано или поздно встречает «легитимный дубль» (возвратная единица, повторная заявка). Схема не требует изменения протокола sync и переживает произвольный порядок доставки, если писать флаг раньше ключевого поля.
