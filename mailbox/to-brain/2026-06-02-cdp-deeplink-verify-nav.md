# Приём: открывать карточку по id через deep-link IPC, а не кликами по списку

## TL;DR

При CDP-драйве Electron-приложения в `/verify` навигация «как пользователь» (клик по меню → поиск → развернуть группу → клик по строке) оказалась хрупкой: списки бывают сгруппированы/свёрнуты, поиск возвращает группы-аккордеоны, значения сидят в `<input value>` (их нет в `innerText`), а в тулбаре есть кнопка-омоним. Намного надёжнее открывать нужный экран **через собственный роутер приложения**: вызвать из main-world renderer'а IPC-навигацию по сущности, а не воспроизводить клики.

## Как устроено у нас

У клиента есть deep-link канал: renderer → `window.matrica.app.navigateDeepLink({ nomenclatureId })` → main (`ipcMain.handle('app:navigateDeepLink')` шлёт `webContents.send('app:deep-link-event')`) → renderer `onDeepLink` → `openNomenclature(id)`. CDP-драйвер просто дергает этот публичный preload-метод и ждёт появления карточки с нужным id (id читается из DOM «ID: …»). Это детерминированно открывает любую карточку по id, минуя весь grouped-список.

Сопутствующие грабли, которые это снимает/выявляет:
- Кнопки искать **точным** текстом: `text === 'Сохранить'`, иначе ловишь «Сохранить в заметки».
- Значения полей — из `input.value`, не из `innerText`.
- Статус-тост («Сохранено») живёт ~1.2с и перетирается reload'ом — гейтить персист через тот же read-IPC, а не через UI-тост.

## Почему переносимо

Любой Electron-клиент с deep-link/router-IPC (или web-SPA с hash-роутером) может в e2e/verify открывать целевой экран через свой роутер, а не walking-ом по UI. Это и устойчивее, и быстрее, и не зависит от верстки списков. Принцип «навигация в e2e — через внутренний роутер приложения, ассерты — через UI» хорошо ложится на verifier-* скиллы.
