# Акты двигателя: Фаза 3 построена (3-веточный workflow + гейт)

PR #304, /verify PASS. Закрывает GAP-2.

## Что построено

- **`replenishment_branch` per-деталь** (`customer`/`repair`/`purchase`/null) — **логическая колонка `EngineInventoryRow`** (нормализуется, синкается как часть операции, без миграции; переживает brand-resync, как `__photos`/`__selected`). Селектор в таблице, активен при `replace_qty>0`.
- **Закупка фильтрует на ветку** — `rowGoesToPurchase` = `replace_qty>0` И ветка ≠ `customer`/`repair`. `null` → закупка (обратная совместимость с MVP-1: непомеченные негодные по-прежнему идут в требование). Явный выбор «заказчик»/«свой ремонт» уводит строку из черновика заявки.
- **Гейт = переиспользован `director_approved`** (твой ответ владельца #2). Нового механизма НЕ плодил: кнопка создаёт **черновик** заявки, нота в UI «уходит в требование только после одобрения директором». Сам переход draft→director_approved — существующий workflow снабжения.

## Рамка

- §2-предохранитель соблюдён: поле плоско в `operations.meta_json` (часть строки списка), НЕ EAV двигателя.
- Ветки «заказчик»/«свой ремонт» — пока только маршрутизация + сводка; **закрытие** в «есть» по приходу/готовности из ремфонда — Ф4/Ф5 (зависит от ремфонд→наряд→прогноз, GAP-4/5).
- Объём — клиентский (web-admin не имеет engine_inventory); фильтр живёт в shared, синкается.
