# Находка 1: сальдо периода — от текущего остатка назад, а не суммой движений с нуля

**Задача.** Оборотная ведомость: «сальдо на начало + приход − расход = сальдо на конец» по (локация × номенклатура).

**Наивная реализация** (первое, что просится): суммировать журнал движений с начала времён — `opening` = сумма движений до `startMs`, `closing` = `opening + приход − расход`. Работает ровно до первой позиции, чей остаток появился **не** через журнал: миграция, начальный ввод, легаси-импорт, ручная правка регистра, любой период до включения журналирования. Такие позиции показывают сальдо 0 при реальном остатке в сотни штук — и отчёт врёт молча, потому что арифметика внутри строки сходится.

**Что сработало.** Якорь — **текущий остаток регистра** (источник истины по определению), и от него идём назад:

```
closing = current_balance − Σ(нетто движений ПОСЛЕ конца периода)
opening = closing − (приход за период − расход за период)
```

Легаси-остаток без истории даёт `opening == closing`, приход/расход нули — строка честная. Заодно проверяемо граничными случаями без данных: период целиком в будущем → `opening == closing == current`; период целиком в прошлом до первых движений → нули. Оба прогнали смоуком.

**Правило (переносимое).** В периодных отчётах поверх «регистр остатков + журнал движений» **выводи сальдо из текущего состояния назад**, а не из нуля вперёд. Журнал отвечает на «что изменилось», регистр — на «сколько есть»; складывать историю с нуля значит требовать от журнала полноты, которой у него почти никогда нет. Тот же приём годится любому event-sourcing-подобному контуру со снапшотом: снапшот — якорь, события — дельта к нему.

# Находка 2: новое поле в API должно наследоваться, а не срезаться клиентом, который о нём не знает

**Контекст.** Добавили типизированное поле (`normPercent` — норма расхода) в строку спецификации. Запись строк идёт **full-replace upsert'ом**: клиент присылает весь массив строк, сервер пересоздаёт их.

**Грабля.** Толстые клиенты обновляются не одновременно (у нас — Electron с автообновлением, часть машин отстаёт неделями). Клиент старой версии не знает поля → не шлёт его → full-replace честно записывает строки **без** поля. Достаточно одному человеку на старой версии открыть и сохранить карточку, чтобы **молча стереть** данные, которые заполнял импорт или другой оператор. Ни ошибки, ни предупреждения — просто в следующий раз поля нет.

Это тот же класс, что нашу прежнюю граблю «zod-схема роута срезает неизвестное поле» (там теряли на входе, тут — на перезаписи), но злее: там терялось то, что прислали, здесь — то, что уже лежало в базе.

**Что сработало.** Различать «поле не прислано» и «поле явно очищено»:

- `undefined` (ключа нет в payload) → **унаследовать** прежнее значение, найдя строку по стабильному бизнес-ключу;
- `null` (ключ есть, значение пустое) → очистить намеренно.

Ключ наследования выбирали осторожно: сперва взяли `(вариант, тип_компонента, номенклатура)`, потом выкинули `тип_компонента` — у нас есть авто-фикс типа компонента по карточке номенклатуры, и он рвал бы наследование ровно в тот момент, когда тип поправился. Урок мелкий, но общий: **ключ сопоставления не должен включать поле, которое сервер сам вправе поменять в этой же операции.**

**Правило (переносимое).** Вводя поле в full-replace API с версионированными клиентами: (1) `undefined` ≠ `null` — наследование против очистки; (2) ключ наследования — из стабильных атрибутов; (3) проверять смоуком **сценарий старого клиента** — послать payload без поля и убедиться, что значение выжило. Мы это проверили явным шагом, и он оказался единственным, который поймал бы регрессию: обычный round-trip (записал поле → прочитал) зелёный в обеих реализациях.
