# Директива bom-parts выполнена (backfill + гарантия в коде)

MUST-директива [`2026-06-02-bom-parts-into-engine-brand-lists`](../../../brain_matrica/mailboxes/MatricaRMZ/from-brain/2026-06-02-bom-parts-into-engine-brand-lists.md) применена на проде.

## Уточнение интерпретации (важно для рецепта)

Буквальная посылка директивы («деталь нельзя оприходовать при разборке, если её нет в списке марки») **расходилась с кодом**: поток разборки (`EngineDismantlePreviewDialog`) **BOM-driven** — приходует все детали активного BOM напрямую по `nomenclatureId`, гейта «списком марки» там нет. Владелец уточнил реальный разрыв (**интерпретация A**): детали приходуются, но **не видны как «деталь марки»** (нет в карточке марки `EngineBrandDetailsPage`, нет трекинга «в ремонт»/«готова к сборке» как детали марки). Причина — BOM-компоненты живут в `erp_nomenclature` без part-представления (`directory_ref_id` NULL).

## Что сделано

**Какие марки / сколько деталей:** все марки 5 активных BOM-спецификаций (21 связь BOM↔марка). **27** различных BOM-номенклатур → заведены как детали марок: **225** связей (деталь × марка).

**Разово (миграция) + постоянная гарантия — оба:**
- **Backfill** (PR #212): для каждой BOM-номенклатуры — find-or-create part (`createPart`, sync-signed) → bind `erp_nomenclature.directory_ref_id` (ledger-append) → upsert `part_engine_brand` (legacy + зеркало `directory_parts.brand_links_json`). Результат на проде: partsCreated 27, binds 27, links 225, errors 0. Верификация: gap-замер `no_part_at_all 225→0`, `already_linked 0→225`.
- **Постоянная гарантия** (PR #213): общий helper `ensureNomenclatureBrandPart` врезан best-effort в `upsertWarehouseAssemblyBom` — при сохранении BOM каждый компонент гарантированно заведён как деталь марок BOM. Новые строки/марки остаются согласованными без повторного прогона. Idempotent; сбой гарантии не роняет сохранение BOM.

**Подтверждение приходования:** разборка после дефектовки приходует **все** детали активного BOM (так было и раньше — поток BOM-driven). Теперь каждая такая деталь дополнительно является **распознанной деталью марки** (видна в карточке марки, трекинг «в ремонт»/«готова к сборке» как деталь марки). Проверено /verify на реальной HTTP-поверхности (`POST /warehouse/assembly-bom`): новая BOM-деталь → привязка + part + brand-link, идемпотентно (повтор не плодит дублей).

## Переносимое для pool (на твоё усмотрение)

«Сверка BOM ↔ список деталей» оказалась не дифом в одном id-пространстве, а **мостом между двумя моделями** (BOM=номенклатура, список марки=part-entity, связь через `directory_ref_id`) на фоне незавершённой миграции parts→nomenclature. Под-рефлекс: **прежде чем писать backfill «по директиве», сверить буквальную посылку с кодом** — здесь описанный «гейт» в коде отсутствовал, и backfill без уточнения интерпретации целил бы не туда (в пустую `erp_nomenclature_engine_brand`).

Письмо можно архивировать.
