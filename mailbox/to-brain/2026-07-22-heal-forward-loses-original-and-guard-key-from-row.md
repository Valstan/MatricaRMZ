# TL;DR

Чинили две задачи владельца, по ходу вылезли три переносимых находки. Все три — про то, как «правильно закрытый» баг оставляет за собой хвост, и про то, где серверный backstop протекает даже когда он написан.

## 1. Самолечение вперёд (heal-forward) чинит инвариант, но теряет исходное значение

**Контекст.** Наряды получают номер при первой материализации (deferred-create, `max+1`). Устаревший recovery-черновик карточки нёс сентинел `workOrderNumber: 0` и при сохранении затирал присвоенный номер. Баг закрыли (PR #283): «номер строки БД побеждает payload», а если сама строка сломана (0) — присваивается свежий `max+1`.

**Находка.** Фикс восстанавливает **инвариант** («номер не ноль»), но не **значение**. Пострадавший наряд при первом же сохранении получает номер в конце нумерации (у нас 86 → 103), а его собственный номер остаётся дырой. Пока в базе стоял ноль, инцидент был виден; после самолечения он выглядит здоровым и становится ненаходимым — прежний номер живёт только в журнале аудита. Владелец видит «два наряда с нулём», а к моменту разбора один из них уже «вылечился» в неправильный.

**Мораль (переносимо).** Для полей, у которых значение — человеческий ключ (номер документа, инвентарный номер, табельный), самолечение должно быть **раздельным решением**: восстановить инвариант можно молча, а восстановление значения — только из истории (audit/ledger) и с явным следом. Как минимум: heal-forward обязан оставлять запись «было X, стало Y, X утрачен» — иначе диагностируемость инцидента сгорает вместе с симптомом. Практический приём, который сработал: исходные значения нашлись поиском по `audit_log` (`action='*.create'`, payload несёт поле) — журнал создания оказался единственным местом, где значение пережило и затирание, и самолечение.

## 2. Серверный гейт брал ключ строки из клиентского поля `row_id`, а не из `row.id`

**Контекст.** Написали field-level backstop: «номер наряда неизменяем, кроме осознанной смены суперадмином». Гейт читает сохранённое значение из БД по id строки и сравнивает с входящим.

**Находка (состязательное ревью, подтверждена).** Транспорт sync-tx несёт id **дважды**: отдельным полем `row_id` и внутри `row.id`. Запись в PG идёт по `row.id`, а гейт брал `row_id`. Клиенту достаточно послать `row_id` несуществующей строки — гейт не находит stored-значение, решает «первая материализация, не наше дело», пропускает, а запись уходит по `row.id` в защищаемую строку. Гейт есть, тесты зелёные, защиты нет.

**Мораль.** Любой серверный гейт обязан ключеваться **тем же полем, по которому произойдёт запись**, и это надо проверять явно, а не считать поля-дубликаты синонимами. Это ровно та же семья, что наша прошлая находка про подложный `attribute_defs` из того же батча (G36): гейт резолвит сущность одним путём, а запись идёт другим. Полезное правило: в тесте гейта заводить кейс, где два «одинаковых» идентификатора намеренно расходятся.

Второй экземпляр той же семьи в том же гейте: фильтрация кандидатов по `input.type === 'upsert'`. Ниже по конвейеру операция выводится заново из `row.deleted_at`, поэтому строка, помеченная клиентом как `delete`, но приехавшая без `deleted_at`, проходит мимо гейта и применяется как обычный upsert. **Не доверяй декларации намерения от клиента, если конвейер её потом пересчитывает.**

## 3. Привилегированную правку пускать по маркеру намерения, а не по одной роли

**Находка.** Первая версия гейта звучала как «номер может менять суперадмин». Но роль — это свойство **сессии**, а не операции: любой обычный push суперадмина (автосейв карточки, догон оффлайн-очереди, старый клиент) попадает под то же исключение, и гейт молча пропускает случайную смену номера — ровно то, ради чего его писали.

**Лечение.** Легальный путь (кнопка → сервис) кладёт в payload маркер осознанной правки (`auditTrail: action='number_change', note='№<новое значение>'`), гейт пускает изменение только при совпадении маркера с входящим значением. Роль остаётся необходимым условием, но перестаёт быть достаточным. Маркер ищется по всему следу, а не в последнем элементе: между правкой и push'ом карточку могли ещё раз отредактировать.

**Мораль (переносимо).** Исключение из инварианта должно опираться на **признак операции**, а не только на атрибут актора. Формула: «роль разрешает, маркер подтверждает». Иначе привилегированный пользователь превращается в дыру размером со свою сессию.

## Наш статус

Всё три учтены в PR #304 (v2026.722.1416): гейт ключуется по `row.id`, не фильтрует по клиентскому `type`, требует маркер намерения; симптом «наряд с нулевым номером / номер новый вместо своего» записан в GOTCHAS **M37** вместе с рецептом археологии по `audit_log` / `ledger_tx_index`.
