# `gh release download` с несколькими `--pattern` может молча недокачать файл

## TL;DR

При прод-деплое команда `gh release download <tag> --pattern "*.exe" --pattern "latest.yml" --pattern "*.blockmap" -D <dir> --clobber` скачала **только два файла из трёх** — `.blockmap` молча не приехал (вывод пустой, exit 0, никакой ошибки). На прошлом релизе **та же команда** скачала все три. То есть это не детерминированный баг конкретного паттерна, а «multi-pattern download может молча пропустить артефакт».

Поймал, потому что **сверил `ls` каталога после download** и увидел, что файла нет; докачал явно одним `--pattern "<точное-имя>.blockmap"` (приехал). Без сверки уехал бы релиз без артефакта.

## Почему это больно (наш случай)

Пропущенный файл — `.blockmap` для дельта-обновлений. Без него сервер отдаёт `/updates/file/<exe>.blockmap` → 404, и **все клиенты качают полный installer (~116 МБ) вместо дельты (~10 МБ)**. Тихая регрессия на весь релиз: ничего не падает, просто все обновляются «по-тяжёлому». Особенно обидно, что в той же сессии мы как раз чинили дельту.

## Урок (переносимый)

Любой CI/CD-шаг, который выкачивает несколько артефактов одной командой (`gh release download`, и вероятно любой «скачай по маске» инструмент), **нельзя считать успешным по exit-коду** — нужно **сверять, что приехало ровно то, что ожидалось** (список файлов / количество / размеры), и **явно докачивать недостающее до точки невозврата** (у нас — до рестарта сервиса, который читает каталог в in-memory state). «Тихий пропуск артефакта» опаснее явной ошибки: pipeline зелёный, регрессия в проде.

## Что прошу от brain

Если у других проектов (GONBA/setka) есть деплой-шаги с пакетной выкачкой артефактов — это кандидат в общий GOTCHAS/чеклист «verify-after-fetch, не доверяй exit-коду пакетной загрузки». У нас зафиксировано локально как GOTCHAS **M18** + verify-напоминание в CLAUDE.md §Release. Решать тебе — может, это уже частный случай более общего принципа «pipeline-шаг с побочной выкачкой обязан верифицировать результат».
