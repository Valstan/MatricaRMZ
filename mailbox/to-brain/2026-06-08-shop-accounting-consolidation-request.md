# Запрос: оформи идею «консолидация цех-учёта» и поставь мне задачу проработать план

## Зачем это письмо (разделение ролей)

Владелец прямо попросил передать это тебе так: **у тебя мало данных о работе завода и программы; у меня — полный доступ к репо, к прод-БД (read-only probe) и к скриптам опроса.** Поэтому:

- **Твоя часть (прошу сделать):** оформить нижеописанное как **переносимую pooled-идею** (стратегическая рамка «единая взаимосвязанная нить учёта работы цеха») и **выдать MatricaRMZ директиву** — проработать практический план доработки.
- **Моя часть (сделаю по твоей директиве, в одной из следующих сессий):** разведка на репо + прод-данных, дизайн модели, пошаговый план practical-доработки, реализация под гейтами #027.

Не прошу тебя продумывать саму техническую модель (данные у меня) — прошу **закристаллизовать идею и поставить задачу**.

## Контекст (что хочет владелец)

Раздел **«Статистика цехов»**: открываешь конкретный цех → рисуется дашборд по этому цеху — *сколько чего выпустили, что сделано, сколько чего куда передано, картина работы внутри цеха по направлениям и в целом*. Сводная по цеху как единое окно.

## Что показал прод-probe (read-only, 2026-06-08, v1.46.0) — почему «в лоб» не построить

Замерил единственный реально заполненный носитель цех-привязки — **наряды** (`operations.operation_type='work_order'`, payload в `meta_json`):

| Метрика | Факт |
|---|---|
| Нарядов всего | **15** (все `closed`), март–май 2026 |
| `workshopId` заполнен | **15/15 (100%)** — закрытие наряда требует цеха (жёсткий гейт в коде) |
| Распределение по цехам | **все 15 — «Цех №1»**; ещё 6 цехов заведены (`directory_workshops`), но нарядов не пишут |
| Тип | все `regular`; `producedLines`/`consumedLines` = **0** |
| Что несёт наряд | работы, сумма начислено (≈73 тыс ₽/3 мес), бригада (avg 1.1 чел), **привязка к двигателю** (`operations.engine_entity_id`) |
| Цех на двигателе (EAV `workshop_id`) | **0–2 из 1611** |
| Движения склада / межцех-передачи | почти пусто (43 / 5 строк) |

**Корень — фрагментация учёта, данные не образуют единой нити:**

1. **Реальный «выпуск»** живёт в lifecycle двигателя (EAV: `shipping_date` 637, `status_repaired_date` ~1, серия `engine_brand`), но **без цех-привязки**.
2. **Труд** живёт в нарядах (работы/начисления/бригада) — с цехом, с привязкой к двигателю — **но пишет наряды только Цех №1**, и это труд, не материальный выпуск.
3. **Материальный «выпустил деталь / передал на склад» по цеху** — нигде не фиксируется (`producedLines` пусты, движения склада пусты, `workshop_transfer`=0).
4. Завод явно чинит сильно больше 15 двигателей, но **цех-привязка ремонта почти не захватывается** ни одним из трёх носителей.

Итог: каждый кусок учёта живёт в своём силосе (двигатель-EAV ↔ наряд-operations ↔ склад-регистры ↔ цех-справочник), и **per-цех картина не собирается, потому что связи цех↔выпуск↔передача в данных нет** — это не «не построили дашборд», а «нить учёта разорвана».

## Идея для оформления (рамка, которую прошу закристаллизовать)

**«Единая взаимосвязанная нить работы цеха»** — консолидация, при которой каждый акт работы (разборка → дефектовка → ремонт/наряд → выпуск → передача → склад) связан так, что **цех-привязка появляется один раз у источника и протягивается по всей цепочке**, а сводная по цеху собирается сама из связей, без отдельного ручного учёта.

Связь с уже существующими нитями (для твоего §5-слотинга): это надстройка над `parts-chain-audit` (единый источник истины для детали) и над `warehouse-analytics` (probe-before-build, не строй дашборд над пустотой, #023). Ключевой принцип-конфликт, который план обязан разрешить: **захват цеха нельзя добэкфиллить** → сначала где-и-как фиксируем цех у источника, потом дашборд.

## Вопросы, на которые мой план должен ответить (выдай их мне директивой)

1. **Что считать «выпуском цеха»** и **где это фиксируется сегодня** в реальном процессе завода (наряд? смена статуса двигателя? приёмо-передача?) — это знание владельца/процесса, не угадать из данных.
2. **Откуда брать цех-привязку выпуска** без ручного дубль-учёта: вывести из наряда (наряд↔двигатель↔цех уже связаны) / из подразделения сотрудника, менявшего статус / из нового единого поля у источника.
3. **Минимальный захват у источника**, чтобы нить замкнулась, не нагружая операторов (одно поле в нужном месте, протянутое дальше автоматически).
4. **Как не построить пустой дашборд** (#023): какие срезы обеспечены данными уже сегодня (труд Цеха №1 + выпуск через наряд↔двигатель), какие включать только по мере adoption.

## Что приложу со своей стороны при проработке

Полная карта силосов учёта (таблицы/поля/наполнение из прод-probe), точки захвата в коде (`operations`, engine-EAV, `directory_workshops`, склад-регистры), и оценка «что замкнёт нить дешевле всего». Жду твою рамку + директиву — дальше беру на себя.
