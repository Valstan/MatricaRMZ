# Находка: pointer-жест + state = тихая потеря результата (не ошибка, а «фича не работает»)

## Что случилось

Делал marquee-выделение (рамка на холсте) в эскизнике. Реализация «как в учебнике»: `pointerdown` кладёт стартовую точку, `pointermove` пишет прямоугольник в `useState`, `pointerup` читает этот state и вычисляет попавшие элементы.

Гейт (CDP-смоук) показал: рамка не выделяет **ничего**. Ни ошибки, ни исключения — просто пустой результат, будто фича не написана. Диагностика вела не туда: сначала подозревал селектор попадания, потом синтетические pointer-события драйвера.

Причина: обработчик `pointerup` замыкается на **тот** снапшот state, что был на момент рендера. Когда события жеста приходят быстрее, чем React успевает ре-рендерить (а в смоуке они летят подряд, и у живого пользователя при быстром движении — тоже), `pointerup` видит `null`/устаревший прямоугольник. Лечение — держать данные жеста в `useRef` (state оставить только для отрисовки рамки).

## Почему считаю переносимым

1. **Класс, а не случай.** Любой жест «начало → серия move → завершение» в React (drag&drop, resize, lasso, свайп, рисование) имеет ту же ловушку: завершающий обработчик читает накопленное в state. Пишется естественно и выглядит корректно на code review.
2. **Отказ тихий.** Нет исключения, нет красной консоли — фича просто «не работает», и подозрение падает на что угодно (селекторы, события, библиотеку), но не на замыкание. Это дорогая по времени отладка при дешёвом фиксе.
3. **Гейт ловит только если он есть.** Ручная проверка «мышкой поводил — вроде выделяет» может пройти: медленный человеческий жест успевает отрендериться между событиями. Ошибка вылезает на быстрых жестах и в автотестах — то есть у пользователя изредка и «невоспроизводимо».

## Формулировка правила (если пулить)

> **Данные жеста — в `ref`, не в `state`.** Если обработчик завершения жеста (`pointerup`/`mouseup`/`touchend`) читает то, что накопили предыдущие события, храни это в `useRef`; `state` держи только для отрисовки. Симптом нарушения — жест **молча** не даёт результата (без ошибки), причём чем быстрее жест, тем стабильнее промах.

Смежное наблюдение из того же прогона (возможно, уже покрыто пулом): CDP-драйвер, шлющий все фазы жеста в **одном** `Runtime.evaluate`, воспроизводит ровно этот класс багов, но и **маскирует** корректный код — фазы стоит разносить по отдельным eval'ам с паузами, иначе смоук репортит ложный «не работает».
