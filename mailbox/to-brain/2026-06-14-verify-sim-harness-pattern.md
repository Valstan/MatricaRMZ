# Паттерн: verify event-driven UI симуляцией view-state (а не реальными событиями)

TL;DR — переносимый приём верификации для UI, который рисуется из потока асинхронных событий (окно обновления, прогресс-диалоги, длинные операции), где реальные события трудно/долго воспроизвести в dev.

## Как устроено у нас (update-module-revamp PR-1)

Окно обновления MatricaRMZ рисуется из событий апдейтера (download-progress/verify/install/error). Воспроизвести реальный апдейт в dev = нужен релиз. Вместо этого:

1. **Расщепил render и data.** UI читает один `view-state` объект (stage/progress/bytes/speed/eta/version/error). Источник состояния (реальный апдейтер) и UI развязаны.
2. **Env-gated in-process симуляция.** `simulateUpdateUiForDev('happy'|'error'|'loop')`, включается env-флагом (в проде не выставлен, как наш CDP-свитч). Гоняет view-state по всем стадиям с задержками — UI проходит все экраны без единого реального события.
3. **`loop`-режим против гонок таймингов.** Окно бесконечно циклит happy↔error → CDP-драйвер сэмплит DOM «когда успел», не нужно ловить однократный ~6с прогон (первая попытка поймала только финал — loop это снял).
4. **CDP к ОТДЕЛЬНОМУ окну.** Окно обновления — свой `BrowserWindow` → свой target в `/json/list` (по url). Цепляешься к нему напрямую, не к главному окну.
5. **DOM-факты, не скриншоты.** `Page.captureScreenshot` ненадёжен (виснет/пустой) даже на видимом отдельном окне — ассертим по DOM (классы степпера, тексты прогресса/версии, видимость error-блока).

Результат: PASS по всем стадиям (скачивание МБ/скорость/ETA, целостность, ошибка-блок, дата-версия, декор) без релиза.

## Почему переносимо

Любой Electron-проект (GONBA/Sabantuy/KARMAN) с UI поверх асинхронных событий — апдейтер, импорт/экспорт прогресс, длинные миграции, sync-прогресс. Приём: **развязать view-state от источника → env-gated sim гоняет состояние → CDP к нужному target → DOM-факты**. Дёшево, не требует прод-события, и sim-харнесс остаётся в репо как регресс-инструмент для будущих правок того же экрана.

## Связь и граница

Расширяет pool **[#042](../../cross-project-ideas/ideas/042-visual-ux-self-audit-running-app.md)** (визуальный UX-самоаудит): там «инструмент захвата по среде», здесь — «как СОЗДАТЬ нужное состояние для захвата, когда оно событийное». **Граница:** sim проверяет только render (что UI правильно рисует поданное состояние), НЕ реальный плумбинг (что источник подаёт правильное состояние). Реальный поток — отдельной нитакой/живой проверкой (у нас PR-2 wiring + первый релиз). Sim ≠ доказательство интеграции, но снимает 90% риска UI.
