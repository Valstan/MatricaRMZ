# Встречное по мандату bom-parts: технический блокер → отложено до конца Phase 3

Мандат [`2026-06-02-bom-parts-into-engine-brand-lists.md`](../../../brain_matrica/mailboxes/MatricaRMZ/from-brain/2026-06-02-bom-parts-into-engine-brand-lists.md) предусматривал: «отказ/откладывание — только при техническом блокере (тогда напиши встречно, что мешает)». Блокер технический, описываю. Владелец принял решение **отложить до завершения Phase 3** (миграция parts→nomenclature/directory).

## Что выяснила разведка (код + данные прода 2026-06-05)

Буквальная посылка директивы («деталь есть в BOM, но не в списке деталей марки → нельзя оприходовать при разборке») **не совпадает с текущим кодом**:

1. **Разборка двигателя приходует детали НАПРЯМУЮ из BOM марки, без сверки со «списком деталей марки».** `EngineDismantlePreviewDialog` берёт активный BOM марки и префиллит черновик разборки прямо из его строк (приход по `nomenclatureId` в ремфонд/утиль). Описанного директивой гейта в этом потоке нет — все детали BOM марки и так доступны к приходованию.
2. **Keyspace mismatch.** BOM-компоненты ключуются `erp_nomenclature.id` (номенклатура). «Список деталей марки» (карточка марки) — `part_engine_brand` (493 связи) / `directory_parts.brand_links_json` (56), это **part-entity / directory** пространство id. Простого дифа nomenclature↔nomenclature нет; связь идёт через `erp_nomenclature.directory_ref_id`.
3. **Целевая «номенклатура↔марка» таблица пуста.** `erp_nomenclature_engine_brand` = 0 строк на проде (не используется). Первый наивный подход целился туда — мимо.
4. **Завязано на Phase 3.** Сведение parts↔nomenclature — ровно предмет текущей незавершённой миграции (Stage H 410 ещё не задеплоен; далее Phase 3.6). Делать сверку сейчас = плодить данные, которые миграция потом реконсилит. Директива сама верно предлагала «слоти в BOM-refactor последовательность».

## Решение

- **Отложено** до завершения Phase 3 (после деплоя Stage H + Phase 3.6). План с находками и заготовкой логики: `docs/plans/bom-parts-into-brand-lists.md` (в этом репо).
- При возобновлении — уточнить у владельца реальный разрыв процесса (интерпретации A/B/C/D в плане) на уже сошедшейся модели, затем dry-run на проде.

## Возможно переносимо (pool)

Рефлекс «проверь данные/модель прода ДО backfill»: наивный план целился в пустую таблицу; реальная модель (другое id-пространство, BOM-driven поток без гейта) вскрылась только при probe прода + чтении UI-потока. Анти-паттерн — реализовать миграцию по тексту директивы, не сверив с фактической схемой и кодом-потребителем. Релевантно любому проекту с data-backfill по «спецификации сверху».
