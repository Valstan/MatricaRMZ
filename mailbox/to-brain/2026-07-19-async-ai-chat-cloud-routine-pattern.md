# Асинхронный AI-чат: очередь в БД + облачная рутина вместо API

Отгружено v2026.719.1555 (PR #277–#281, план `docs/plans/ai-chat-async-2026-07.md`).

## Переносимая находка 1: «AI-фича без Anthropic API»

Anthropic API из РФ гео-блокирован и платный. Вместо backend→API: **очередь вопросов в
БД приложения + scheduled cloud agent claude.ai (как рутина Сарафана), который по SSH
заходит на прод, читает очередь, анализирует БД read-only и пишет ответы через штатный
серверный канал записи**. Ноль API-ключей, ноль прокси, в рамках подписки. Trade-off —
латентность (раз в час), лечится UI-баннером «когда ответит ИИ» (детерминированное
расписание — pure-helper, клиент считает офлайн). Применимо в любом проекте с
клиентами + сервером, где ответы не нужны мгновенно.

Комплект паттерна: лимит N вопросов/час (server-side в транзакции push), редактирование
вопроса до ответа, `--expect-updated-at` от гонки «отредактировал после чтения рутиной»,
эскалация непонятных вопросов владельцу + вердикты копятся в самообновляемый файл правил
(«конституция ответов») в БД с append-only историей.

## Переносимая находка 2 (грабля): field-level push-guard ловит и СЕРВЕРНЫЕ записи

Написал для новой synced-таблицы push-guard, затирающий «серверные» поля из клиентских
пушей. Смоук показал: серверный писатель (`writeSyncChanges`) и **ledger-replay** идут
через тот же `applyPushBatch` → guard молча затирал легитимные ответы (ledger-seq рос,
а проекция не менялась — выглядит как «запись прошла, данных нет»). Лечение: явный
`trustedServerWrite = isReplayClient || allowSyncConflicts` bypass. Правило: любой
field-level guard в общем apply-пути обязан различать клиентский push и
серверную/replay-запись, и это надо проверять смоуком по обоим путям.
