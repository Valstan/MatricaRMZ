## TL;DR

Директива #474 реализована — весь код в `main` (M1–M4, PR #524–#527), **раскат на прод ещё не делался** (релиз + назначение ролей реальным логинам + сброс кэша — осознанный шаг под владельцем). Ниже — финальная карта логин→роль для сверки «перед закрытием доступов», как просила директива (Ф0-промежуток).

**Главная находка (важна для #015):** рекомендация brain «гейтить на серверном слое» была **не просто не выполнена — дыра была открыта настежь**. Основные сущности (двигатели/детали/заявки/наряды) пишутся через `/ledger/tx/submit`, который авторизовал **только «ты залогинен»**. Гранулярные permission-коды (`engines.edit`/`parts.create`/…) оказались **UI-only флагами** — нигде на сервере не проверялись. Любой залогиненный оператор мог записать что угодно мимо UI.

## Что сделано (две оси, server-enforced)

- **M1 — роли-пресеты:** уровень (superadmin/admin/оператор/нет-доступа) × участок. `normalizeRole` распознаёт новые ключи `engineer/technolog/master/supply/timekeeper/viewer`.
- **M2a — ledger write-гейт (ядро, ответ на #015):** авторизация в `applyLedgerTxs` по **entity_type / operation_type**, не по таблице — потому что почти всё EAV (`entities`+`attribute_values`, различие — тип). Внутренние каскады правки карточки проходят сами (все атрибуты сущности → один тип → одно право) — это снимает «ключевую сложность» владельца на уровне дизайна. Чужое → `skipped:forbidden` (батч не валится, offline-очередь цела).
- **M2b — read-ограничение:** операторы не видят чужие зарплату/ЛД (`HIDDEN_ATTRIBUTE_NAMES`), фильтр на pull-пути.
- **M3 — deny-лог:** отказы → `server.authz.denied` (login+роль) для эмпирической петли «по ходу пьесы».
- **M4 — очередь одобрения изменений пользователей:** admin предлагает add/edit → superadmin одобряет (расширение существующего `change_requests`); пароль хешируется при submit.

**Миграционная безопасность:** все гейты бьют **только новые операторские роли**; `user`(legacy)/`admin`/`superadmin` проходят всё → ноль регрессий, пока логину не назначена операторская роль.

## Переносимое (pool-кандидаты)

1. **Recon перед «закрыть доступ»: какие коды РЕАЛЬНО энфорсятся vs UI-only.** Прежде чем строить RBAC, проверь, где сервер реально проверяет право, а где это просто флаг видимости кнопки. У нас 86 кодов выглядели операционными, а на горячем write-пути не проверялся ни один. Дешёвый grep `requirePermission(<code>)` по коду до дизайна.
2. **Гейт по entity_type, не по таблице, в EAV-системах.** Когда всё — `entities`+`attribute_values`, табличная авторизация бесполезна; дискриминатор — тип сущности строки. Группировка по типу заодно бесплатно пропускает внутри-сущностные каскады (лечит дилемму «запрет B ломает A»).
3. **deny-лог как петля, а не как алерт.** Severity=warn, отдельный код, дедуп — чтобы во время раската видеть, кому реально не хватило прав, и доращивать роли по факту.

## Статус и следующий шаг

Код в `main`, гейты зелёные (shared 222 / backend 305 тестов, CI зелёный), поведение проверено через реальный HTTP (admin→заявка, пароль хеширован, approver-гейт 403). **Раскат** (релиз RBAC на прод → назначение ролей по карте выше → разовый сброс кэша операторам для read-ограничения) — отдельный осознанный шаг под владельцем, ещё не выполнен.
