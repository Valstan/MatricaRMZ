## План аудита (когда v1.22.0 уйдёт в прод)

1. Сверка терминологии стадий с фактическим кодом — `docs/PROJECT_STATE.md` + UI-меню. Список стадий из письма (контракт → приёмка → дефектация → разборка → ремонт → сборка → испытания → отгрузка) ожидаемо совпадает, но проверю наличие «промежуточных» (технологическая подготовка, согласование сметы), если они есть.
2. Тестовый прогон сценария — фейковый двигатель «End-to-end audit 2026-XX», проход через все стадии, отметки в чек-листе A–E из письма.
3. Найденные баги / недоделки — фиксируются в [`docs/PENDING_FOLLOWUPS.md`](../docs/PENDING_FOLLOWUPS.md) по приоритетам (🔴 / ⏳ / 🟡 / 🟢) либо точечные fix-коммиты, если мелкое.
4. Отчёт `mailbox/to-brain/2026-XX-XX-end-to-end-audit-results.md` (kind=feedback) по структуре из письма.

## Заметки в процессе

Пока v1.22.0 идёт — если в коде попадётся «узкое место» которое явно повлияет на финальный аудит (например, стадия плохо стыкуется с BOM) — записываю в [`docs/SESSION_HANDOFF.md`](../docs/SESSION_HANDOFF.md) или сразу в PENDING_FOLLOWUPS под пометкой «для финального аудита». Так к моменту аудита заметки будут собраны.
