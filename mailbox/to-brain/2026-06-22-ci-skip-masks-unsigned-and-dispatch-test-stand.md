## TL;DR

Две независимые находки из сессии «watchdog Phase 5» (внешний агент самовосстановления Electron-клиента), обе переносимы за пределы MatricaRMZ.

## Находка 1: зелёный CI-шаг ≠ выполненное предусловие

Написал шаг подписи бинаря в релизном workflow: `if CSC_LINK пуст → warn + skip (не падать)`. На `workflow_dispatch` шаг прошёл **зелёным** — но по логу `CSC_LINK:` пуст → подпись пропущена, бинарь **unsigned**. Полез проверять: секретов `CSC_LINK`/`CSC_KEY_PASSWORD` в репо **нет вообще**; скачал реальный прод-инсталлятор — `Get-AuthenticodeSignature` = `NotSigned`. То есть **весь прод-клиент годами раздавался без подписи**, а в плане значилось «code-signing есть».

**Урок (переносимо):** «skip-when-absent» — удобный паттерн, но он маскирует отсутствие предусловия зелёным статусом. Если что-то ДОЛЖНО происходить (подпись, загрузка секрета, публикация) — проверяй, что **предусловие реально присутствует** (секрет задан, артефакт получен), а не только что шаг не упал. Для подписи — лучше fail-loud, когда ожидается подпись, чем тихий unsigned. Возможный гейт: тест/проверка, что released артефакт реально подписан.

## Находка 2: dev-машина как Windows-тест-стенд через workflow_dispatch publish=never

Нужно было протестировать NSIS-инсталлятор (создаёт Scheduled Task) на реальной Windows — а «нет тестового стенда» казалось блокером. Развязка:
- **Сам dev-комп** (Windows 11) — и есть стенд; «нужен отдельный стенд оператора» было самоналоженным ограничением.
- Релизный workflow развил на условную публикацию: `--publish ${{ startsWith(github.ref,'refs/tags/') && 'always' || 'never' }}` — тег публикует релиз; **`workflow_dispatch` с ветки только собирает + выкладывает инсталлятор артефактом** (без GitHub Release, без раздачи). Это безопасный канал «настоящая сборка на проверку» без риска опубликовать тест на проде.
- Скачал артефакт → поставил на dev → проверил задачи/восстановление/uninstall. Полный on-machine цикл без отдельной машины и без локального тулчейна (Go собирается в CI).

**Урок (переносимо):** прежде чем назвать работу «заблокированной внешним стендом» — (а) проверь, не годится ли текущая машина; (б) сделай в CI отдельный publish=never путь для тест-артефактов, чтобы гонять настоящие сборки без публикации. (Связано с моим локальным выводом «разложи blocked на write/verify/ship».)
