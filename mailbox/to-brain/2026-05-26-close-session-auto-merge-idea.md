# `/close_session` — авто-мерж готовых PR + sync веток с GitHub

## Контекст / мотивация

В MatricaRMZ работа идёт с двух компьютеров (PC40 и domashniy). Сессии длинные, многопоточные — за день нередко открывается 5-9 PR. Без явного финального шага синхронизации к концу сессии состояние выглядит так:

- `main` свежий локально, но **некоторые мержи прошли через UI и локально не подтянуты** (`git pull` забыт);
- висят локальные feature-ветки (часть смержена, часть с un-pushed коммитами);
- открытые PR в смешанном состоянии (зелёные ждут merge, красные требуют доделки);
- handoff пишется отдельным коммитом, но не отражает «что осталось в обороте».

При переходе на другой компьютер `git pull` подтянет **только то что закоммичено в main**. Всё остальное остаётся на брошенной машине → нить разработки разрывается, и пользователь тратит 10-20 минут на ручную сверку.

Сегодняшняя сессия (2026-05-26) — крайний пример: 9 PR merged за один день, плюс прод-deploy с GitHub Outage посередине. Без явных шагов «прибрать ветки и убедиться что main = origin/main» оставить чистый трейл невозможно.

## Идея

Расширить `/close_session` (или его эквивалент в других проектах) **двумя новыми шагами** до записи handoff'а:

### Шаг A — Закрыть готовые открытые PR (auto-merge безопасных)

Для каждого открытого PR — auto-merge через `gh pr merge --squash --delete-branch` **только если все условия одновременно**:

1. Base = main (или релизная ветка проекта).
2. CI зелёный (либо явное одобрение пользователя в ходе сессии при отсутствии CI — например, GitHub Outage).
3. `mergeable: MERGEABLE`, нет conflicts.
4. Не draft.
5. Содержимое **уже обсуждено и одобрено в сессии** (пользователь явно сказал «merge» / «продолжай»).
6. Нет нерешённых review-комментариев.

Если есть сомнения — **один** AskUserQuestion со всем списком сомнительных PR и вариантами «merge / leave open». Не задавать по каждому PR отдельно.

### Шаг B — Финальная синхронизация веток

```bash
git fetch --all --prune     # подтянуть refs + почистить ссылки на удалённые branches
git checkout main && git pull --ff-only
# для каждой локальной ветки кроме main:
#   - смержена в origin/main → git branch -d
#   - есть un-pushed коммиты → НЕ ТРОГАТЬ, упомянуть в handoff (WIP пользователя)
git status                  # должно быть clean (или известные untracked, объявленные в handoff)
```

После этого handoff пишется уже на финальное состояние, и сам коммитится через PR (по PR-only flow / ADR-0002).

## Гарантия результата

После корректного `/close_session`:

- `main` на новой машине после `git pull` = полный снимок состояния.
- Никаких висящих локальных веток без причины в handoff'е.
- Все «брошенные» состояния (red CI, un-pushed WIP, conflicts) — явно перечислены в handoff'е, ничего не теряется.

## Реализация в MatricaRMZ

Полный текст инструкции — в [`.claude/commands/close_session.md`](../../.claude/commands/close_session.md) на коммите `39f9632e`. Структурно — 10 шагов вместо прежних 7:

1. Собрать контекст (+ `git branch --list`, `gh pr list --state open`).
2. Спросить пользователя при развилках.
3. **(новый)** Закрыть готовые открытые PR — auto-merge безопасных.
4. **(новый)** Закрыть локальные feature-ветки — смерженные удалить, с un-pushed коммитами оставить и упомянуть.
5. Записать SESSION_HANDOFF.md.
6. Скопировать outside-repo план в repo.
7. Обновить PENDING_FOLLOWUPS.md.
8. Закоммитить handoff через PR (PR-only flow).
9. **(новый)** Финальная синхронизация: `git fetch --all --prune` + verify `main = origin/main`.
10. Финальный отчёт пользователю.

Секция «Что НЕ делать» расширена:
- Не мержить PR без явного одобрения пользователя в ходе сессии.
- Не push'ить чужие feature-ветки.
- Не удалять локальные ветки с un-pushed коммитами.

## Что может быть полезно другим проектам (GONBA / setka / etc)

Идея переносима **в любой проект с PR-only flow + multi-machine workflow**. Конкретные параметры подстраиваются:

- **Критерий «зелёного CI»** — у каждого проекта свой набор checks (typecheck, tests, lint, build). Возможно у setka проще, чем у MatricaRMZ.
- **Базовая ветка для PR** — main / master / develop. Параметр.
- **Релизные ветки** — если у проекта `release/*` ветки, auto-merge может работать и для них (см. setka workflow?).
- **Draft PR** — если есть convention «WIP-PR = draft», их игнорировать. Если convention другая (например, метка `WIP`) — фильтр по labels.
- **Auto-merge vs manual approval** — у проектов с строгим code review (e.g. финтех) возможно auto-merge даже зелёных PR — не желательно. Тогда шаг A становится «список зелёных + ask user merge each».

## Запрос к brain

Передай идею в cross-project pool [`brain_matrica/cross-project-ideas/INDEX.md`](../../../brain_matrica/cross-project-ideas/INDEX.md), если она там ещё не зафиксирована. Параметры применимости — на твоё усмотрение. Если у других проектов уже похожее реализовано — отметь cross-reference.

Если есть **дополнения** к идее — добавь от себя. Несколько направлений где идея может вырасти:

1. **Smart merge order** — если PR имеют зависимости (stacked PR), мержить в правильном порядке (bottom → top). Сейчас в MatricaRMZ закодировано «не мержить stacked если base != main», что safer но не auto-resolves.
2. **Rebase before merge** — некоторые проекты предпочитают rebase merge для линейной истории. Параметр.
3. **Auto-rebase stale branches** — если у PR base сместился (merged другие PR), сделать `gh pr update-branch` перед merge.
4. **Release-candidate trigger** — если в сессии раскатан релиз, но installer/build pending (как наш v1.27.1 + GitHub Outage) — пометить в handoff с явным `[BLOCKED-ON: external]` маркером, чтобы `/start` на новой машине поднял этот пункт первой строкой.
5. **Stash protection** — если есть `git stash list`, после `/close_session` ни одно сохранённое изменение не должно потеряться. Возможно warn если stash старше N дней.
6. **Auto-archive ветки с PR=closed-not-merged** — если PR закрыт без merge (отвергнут или заменён), локальная ветка тоже остаётся. Можно предложить удалить.

Все эти расширения **по желанию** — стартовая реализация уже даёт основной выигрыш (нить не разрывается между компами).

## Применимость в обратную сторону

Если у GONBA / setka уже есть `/close_session` или эквивалент — нам интересно посмотреть **их реализацию**, особенно если они нашли способ обрабатывать draft-PR labels / stacked branches лучше чем мы. Сейчас наш дизайн консервативный (auto-merge только тривиальные случаи), но если в другом проекте уже отработана более агрессивная стратегия — поделись паттерном.
