# Ф4 построена: финал дефект-статусов + претензия заказчику

PR `feat/engine-acts-phase4-customer-claim`. Гейты #027: build shared+ledger, `-r typecheck`, `-r lint`, backend tests (248), shared tests (153 + 12 новых), CDP `/verify` PASS (`cdp-acts-phase4.mjs`, дельта-ассерты).

## Что сделано (рамка owner-answers, HOW моё)

1. **Триггер «дефект»** — ветка восполнения активна при `scrap>0 || replace>0` (`rowDefectQty`/`rowHasDefect`), не только при «заменить»: утиль тоже выводит деталь из двигателя. `rowGoesToPurchase` сознательно не тронут — в закупку идут только единицы «заменить новой» (утиль закупкой не восполняется; для него ветка значит «заказчик»/«свой ремонт»).
2. **Претензия = третий версионируемый акт** (`claim_act`), на носителе Ф2 без нового механизма: `computeCustomerClaim` (строки ветки `customer` с дефектом, `claim_qty = scrap + replace`) + печатная форма с двумя секциями — дефекты к восполнению заказчиком и недостача комплектности (reuse `computeInventoryShortage`). Недостача в *позиции* претензии не входит — отдельная секция (держит рамку «претензия = ветка customer по дефекту», некомплект идёт как естественное основание без расширения модели).
3. **Сигнатура снимка** дополнена `replenishment_branch` — латентная дыра Ф2/Ф3: смена ветки не плодила новую версию при печати, для претензии это было бы ложным дедупом.
4. Акт дефектовки получил печатную колонку «Восполнение».

## Наблюдения для библиотеки

- **G47 подтвердилась дословно** на старте verify: `/health` 200, реальный запрос (`/auth/login-options`) — failed query, PG лежал. Диагностика по твоему рецепту заняла секунды.
- **Новая грабля verify-стека (кандидат в GOTCHAS):** `stop.ps1`/ручной kill `electron.exe` не убивает **node-обёртку `electron-vite dev`** — она авто-респавнит electron и держит Vite-порт; PID-файл указывает на cmd-обёртку, и повторный `start-electron.ps1` выходит по «already running». Симптомы зомби-стека: CDP то жив, то мёртв, evaluate висит. Лечение: киллить по `Win32_Process.CommandLine -match 'electron-vite|vite|electron-app'` (кроме backend) перед стартом.
- **`Page.captureScreenshot` через CDP флакает при перекрытом/свёрнутом окне** (композитор не отдаёт кадр) — в драйвере скриншоты сделаны нефатальными (20s timeout + `Page.bringToFront`), вердикт держат DOM-ассерты и захваченный print-HTML, не пиксели.
- **Дельта-ассерты в CDP-драйверах:** `operations` персистентны между прогонами — абсолютные ожидания («версий: 1») ломаются на втором прогоне. Счётчики версий проверять как `base+N`.
