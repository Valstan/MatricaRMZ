# zod-strip drift: новое поле контракта молча теряется на route-слое

## TL;DR

Расширили доменный тип (`PartSpecBrandLink` + два optional-флага), хранение — JSON-passthrough колонка (без DDL), сервис сериализует verbatim, клиент шлёт verbatim. Всё типобезопасно, все юнит-тесты зелёные. А флаги в БД не появлялись: **между клиентом и сервисом стоит zod-схема Express-роута, которая по умолчанию (strip mode) молча выбрасывает неизвестные ключи**. Нашлось только живым CDP e2e (галочка в UI → re-read из БД → поле исчезло).

## Как устроено у нас

- Симптом: UI-галочка «сохраняется» (нет ошибок), повторное чтение — поля нет; load≈0 признаков нет вообще.
- Диагноз за один шаг: direct-probe мимо UI (bridge-вызов write→read) показал, что теряется на сервере; grep route → `z.object({...})` со старым списком полей линка.
- Фикс тривиален (добавить поля в схему + у нас ещё exactOptionalPropertyTypes-маппинг), найти — нетривиально.

## Почему переносимо

Класс: **любой проект, где (а) контракт расширяется через optional-поля в JSON-хранилище «без миграций» и (б) на пути стоит strict-валидатор (zod/joi/ajv removeAdditional/protobuf-фильтрация)**. Чем «дешевле» расширение модели (наш случай: JSON-колонка, никакого DDL), тем выше шанс забыть слой-вахтёр посередине — он-то задеклан давно и в diff не попадает.

Рецепты:
1. **Чеклист расширения контракта**: «новое поле прошло ВСЕ слои? тип → клиент-сериализация → route-валидатор → сервис → хранилище → обратное чтение». Слой-валидатор — самый забываемый, потому что не падает, а молча чистит.
2. **Live round-trip probe как гейт** для «безмиграционных» расширений: write через реальный стек → read → assert поле на месте. Юнит-тесты мокают route-слой и класс не ловят принципиально.
3. Альтернатива по вкусу: `.passthrough()`/`.strict()` осознанно — strict хотя бы падает громко вместо молчаливой чистки (мы оставили strip + расширили схему: passthrough на публичном API не хотим).
