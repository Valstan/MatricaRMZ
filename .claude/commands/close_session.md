Закрытие сессии разработки в MatricaRMZ. Цель — оставить **explicit pointer** «куда мы шли» так, чтобы следующая сессия (на этом или другом компе, с тобой или другим Claude) могла мгновенно подхватить работу. **И синхронизировать всё через GitHub**, чтобы при переходе на другой компьютер не было разрыва нити: на новой машине достаточно `git pull` чтобы получить актуальное состояние.

## Когда вызывать

- Перед паузой в работе (конец рабочего дня, переход на другой проект)
- После значимого milestone (релиз раскатан и пользователь хочет проверить вживую перед следующим шагом)
- Перед сменой компа (закрыть на одном — открыть на другом через git pull)
- Перед длительным перерывом (отпуск, переключение приоритетов)
- **НЕ вызывать** — после одиночного hotfix без хвостов, после полностью завершённой и зафиксированной задачи (нитки нет)

> **Разгрузка (мандат владельца 2026-07-15, token economy).** `/close_session` — лёгкая фиксация состояния, не марафон. **Быстрый путь:** если единственное изменение сессии — handoff-доки (+ PENDING, если правился) — сделай **один** docs-only PR, auto-merge, **один** прогон sync-гейта. Не пересобирай, не перечитывай всё дерево, не создавай второй PR. Никаких build/typecheck внутри close (даже «на всякий случай проверить сборку»). Релиз через `/reliz` уже включает закрытие сессии (fusion) — после `/reliz` этот скил **не** вызывать.

## Алгоритм

### 1. Собрать контекст последней работы

Параллельно (read-only):

- `git log --oneline -10` — последние коммиты этой сессии
- `git status --short` — что не закоммичено
- `git stash list` — отложенные правки
- `git tag --sort=-v:refname | head -3` — последние теги
- `git branch --list` — локальные ветки (что есть кроме main)
- `gh pr list --state open --limit 20` — открытые PR
- TaskList — текущее состояние задач этой сессии
- Прочитать текущий `docs/SESSION_HANDOFF.md` (если есть) — что было заявлено в начале сессии, чтобы сравнить факт vs план
- Прочитать `docs/PENDING_FOLLOWUPS.md` — что должно быть обновлено

### 2. Спросить пользователя (если контекст не очевиден)

Если из коммитов и TaskList понятно «нитка ACTIVE, следующее = X» — не спрашивай, сразу пиши handoff. Если развилка — короткий вопрос пользователю:

- **Какая нитка активна для следующей сессии?** (если их несколько)
- **Что считается «следующим шагом»?** (если задача завершена и нитки нет — пиши `Status: IDLE`)
- **Есть ли скрытый контекст?** (открытые вопросы, гипотезы, что не доделано)

Используй `AskUserQuestion` для развилок, не задавай больше 2 вопросов.

### 3. Закрыть готовые открытые PR (auto-merge безопасных)

**Цель — оставить main с актуальным состоянием, чтобы на другом компе после `git pull` всё было готово к продолжению.**

**Один вызов `gh pr list --state open` — и только если он что-то вернул, работай по нему. Пусто → пропусти §3 целиком, без повторных сканов.** Для каждого открытого PR:

#### Можно мержить автоматически (squash + delete branch), если все условия одновременно:

1. Base = `main`.
2. `gh pr checks <num>` — все checks `pass` (либо CI вообще не нужно — например, если идёт GitHub Actions outage и checks отсутствуют, и при этом локально typecheck + tests зелёные, и пользователь явно одобрил merge в ходе сессии).
3. `gh pr view <num> --json mergeable,mergeStateStatus` — `mergeable: "MERGEABLE"`, никаких conflicts.
4. PR не в `draft` state.
5. Содержимое PR было обсуждено и одобрено в ходе сессии (пользователь сказал «merge» или «продолжай» по нему).
6. Нет нерешённых review-комментариев.

#### НЕ мержить автоматически (оставить открытым + упомянуть в handoff):

- CI failed или pending без явного одобрения пользователя.
- Conflicts с main.
- Draft.
- Открыт другим пользователем (не Claude в этой сессии).
- Пользователь явно сказал «оставь открытым» или «доделаю на другом компе».
- Stacked PR с base != main (сначала смержить base).

#### Если есть сомнения — спроси пользователя одним вопросом с конкретным списком PR и вариантами «merge / leave open». **Не задавай по каждому PR отдельно.**

После merge: `git checkout main && git pull --ff-only`, чтобы локальная main была на актуальном `origin/main`.

### 4. Закрыть локальные feature-ветки

**Один вызов `git branch --list` — и только если кроме `main` что-то есть, работай по нему. Иначе пропусти §4.** Для каждой локальной ветки кроме `main`:

- **Уже смержена в `origin/main`** (`git branch --merged origin/main` показывает её): удалить локально (`git branch -d <name>`).
- **Не смержена, но без open PR**: проверить `git log main..<branch>` — есть ли уникальные коммиты.
  - Если коммитов нет — удалить.
  - Если есть un-pushed коммиты → это потенциально WIP пользователя, **не трогать**. Упомянуть в handoff: «локальная ветка X с N неотправленными коммитами».
- **Не смержена, есть push'нутая копия на origin, но нет PR**: оставить как есть, упомянуть в handoff.

### 4.5. Шеринг находки в мозг (условный) — pool #009

Родилась ли в этой сессии значимая **переносимая** находка (скилл / фича / паттерн / решённая нетривиальная боль)? Прогони через фильтр. Если **да** — напиши `mailbox/to-brain/YYYY-MM-DD-<slug>.md`. Если нет — **ничего не пиши** (тишина = норма, по умолчанию молчим).

**Фильтр (слать, только если выполнены ВСЕ три):**

1. **Значимость** — новый скилл/фича/паттерн/решённая нетривиальная боль, не рутинный фикс/бамп/локальный рефактор/релиз.
2. **Переносимость** — применимо за пределами домена MatricaRMZ (другой проект мог бы переиспользовать).
3. **Неочевидность** — без этой работы сам бы не знал; есть «урок».

Любое «нет» → молчим.

- ✅ Делиться: «CODEBASE_MAP для cold-start», «изоляция deploy-key», новый CI/деплой-паттерн, обход бага инструмента, удачный приём token-economy.
- ❌ Молчать: «пофиксил расчёт BOM», «обновил Drizzle», «отрефакторил Electron-окно», «выпустил v1.21.x».

**Формат письма** (только в свой репо, brain прочитает на своём `/start` — асимметрия ADR-0001): frontmatter `from: MatricaRMZ`, `to: brain`, `date`, `kind: idea`, `topic`, `compliance`, `urgency`; тело — TL;DR / как устроено у тебя / почему переносимо / что просишь от brain. См. [pool #009](../../../brain_matrica/cross-project-ideas/ideas/009-share-findings-reflex.md). Письмо войдёт в тот же handoff-PR (§8) либо отдельным `chore/mailbox-*`.

### 5. Записать `docs/SESSION_HANDOFF.md`

Перезаписать **целиком**. Структура:

```markdown
# Session Handoff

> Sticky-note для непрерывности... (короткая шапка из шаблона)

**Status:** ACTIVE | IDLE
**Updated:** YYYY-MM-DD (Claude session, на машине <name>)
**Branch:** <git current branch>
**Last released version:** <vX.Y.Z на проде>

## Текущая нитка
1-3 предложения: что делаем, в какой стадии, почему это длинная задача.

## Следующий шаг
Конкретно: первое действие новой сессии. С file paths и командами, чтобы можно было выполнять без додумывания.
⚠️ **Сверить с реальностью перед записью:** не копировать «крупный кандидат» из прошлого handoff'а вслепую — проверить по `PENDING_FOLLOWUPS` (открыто ли ещё) и `COMPLETED.md` / git-тегам (не отгружено ли). Устаревший next-step = «всплытие сделанного» (инцидент 2026-06-16: Phase 3/T3 числились «к выкату», будучи на проде).

## Контекст
- План: <docs/plans/X.md или ссылка outside-repo с пометкой>
- Связанные коммиты: <hashes + одной строкой>
- Прод: <версия, deploy status>
- Открытых PR: <link или «нет»>
- Локальные ветки с un-pushed коммитами: <list или «нет»>

## Открытые вопросы для пользователя
- <list, если есть>

## Не забыть (low-priority)
<список напоминалок которые ещё не дотягивают до PENDING_FOLLOWUPS>
```

Если статус IDLE — заполнить только шапку и оставить пустые секции с маркером `_n/a_`.

### 6. Если есть outside-repo план — скопировать в repo

Если в `Контекст → План` упоминается `~/.claude/plans/X.md` — предложить (или сразу сделать, если auto mode) скопировать в `docs/plans/<slug>.md`. Outside-repo путь оставить только если пользователь явно отказался — с пометкой «не виден на другом компе».

### 7. Поддержать раскол «открытое vs сделанное» ([memory-reorg](../../docs/plans/memory-reorg-2026-06.md))

Рабочие файлы держат **только открытое**; завершённое → тонкий индекс + git/PR. При закрытии:

- **`docs/PENDING_FOLLOWUPS.md`** — оставить **только открытое**. Закрытый в этой сессии пункт **не вычёркиваем «зачёркиванием», а удаляем** и переносим **одной строкой** в `docs/COMPLETED.md` (см. ниже). Новые техдолги — добавить в правильный приоритет (🔴 / ⏳ / 🟡 / 🟢) + метку `🗓 since:YYYY-MM-DD`. **Не копить ✅-историю в PENDING.**
- **`docs/COMPLETED.md`** — если в сессии **завершена нитка/релиз**, добавить 1 строку в нужную секцию (newest сверху): `<✅дата> | Нитка | Релиз/PR | итог одной строкой`. **Без описаний** — granular в git/PR. Это ledger «не гонять повторно».
- **`docs/plans/_archive/`** — если завершён план из `docs/plans/<slug>.md` — `git mv` его в `docs/plans/_archive/<slug>.md` (история сохраняется) и сослаться из строки COMPLETED.
- **`docs/GOTCHAS.md`** — если в сессии словил **повторяемую граблю** (симптом → корень → лечение), добавить запись (Tier-1 индекс `M##` + Tier-2 блок). Кросс-проектную граблю — не сюда, а письмом в brain (§4.5).

### 7.5. Журнал эффектов программы (завод-контур)

Если в сессии **отгружена функциональность** (релиз на проде или смержена фича, которая уедет ближайшим релизом) — добавить 1-3 строки в [`docs/zavod/PROGRAM_EFFECTS.md`](../../docs/zavod/PROGRAM_EFFECTS.md) (новое сверху): дата | версия | модуль | **зачем сделано → что улучшило для производства**. Писать смысл/эффект на производственном языке (как скажется на работе операторов/бригад/ППО), не техдетали и не текст задания. Рутинные фиксы, рефакторинги, инфраструктура, доки — НЕ попадают (тишина = норма). Это мост dev-потока в завод-режим `/zavod` — консультант по производству читает этот журнал вместо dev-гущи.

### 8. Закоммитить handoff через PR (PR-only flow, ADR-0002)

Прямой push в `main` запрещён политикой репо. Стандартный путь:

```bash
git checkout -b chore/session-handoff-<short-slug>
git add docs/SESSION_HANDOFF.md  # + PENDING_FOLLOWUPS.md если правился
git commit -m "chore(session): handoff — <task short name>

<2-3 строки описания>

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>"
git push -u origin chore/session-handoff-<short-slug>
gh pr create --base main --head chore/session-handoff-<short-slug> \
  --title "chore(session): handoff — <short>" \
  --body "Закрытие сессии YYYY-MM-DD. <одно предложение про нитку>."
gh pr merge <num> --squash --delete-branch
git checkout main && git pull --ff-only
```

**НЕ коммить handoff вместе с другими изменениями** — handoff живёт отдельным PR'ом для прозрачности истории.

**Исключение:** если в `git status` есть untracked / modified файлы, которые НЕ относятся к handoff'у (например, нечайно сгенерированные `dist/`, или WIP-правки пользователя) — НЕ добавлять их в коммит. Упомянуть в handoff'е «непокоммиченные WIP-изменения в файлах X, Y» и предупредить что они останутся локально.

### 9. Финальная синхронизация и проверка

Убедиться что после `/close_session` следующий компьютер сможет «подхватить как есть» одной командой `git pull`:

```bash
git fetch --all --prune        # подтянуть актуальные refs + почистить ссылки на удалённые ветки
git status                     # должно быть clean кроме известных untracked (если они объявлены в handoff'е)
git log --oneline -3           # верхний commit = handoff
```

Проверки:

- `main` находится на свежем `origin/main` (`git status` не отстаёт от remote).
- Все feature-ветки которые мы смержили — удалены и локально, и на origin (`gh pr merge --delete-branch` это уже делает).
- Если есть локальные ветки с un-pushed коммитами — упомянуты в handoff.
- Если на проде что-то не доделано (например, installer pending) — это явно указано в «Следующий шаг» handoff'а.

### 9.5. Жёсткий sync-гейт (brain #010 — обязательно)

**Сессия не считается закрытой, пока вся работа не на GitHub.** GitHub — источник истины между машинами; авто-архивация сессии (тумблер Cowork «Classify session states») может оставить незапушенную работу → version-skew на другом компе.

После всех merge'ей и `git pull --ff-only` прогнать гейт:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File "scripts/git_sync_check.ps1" -Gate
```

- **`exit 0`** — дерево чистое и всё запушено → сессия закрывается, переходим к финальному отчёту.
- **`exit 1`** — есть незакоммиченное или незапушенное → **не закрывать сессию**. Закоммитить/запушить через PR-flow (ADR-0002: ветка → push → PR → merge), затем прогнать гейт **ещё один раз**. **Один прогон на состояние, не циклом «на всякий случай»:** зелёный — закрываем; повторный прогон только после реального fix'а.

Гейт живёт здесь, в команде, **а не в хуке**: хуки при авто-архиве не срабатывают надёжно (прямое указание директивы #010). SessionStart-хук в `.claude/settings.json` лишь предупреждает (`-Warn`) на входе в сессию — он не блокирующий.

### 10. Финальный отчёт пользователю

Короткий (6-10 строк), на русском:

- Что закрыто в этой сессии (1 строка).
- Сколько PR смержено в рамках `/close_session` (если > 0).
- Handoff сохранён в `docs/SESSION_HANDOFF.md` ✅ (с указанием PR / hash).
- Состояние main: `<hash>` синхронизирован с origin/main.
- Что подхватит `/start` в следующей сессии: «<Текущая нитка>, следующее = <Следующий шаг>».
- Если остались открытые PR / un-pushed ветки — перечислить с причиной почему не смержены.
- Если есть открытые вопросы для пользователя — напомнить, что ответ ждётся в следующей сессии.

## Что НЕ делать

- Не вызывать `/close_session` если сессия — это короткая консультация без работы с кодом (handoff = инструмент непрерывности, не журнал общения).
- Не дублировать в SESSION_HANDOFF содержимое PENDING_FOLLOWUPS — handoff **ссылается** на конкретный пункт PENDING.
- Не вести historic log в самом handoff — он перезаписывается. История уже есть в `git log` (в т.ч. `git log -- docs/SESSION_HANDOFF.md`).
- Не запускать прод-команды, билды или тесты в рамках `/close_session` — это инструмент фиксации состояния, не финализации работы. **Усилено (2026-07-15): и не «на всякий случай проверить сборку/typecheck» — никаких build/typecheck внутри close вообще.**
- Не делать `git rebase` / `git reset --hard` / `git push --force` — handoff коммит обычный.
- Не мержить PR без явного одобрения пользователя в ходе сессии. Если непонятно — спросить.
- Не push'ить чужие feature-ветки или ветки с непонятным содержанием — оставить, упомянуть в handoff'е.
- Не удалять локальные ветки с un-pushed коммитами — могут быть WIP пользователя.
