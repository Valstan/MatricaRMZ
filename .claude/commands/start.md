Начало новой сессии разработки в MatricaRMZ. Цель — собрать актуальное состояние и доложить пользователю, **ничего не меняя в коде**.

Выполни в указанном порядке:

## 0. Mailbox от brain_matrica

**Это самый первый шаг — раньше всего остального.** brain_matrica — meta-репо стратегического управления (`../brain_matrica/`). Связь с ним по ADR-0001 ([асимметричный mailbox-протокол](../../../brain_matrica/adr/0001-brain-projects-mailboxes.md)).

**Схема асимметричная:** каждая сторона пишет только в свой репо.
- brain пишет директивы → `brain_matrica/mailboxes/MatricaRMZ/from-brain/` (мы только читаем).
- MatricaRMZ пишет ответы → `mailbox/to-brain/` **этого** репо (brain только читает).
- Запись в `brain_matrica` из проектной сессии запрещена (исключение — clone bootstrap при отсутствии локального).

### Входящие (read-only из brain_matrica)

1. Подтяни актуальный brain: `git -C ../brain_matrica pull --ff-only origin main`.
   - Если ветка разъехалась или dirty working tree — не делай rebase/merge сам, доложи пользователю и продолжай с тем что есть локально.
   - Если `../brain_matrica/` отсутствует — упомяни в отчёте «brain_matrica не клонирован рядом»; пропусти сканирование.
2. Просканируй `../brain_matrica/mailboxes/MatricaRMZ/from-brain/*.md` — **только корень**, не `DRAFTS/`, не `ARCHIVE/`.
3. Если письма есть — доложи их пользователю **до §1** в формате:
   ```
   📬 N писем от brain_matrica:
   - [urgency COMPLIANCE] slug — topic
   ```
   - `urgency` — из frontmatter (`low` / `normal` / `high`).
   - `COMPLIANCE` — для `kind=idea` и `kind=directive`: из frontmatter (`MAY` / `SHOULD` / `MUST`); если поля нет — retroactive по [ADR-0001 v2](../../../brain_matrica/adr/0001-brain-projects-mailboxes.md#compliance-levels): `directive` → `MUST`, `idea` → `SHOULD`. Для `kind=question/feedback/report` поле compliance пропускается.
   - Любое `urgency: high` поднимай отдельной строкой даже если письмо одно.
4. **Не обрабатывай директивы сейчас.** Только доклад. Применение — отдельным шагом по явному OK пользователя или сразу если compliance=mandate и задача того требует.
5. Если активная нитка сессии (§1) важнее (deep flow, релизный цикл) и письма не `high` — после доклада спроси: «обработать почту сейчас или после релиза?».

### Исходящие (write only в свой репо)

Если в сессии нужно ответить brain'у (acknowledgement директивы, идея, вопрос, feedback):

1. Создай файл `mailbox/to-brain/YYYY-MM-DD-slug.md` **в этом репо** с корректным frontmatter (`from: MatricaRMZ`, `to: brain`, `kind`, `compliance` если applicable, `urgency`, опционально `ref`).
2. Закоммить в свой PR (либо в тематическом, либо отдельным `chore/mailbox-*`). brain прочитает через `git pull` твоего репо.
3. **Никаких `git -C ../brain_matrica add/commit/push` из этой сессии.** Архивацию и обработку держит у себя brain.

### Что НЕ делать

- ❌ Не писать и не коммитить в `../brain_matrica/` (никакие файлы, включая `.last-seen` и `mailboxes/`).
- ❌ Не архивировать письма brain'а у него — это его работа в его репо.
- ❌ Не push'ить в чужой clone.

## 0.5. Машинный профиль (этот компьютер)

**Раньше git-операций и любых запусков инструментов/скиллов.** Разные компьютеры настроены по-разному (порты dev-стенда, пути к инструментам, состояние скиллов) — без профиля нейросеть тратит токены, заново нащупывая «как тут всё запускается» и путаясь в портах. Профиль это устраняет.

1. Определи hostname: `hostname` (или `$COMPUTERNAME` на Windows) — это ключ профиля.
2. Прочитай `docs/machines/<hostname>.md`:
   - **Есть** → загрузи в рабочий контекст: порты (PG / backend dev / vite / CDP), пути к инструментам (psql и т.п.), как поднимается `verifier-electron` и другие скиллы, машинные грабли. **Используй эти значения вместо повторного нащупывания.** В §6 одной строкой: «машина: `<hostname>` — профиль загружен (PG `<порт>`, …)».
   - **Нет** → новый/незаведённый комп. В §6 отметь «машина: `<hostname>` — профиля нет, заведу по мере настройки». Не блокируйся; при первой же настройке инструмента создай файл из шаблона `docs/machines/README.md`.
3. **Пиши в профиль по мере того, как узнаёшь** (в течение сессии, не только на старте): когда впервые выяснил/установил, как на ЭТОМ компе запускается инструмент или скилл (порт, путь, последовательность команд, ABI-нюанс, рабочая/нерабочая команда) — добавь/обнови строку в `docs/machines/<hostname>.md`. Это снимает повтор нащупывания в следующих сессиях.

**Правила профиля** (детали и шаблон — в [`docs/machines/README.md`](../../docs/machines/README.md)):
- Один файл на hostname (`docs/machines/<hostname>.md`); правит его **только** этот комп → межмашинных merge-конфликтов нет. Коммитится обычным PR-flow (как прочие доки) → переживает реклон.
- **Без секретов** (паролей/ключей/токенов) — только «как запускать» (порт, путь, команда). Пароли остаются в `*.env*` (gitignored).
- Это профиль ОКРУЖЕНИЯ, не проектного состояния (то — в `SESSION_HANDOFF`/`PENDING_FOLLOWUPS`). Сюда — специфика железа/настройки именно этого компа.

## 1. Подхватить нитку из прошлой сессии

**Это второй шаг — раньше git fetch, раньше чтения остальной документации.**

1. Прочитай `docs/SESSION_HANDOFF.md` (если есть).
2. Если `Status: ACTIVE` и `Updated:` не старше 7 дней:
   - В самом верху итогового отчёта (шаг 6) выдели блок **🧵 Прошлая сессия оставила нитку**: «Текущая нитка» + «Следующий шаг» из handoff.
   - Если в handoff упомянут план в `docs/plans/...` — прочитай его (или хотя бы Context-секцию), чтобы понимать большой план куда вписывается следующий шаг.
   - В финальном вопросе пользователю спроси: **«Продолжаем нитку?»** вместо общего «Чем займёмся?».
3. Если `Status: ACTIVE` но `Updated:` старше 7 дней — отметь «handoff устарел, возможно нитка не актуальна — подтверди или начнём с нуля».
4. Если `Status: IDLE` или файла нет — просто пропусти этот шаг, дальше как обычно.
5. Если в handoff есть «Открытые вопросы для пользователя» — напомни их в отчёте, ответы ждутся в этой сессии.

## 2. Документация — узкий cold-start (token economy)

Принцип: читай минимально-необходимое, остальное — по требованию задачи. Это ADR-0003 brain'а («карта прежде разведки»).

**Обязательно читать всегда:**

1. `CLAUDE.md` — правила проекта (язык общения, code style, релизный процесс, архитектурные решения).
2. `docs/CODEBASE_MAP.md` — куратируемая карта монорепо. **Заменяет** широкое чтение `docs/` или ad-hoc `Glob/Read` «на ощупь». Из карты видно где живёт X и когда туда лезть.

**Читать по требованию активной нитки / задачи** (НЕ безусловно):

- `docs/PENDING_FOLLOWUPS.md` — **только открытое** (техдолги/отложенное) — если нитка про open issues, или пользователь спрашивает «что в работе». Завершённого там нет (выпилено). В простом случае достаточно `SESSION_HANDOFF.md`.
- `docs/COMPLETED.md` — done-индекс (Tier-1). Глянуть, **если нужно проверить «это уже сделано?»** (или прежде чем предлагать «крупный кандидат» из handoff — свериться, не отгружено ли). Не читать безусловно.
- `docs/GOTCHAS.md` — грепать по симптому **перед долгой отладкой** знакомо-выглядящей проблемы. Не читать на старте.
- Исторический контекст (когда что выпускалось) — `git log` + тело PR; навигация уровня ниток — `docs/COMPLETED.md`.
- `docs/PROJECT_STATE.md` — если задача про архитектуру / прод / релизный контур / SSH / инфраструктуру.
- `docs/plans/<slug>.md` — если активная нитка ссылается на план (`SESSION_HANDOFF.md` обычно даёт ссылку). Читать **только релевантный раздел**, не весь план. Завершённые планы — в `docs/plans/_archive/`.
- Локальная `memory/` (`MEMORY.md` index + файлы) — **только** USER-преференции/feedback (НЕ проектный архив). Если задача затрагивает преференции пользователя или прошлые корректировки.

**Не читать «для понимания общей картины»** — карта `CODEBASE_MAP.md` уже даёт обзор. Если она не отвечает на конкретный вопрос — точечный Read / Grep по нужному файлу, не серия «давайте посмотрим всё подряд».

**Не читать `docs/zavod/`** — это завод-контур (режим `/zavod`, см. CLAUDE.md «Два режима проекта»). Исключение: точечный взгляд в `docs/zavod/FACTORY_MODEL.md`, если строимая фича касается описанного там производственного процесса.

## 3. Синхронизация с GitHub (обязательно)

Это первое, что нужно сделать — иначе будешь смотреть на устаревший snapshot.

1. `git fetch --all --tags` — подтянуть все новые коммиты и теги с origin. Если ругается `would clobber existing tag` — однократно `git fetch --tags --force` (теги на origin — источник истины, локальные расхождения почти всегда ошибочные).
2. `git status` — есть ли незакоммиченные изменения, untracked файлы.
3. `git stash list` — есть ли отложенные правки. Если есть — упомянуть в отчёте, не трогать.
4. Сравни локальную ветку с `origin/<branch>`:
   - `git log --oneline HEAD..origin/main | head -10` — что есть на origin и нет локально (надо ли pull-ить).
   - `git log --oneline origin/main..HEAD | head -10` — что есть локально и нет на origin (надо ли push-ить, или это WIP).
5. Если локальная ветка отстаёт от origin и working tree чистый — **выполни `git pull --ff-only`** автоматически. Если есть конфликт fast-forward (разъехались) — стоп, сообщи пользователю, не делай rebase/merge сам.
6. После pull, если в diff между прежним HEAD и новым есть:
   - новые миграции (`backend-api/drizzle/*.sql`, `electron-app/drizzle/*.sql`) — упомянуть их, напомнить про особенность журнала (см. `pending_followups`).
   - изменения в `package.json` / `pnpm-lock.yaml` — выполнить `corepack pnpm install`.
   - изменения в `shared/` — пересобрать: `corepack pnpm --filter @matricarmz/shared build`.

## 4. Версии

1. Версия в `electron-app/package.json` и `backend-api/package.json` — должны совпадать.
2. Последний тег: `git tag --sort=-v:refname | head -3`. Сравни с версией в `package.json` — если тег отстаёт от кода, либо релиз не завершён, либо тег ещё не повешен. Если тег впереди — значит релиз шёл с другой машины и эту не успели обновить (теперь уже не должно повторяться благодаря шагу 2).

## 5. Внешний контекст (необязательно, но полезно)

Если `gh` авторизован — быстрая сводка:
1. `gh pr list --state open --limit 5` — есть ли открытые PR.
2. `gh run list --limit 3` — статус последних CI-прогонов (упомянуть если красные).

Не падай если `gh` не работает — это опциональный шаг.

## 5.5. Прод-probe (по умолчанию — НЕ дёргать)

**Мандат владельца 2026-07-15: НЕ спрашивать про прод-probe на старте** (`AskUserQuestion` на каждом старте = лишняя пауза = остывший cache). Дефолт — **прод не трогаем**, идём в §6.

Probe выполняется **только если владелец сам попросил** в команде или сообщении (напр. «/start проверь прод», «глянь прод на старте»). Тогда — read-only, окно ≤30 сек:
- `ssh -o ConnectTimeout=15 matricarmz "systemctl is-active matricarmz-backend-primary matricarmz-backend-secondary"`
- `ssh -o ConnectTimeout=15 matricarmz "curl -fsk https://127.0.0.1/health"`
- `ssh -o ConnectTimeout=15 matricarmz "cd MatricaRMZ && git log --oneline -3"`

Результат — в §6 одной строкой («прод: оба сервиса active, /health 200, HEAD=…»). При ошибке — упомянуть, **не повторять** (fail2ban агрессивен, `CLAUDE.md` §Prod server). `ssh matricarmz` теперь в allowlist (`.claude/settings.json`) — отдельный per-session opt-in не нужен; деструктив на проде остаётся под гейтом #025 (`autoMode.soft_deny`).

## 6. Отчёт пользователю

Краткий отчёт (8–12 строк, не больше) на русском:
- Машина: `<hostname>` — профиль загружен (ключевые порты) / профиля нет (§0.5).
- Ветка / версия / последний тег / соотношение с `origin`.
- Что подтянулось pull-ом (если что-то подтянулось) и были ли затронуты миграции / `shared` / зависимости.
- Активная нитка из `SESSION_HANDOFF.md` (если `Status: ACTIVE`).
- Топ актуальных пунктов из `PENDING_FOLLOWUPS` — **только если читал** его по требованию (см. §2). Если нитка свежая и handoff достаточен — этого блока может не быть.
- Открытые PR и красные workflow (если есть).
- Подозрительные «висяки»: stash, untracked, расхождения тегов.

Закончи вопросом: **«Продолжаем нитку?»** (если handoff ACTIVE и свежий) или **«Чем займёмся в этой сессии?»** (иначе) — и жди указания, ничего не начиная сам.

## 7. После завершения сессии — обновить источники правды

При завершении значимой работы (релиз, hotfix, инфраструктурное изменение):
- Что сделано (версия, ключевые изменения, прод-статус, row counts миграций) — фиксируется в теле PR и в `git log`; отдельного DEVELOPMENT_LOG больше нет.
- `docs/PENDING_FOLLOWUPS.md` — если появились новые техдолги, или удалить закрытые пункты.
- `docs/PROJECT_STATE.md` — если изменилась архитектура / правила работы / контур.

Это нужно сделать **до коммита** релиза, чтобы запись попала в тот же push.

## Что НЕ делать **на этом этапе** (только onboarding, не на всю сессию)

Это ограничения шага `/start`, чтобы он оставался дешёвым и быстрым сбором фактов. В дальнейшей работе всё перечисленное — нормально, по задаче.

- Не запускать `pnpm dev` / `pnpm build` ради «прогрева» (сборка пускается, только если pull притянул изменения в `shared/`, иначе она не нужна).
- Не редактировать файлы и не делать коммитов / push / тегов в рамках самого onboarding.
- Не дёргать прод-сервер самовольно «для прогрева». Прод-probe запускается **только** если владелец явно попросил (§5.5) — вопрос на старте больше не задаём, дефолт «не трогаем». В ходе работы (после onboarding) ssh — обычный инструмент. Помни про fail2ban: после неудачного логина не повторяй попытки подряд — разберись, в чём дело (ключ, IP, бан).
- Не делать `git rebase` / `git merge` / `git reset --hard` автоматически, если ветки разошлись на шаге 3.5 — спроси пользователя, как разрешать.
