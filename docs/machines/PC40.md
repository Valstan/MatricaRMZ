Грузить из стора, как остальные драйверы: перебрать `node_modules/.pnpm/ws@*/node_modules/ws/wrapper.mjs` и взять `m.default ?? m.WebSocket ?? m` (готовый лоадер — в `_diag.mjs`).
- **Deep-link на карточку двигателя: поле `engineId`, не `entityId`** (2026-07-15) — `navigateDeepLink({ kind:'app_link', tab:'engine', engineId })`. Не долбить повтором — проверить `(Get-Service postgresql-x64-17).Status` и попросить владельца подтвердить UAC.
- **CDP-навигация по меню: селектор `span.v2-menu-btn-label`** (2026-07-15) — пункты левого меню это `<span class="v2-menu-btn-label">Двигатели</span>` внутри кнопки. ## Машинные грабли
- **`powershell -File .\.claude\…` бьётся:** относительный путь с бэкслешами из Git Bash → PowerShell теряет точки/слеши. Передавай **абсолютный путь с форвард-слешами**.
- **PG слушает 5433, не 5432** — скилл по умолчанию ждёт 5432; `.env.dev` уже настроен на 5433.
- **`ssh matricarmz` может разово оборваться** «Connection closed by 195.161.41.30 port 49217» ещё до banner/auth — тот же класс сетевого флапа, что и push-мерцание ниже (2026-07-02: первая попытка оборвалась, повтор через минуту прошёл штатно). Одна диагностическая `ssh -vv` попытка → если ключ доходит до «Server accepts key», это флап, не fail2ban; не уходить в разбор бана/ключей сразу.
- **`install-app-deps` может молча НЕ пересобрать better-sqlite3 (стейл `.forge-meta`, 2026-07-03):** если `build/Release/.forge-meta` уже говорит `x64--145`, electron-builder пропускает пересборку, даже когда сам `.node` — Node-ABI (например, файл был заблокирован работающим backend-процессом при прошлой пересборке). Симптом: клиент падает «NODE_MODULE_VERSION 137 … requires 145» СРАЗУ после «successful» install-app-deps. Лечение: остановить backend/electron → `rm -rf node_modules/.pnpm/better-sqlite3@*/node_modules/better-sqlite3/build` → `install-app-deps` заново (проверить mtime `.node`).
- **После мажорного апгрейда Electron (41→43, 2026-07-06) `pnpm install` НЕ дотягивает бинарь Electron:** симптом — `electron-vite dev` падает `Error: Electron uninstall` в `getElectronPath`, `node_modules/.pnpm/electron@43.0.0/node_modules/electron/{path.txt,dist/electron.exe}` отсутствуют. `pnpm install` завершается за ~25с (слишком быстро для загрузки ~235МБ), а **`pnpm rebuild electron` — no-op** (пустой вывод). Лечение: запустить postinstall пакета напрямую — `cd node_modules/.pnpm/electron@43.0.0/node_modules/electron && node install.js` (тянет бинарь через `@electron/get`, ~1–2 мин). После этого `path.txt`=`electron.exe`, `dist/electron.exe` ~235МБ, `verifier-electron` поднимается штатно. (`.npmrc` уже разрешает electron в `only-built-dependencies` — гейт не в этом.)
- **verifier-electron под Electron 43 (2026-07-06): работает.** Последовательность ABI на этой машине после сегодняшнего апгрейда: `pnpm install` → `node install.js` (бинарь, см. выше) → `start-backend.ps1` (Node-ABI) → seed → `pnpm -C electron-app exec electron-builder install-app-deps` (Electron-ABI, пересобирает better-sqlite3 **и** better-sqlite3-multiple-ciphers) → `pnpm --filter electron-app dev` напрямую фоном с CDP. Standalone seed-скрипты (`dev:seed-variant-bom-verify`) грузят `.env` через `dotenv/config`, НЕ `.env.dev` → падают EACCES на PG; запускать с `set -a; . ./backend-api/.env.dev; set +a` перед скриптом (dotenv не перезатирает уже заданное).
- **CDP-драйв редактора BOM (2026-07-06):** не делать `Page.reload` на electron-vite dev-стенде (роняет клиент — та же грабля, что в SKILL.md); драйвить раздельными короткими eval'ами (один длинный IIFE >60с ловит CDP-таймаут). Тяжёлые карточки (BOM В-59, 8 вариантов) вешают `Page.captureScreenshot` при `captureBeyondViewport:true` — снимать `captureBeyondViewport:false`. Скрипты наблюдения — `.verifier-electron/cdp-bom-observe.mjs` / `cdp-bom-open.mjs` (gitignored).
- **CDP-драйв: `innerText`/`textContent`-скан по всем `div` вешает рендерер (2026-07-08).** В своём `cdp-*.mjs` НЕ делай `Array.from(document.querySelectorAll('div')).find(d=>txt(d)...)` — `innerText` форсит reflow на каждом узле, на большой карточке это O(n) reflow'ов → `Runtime.evaluate` уходит в CDP-таймаут (120–240с), при том что тривиальный `1+1` eval мгновенный (main-thread освобождается между вызовами). Лечение: узкие селекторы (`table tr`, `[data-testid=...]`, `input[name=...]`), `textContent` вместо `innerText`, скан статуса — только внутри контейнера модалки (`input[name=propagate-group].closest('div[style*="overflow-y"]')`). Логин в драйвере делай идемпотентным (проверь `auth.status()` перед `login()`) — лишние logout/login добавляют секунды к каждому eval. Полный рабочий драйвер фичи «Распространить на группу» — `.verifier-electron/cdp-propagate.mjs` (gitignored, фазы seed/nav/modalInspect/scenario/readback).
- **verifier-electron под Electron 43 отработал фичу распространения деталей на группу (2026-07-08):** последовательность из профиля (`pnpm -r rebuild better-sqlite3` для Node-ABI backend → `start-backend.ps1` → `migrate-and-seed.ps1` → `install-app-deps` для Electron-ABI → `pnpm --filter electron-app dev` фоном с CDP) прошла штатно; данные группы/деталей засевал через bridge (`nomenclatureDirectoryPartCreate` + `nomenclaturePartSpecUpdate` + `admin.entities.setAttr('engine_brand_ids',[...])`).
- **`git push` к `github.com:443` мерцает (TLS RST по SNI):** push периодически падает на TLS-handshake (`schannel`/openssl RST), при этом `gh` и `api.github.com` работают, а локальные read-операции (`git log`, `git status`) не затронуты. Лечится **повтором push'а** (обычно 2-я/3-я попытка проходит) — это сетевой флап, не проблема ключа/репо. `ssh -p 443 git@ssh.github.com` доходит, но dedicated-ключ там не авторизован → SSH-over-443 как обход не годится. Не уходить в диагностику «сломан remote/креды» — просто ретраить push.
