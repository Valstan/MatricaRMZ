## Машинные грабли
- **`powershell -File .\.claude\…` бьётся:** относительный путь с бэкслешами из Git Bash → PowerShell теряет точки/слеши. Передавай **абсолютный путь с форвард-слешами**.
- **PG слушает 5433, не 5432** — скилл по умолчанию ждёт 5432; `.env.dev` уже настроен на 5433.
- **`ssh matricarmz` может разово оборваться** «Connection closed by 195.161.41.30 port 49217» ещё до banner/auth — тот же класс сетевого флапа, что и push-мерцание ниже (2026-07-02: первая попытка оборвалась, повтор через минуту прошёл штатно). Одна диагностическая `ssh -vv` попытка → если ключ доходит до «Server accepts key», это флап, не fail2ban; не уходить в разбор бана/ключей сразу.
- **`install-app-deps` может молча НЕ пересобрать better-sqlite3 (стейл `.forge-meta`, 2026-07-03):** если `build/Release/.forge-meta` уже говорит `x64--145`, electron-builder пропускает пересборку, даже когда сам `.node` — Node-ABI (например, файл был заблокирован работающим backend-процессом при прошлой пересборке). Симптом: клиент падает «NODE_MODULE_VERSION 137 … requires 145» СРАЗУ после «successful» install-app-deps. Лечение: остановить backend/electron → `rm -rf node_modules/.pnpm/better-sqlite3@*/node_modules/better-sqlite3/build` → `install-app-deps` заново (проверить mtime `.node`).
- **`git push` к `github.com:443` мерцает (TLS RST по SNI):** push периодически падает на TLS-handshake (`schannel`/openssl RST), при этом `gh` и `api.github.com` работают, а локальные read-операции (`git log`, `git status`) не затронуты. Лечится **повтором push'а** (обычно 2-я/3-я попытка проходит) — это сетевой флап, не проблема ключа/репо. `ssh -p 443 git@ssh.github.com` доходит, но dedicated-ключ там не авторизован → SSH-over-443 как обход не годится. Не уходить в диагностику «сломан remote/креды» — просто ретраить push.
