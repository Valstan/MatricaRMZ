# Documentation Policy

Обязательные правила поддержки документации в актуальном состоянии.

## Для всех разработчиков и ИИ-агентов
- Нельзя завершать задачу с изменениями в коде, если затронуты процессы/архитектура/эксплуатация, но docs не обновлены.
- Обновление docs выполняется в той же сессии и тем же PR/коммитом, где меняется поведение системы.
- Если документ устарел или дублирует другой, нужно объединить и удалить дубликат, а не плодить новый файл.
- Для краткой межсессионной памяти использовать `PROJECT_STATE.md`; этот файл должен оставаться коротким и актуальным.

## Когда документацию обновлять обязательно
- Изменения update/release потока, установщика, источников обновлений.
- Изменения sync/ledger контрактов, диагностических эндпоинтов, autoheal логики.
- Новые отчеты/фильтры/KPI или смена бизнес-семантики полей.
- Новые ENV-переменные, системные зависимости, команды запуска/деплоя.
- Изменения в security-практиках и работе с секретами.

## Минимальный процесс для ИИ-сессий
1. Проверить релевантные docs перед правками.
2. После правок в коде обновить соответствующие docs.
3. Если изменился высокий уровень проекта, обновить `PROJECT_STATE.md`.
4. Удалить/слить устаревшие заметки (если появились дубли или больше неактуальные знания).
5. В отчете по задаче перечислить, какие docs обновлены.

## Когда обновлять `PROJECT_STATE.md`
- Изменился основной контур архитектуры или синхронизации.
- Изменился release/update/deploy/SSH к проду/setup flow.
- Изменились обязательные правила разработки или эксплуатации.
- Старое знание стало вредным: модуль удален, технология заменена, документ больше не является источником истины.

## Когда НЕ обновлять `PROJECT_STATE.md`
- Мелкий bugfix без изменения контуров системы.
- Локальный refactor, rename, formatting.
- Изменение, которое полностью достаточно описано в профильном документе и не влияет на следующую ИИ-сессию в целом.

## Карта владельцев по темам
- Архитектура/операции: `OPERATIONS.md`, `BLOCKCHAIN.md`, `TROUBLESHOOTING.md`
- Релиз/обновления: `RELEASE.md`
- Отчеты: `REPORTS.md`
- Безопасность: `SECURITY.md`
- UI стандарты: `UI_VISUAL_STANDARDS.md`

## Правило качества документации
- Коротко, проверяемо, без исторического шума.
- Команды должны быть рабочими для текущего репозитория.
- Ссылки между файлами должны быть валидными.
- Никаких секретов/токенов/паролей в текстах docs.
