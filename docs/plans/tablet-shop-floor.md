## Context — что уже есть (разведка 2026-07-16)

Оффлайн-фундамент **полностью существует** в Electron-клиенте:

- **Полная локальная реплика БД** — better-sqlite3, каждый клиент держит всю базу (`electron-app/src/main/database/schema.ts`, ledger.ts:315). Создание/правка записей оффлайн → `sync_status='pending'` → пуш при появлении сети.
- **Sync v2 (ledger):** push `POST /ledger/tx/submit`, pull `GET /ledger/state/changes?since=<seq>`, cold-start snapshot. Курсор `server_seq`, идемпотентность, schema-fingerprint guard.
- **Конфликты:** stale-seq guard (пуш со старым `last_server_seq` отбрасывается) + LWW по `updated_at` как fallback + pair-LWW для `attribute_values` (`applyPushBatch.ts:410`). Полевого мержа нет — проигравший клиент пуллит и перенакладывает.
- **Оффлайн-auth:** сессия (токены + permissions) шифруется DPAPI и читается без сети (`authService.ts:33,84`). Ограничение: **первый логин нового пользователя требует сети**.
- **RBAC оператора уже есть:** роль `engineer` (Двигателист) = `DefectActEdit`, `EnginesEdit`, `OperationsEdit`, `FilesUpload`, без админ-функций (`shared/src/domain/permissions.ts:347–432`). Скрытие «лишнего» — вопрос UI-режима, не новой системы прав.
- **Дефектовка:** `EngineInventoryRow` (`present`, `actual_qty`, `repairable/scrap/replace_qty`) в `operations.metaJson` — уже синкается оффлайн как обычная строка (`shared/src/domain/repairChecklist.ts:298–324`, `RepairChecklistPanel.tsx`).

web-admin — тонкий онлайн-SPA **без** оффлайна (нет SW/IndexedDB). Renderer Electron жёстко привязан к IPC (`window.matrica`), браузерного пути нет.

## Решение: Windows-планшет + существующий Electron-клиент

| Вариант | Оценка |
|---|---|
| **A. Windows-планшет + Electron-клиент (планшетный UI-режим)** ✅ | Оффлайн/sync/auth/RBAC переиспользуются как есть. Работа = только touch-UI. Недели, низкий риск. |
| B. Android-планшет + PWA/натив | Переписать весь оффлайн-движок (sqlite-wasm/IndexedDB, sync, оффлайн-auth, шифрование) + новый UI. Месяцы, высокий риск, второй движок навсегда в поддержке. |
| C. Модуль в web-admin | Тот же объём, что B (web-admin без оффлайна), плюс смешивание админки с операторским контуром. |

Вариант A — рекомендация. B имеет смысл только если жёстко нужен Android/железо дешевле 25 т.р. — сейчас не нужен.

## Железо (рекомендуемые характеристики)

- **ОС:** Windows 11 (обязательное условие варианта A).
- **Экран:** 10.1–12", IPS 1920×1200, яркость ≥400 нит (цех), ёмкостный touch.
- **CPU:** Intel N100/N150/N200 или выше (Electron + SQLite-реплика тянут спокойно).
- **RAM:** **16 ГБ** предпочтительно (минимум 8): Electron-клиент + полная реплика.
- **SSD:** 128–256 ГБ (реплика БД + вложения/фото — с запасом на годы).
- **Батарея:** ≥8 ч реальной работы; USB-C зарядка (док-станция в конторке мастера).
- **Wi-Fi:** 802.11ac/ax. **LTE не нужен** — синхронизация в зоне Wi-Fi (конторка/офис), в цеху всё равно оффлайн.
- **Обвес обязателен:** противоударный чехол с ремнём на руку, плёнка/стекло на экран, стилус опционально (галочки удобнее пальцем при крупном UI).
- Ориентиры моделей (проверить актуальную доступность/цены в РФ): Microsoft Surface Go 4 (N200/8ГБ — верхняя полка), бюджетно — Chuwi Hi10 Max / UBook, Teclast X-серия на N100 (16/256); защищённые (IP65) Windows-планшеты — дороже в разы, на пилот не нужны: чехла достаточно.
- **Пилот: 1 планшет**, докупать после обкатки Ф1.

## Конфликты: стратегия

Мировые практики: LWW (есть у нас), field-merge/CRDT (дорого, для нашей доменки избыточно), **check-out/reservation — pessimistic offline lock** (стандарт для CMMS/инспекционных систем, где работа естественно делится по объектам). Работа в цеху ровно такая: один оператор ↔ один двигатель.

**Принимаем: advisory-резервирование двигателя (Ф2)** поверх существующего LWW (LWW остаётся страховкой):

- При наличии сети оператор жмёт «Взять в работу» на двигателе → запись-резерв (entity lock: `engineEntityId`, `userId`, `expiresAt` TTL ~24–72 ч) уходит через обычный sync.
- У остальных клиентов карточка двигателя (дефектовка/атрибуты) становится read-only с плашкой «Редактирует Иванов (логин), до 18.07» — по правилу clientLabel (логин + ФИО).
- Снятие: сам оператор («Вернуть»), автоистечение TTL, принудительно — admin.
- Enforcement **мягкий** (advisory): гейт в UI + гейт на сервере при пуше чужих правок в зарезервированный engine → строки в `skipped[]` с reason `reserved`, клиенту — понятное уведомление. Полная строгость невозможна принципиально (два оффлайн-клиента сервер не разведёт до синка) — потому TTL + LWW-страховка.

## План работ

### Ф0 — пилот «как есть» (0.5–1 день + обкатка владельцем)
1. Купить/взять 1 Windows-планшет, поставить текущий клиент (обычный installer, автообновление работает).
2. Владелец/мастер проходит цех: наличие галочками, лист дефектовки. Оффлайн уже работает.
3. Собрать список UX-болей (мелкие цели для пальца, клавиатура, ориентация экрана).
- **Выход:** подтверждение варианта A на реальном железе, вход для Ф1.

### Ф1 — «Планшетный режим» UI (основная разработка, ~3–5 сессий)
1. Настройка/автодетект «планшетный режим» (touch + маленький экран) в SettingsPage; persist локально.
2. Режим оператора: стартовый экран = список двигателей в работе (крупные карточки, поиск по внутреннему номеру `41/26`), меню сжато до разрешённого роли (`operatorRolePermissions`).
3. Touch-оптимизация ключевых экранов: `RepairChecklistPanel` (крупные чекбоксы `present`, степперы qty вместо мелких input), карточка двигателя; цели нажатия ≥44px.
4. Индикатор синхронизации всегда на виду: онлайн/оффлайн, счётчик pending-строк, время последнего успешного синка.
5. Экранная цифровая клавиатура для qty-полей (inputmode/numpad).
- **Гейты:** typecheck/lint/tests + CDP-смоук planшетного режима (окно 1200×800 touch-эмуляция).

### Ф2 — Резервирование двигателя (~2–3 сессии)
1. Домен: `shared/src/domain/engineReservation.ts` (типы, TTL, правила); хранение — EAV-атрибуты на engine-entity или отдельная синкаемая таблица (решить при реализации; EAV дешевле — без DDL).
2. UI: кнопка «Взять в работу»/«Вернуть», плашка «Редактирует …» (формат `formatClientLabel`), read-only гейт карточки.
3. Сервер: мягкий гейт в `applyPushBatch` (чужие правки reserved-двигателя → skipped reason `reserved`), admin-снятие в web-admin/AdminPage.
4. Клиент: обработка skipped `reserved` → уведомление оператору «двигатель занят, изменения не применены».

### Ф3 — оффлайн-hardening (~1–2 сессии)
1. UI разбора «мои неприменённые изменения» (pending error / skipped conflict|reserved) — оператору видно ЧТО не ушло и почему.
2. Проверка длительного оффлайна (дни): размер очереди pending, batch-caps, первый-логин-требует-сети — памятка мастеру.

### Ф4 — развитие (бэклог, после обкатки)
- Сканирование QR/штрихкода внутреннего номера камерой планшета → мгновенный переход на карточку (стыкуется с идеей brain «traceability/QR»).
- Фото дефектов с камеры прямо в лист дефектовки (FilesUpload уже в роли engineer).
- Второй+ планшеты, разбивка по участкам.
