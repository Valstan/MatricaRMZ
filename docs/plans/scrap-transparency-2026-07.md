# Утиль: причины, прозрачность, сводный отчёт

**Запрос владельца 2026-07-17:** при пометке утиля (деталь в дефектовке или двигатель целиком) предлагать заполнить причину; причина видна в программе, печатях, отчётах. Отдельный отчёт по утилю с фильтрами (двигатели, номера, контракты, марки, даты, ветка замещения). Контроль движения утиля по заводу «с любого угла зрения».

## Что уже есть (ревизия 2026-07-17)

- **Строка дефектовки** (`EngineInventoryRow`, `shared/src/domain/repairChecklist.ts`): `scrap_qty` + `replace_qty` + `repairable_qty` = quantity; **`replenishment_branch` = customer | repair | purchase** — ровно три ветки замены владельца (давальческая / свой ремонт / закупка). Данные живут в `operations.metaJson` (стадия engine_inventory).
- **Двигатель целиком:** статусы `status_scrap_confirmed` (ранний утиль по дефектовке) и `status_rework_sent` (утиль отправлен заказчику) + даты; `isScrapEngine()`; отчёт по двигателям имеет колонку/фильтр «Утиль».
- **Склад/прогноз:** утиль исключён из доступного (scrap-локация, авто-отзыв Assembly-нарядов по утилю, `listScrapPartNames`); заявки в снабжение строятся из дефектовки (`buildSupplyRequestItemsFromInventory`).
- **Мировая практика** (nonconformance/MRB): disposition (scrap/rework/use-as-is/return) ≈ наши ветки; **reason не как жёсткий справочник, а свободный текст** — по анти-бюрократия-мандату завода (письмо brain 2026-06-17: «учёт утиля не утонув в бюрократии»). Типовые причины — подсказкой (datalist), не обязательным справочником.

## Дыры (что строим)

### Ф1. Причина утиля — данные + ввод
- `EngineInventoryRow.scrap_reason?: string` — новое опц. поле; нормализация: сбрасывается при `scrap_qty=0`; legacy `undefined ≈ ''` (не считать изменением — как `replenishment_branch`).
- `RepairChecklistPanel`: колонка «Причина утиля» в дефектовке (текст, datalist типовых причин: трещина / износ сверх допуска / коррозия / деформация / обрыв резьбы / не подлежит восстановлению); мягкая подсветка, когда `scrap_qty>0` и причина пуста.
- Двигатель: EAV-атрибут `scrap_reason` на engine; поле ввода в карточке рядом с утильными статусами, появляется при установке `status_scrap_confirmed`/`status_rework_sent`; регистрируется в attr defs карточки.

### Ф2. Видимость причины
- Печать актов дефектовки (`engineInventoryPrintHtml`): колонка причины в секции утиля.
- Печать карточки двигателя: строка «Причина утиля» рядом со статусами.
- Заявка снабжения из дефектовки: причина добавляется в note позиции.
- Отчёт по двигателям: колонка «Причина утиля» (attr `scrap_reason`).

### Ф3. Отчёт «Утиль» (клиентский пресет `scrap_register`)
- Строки-детали: все строки дефектовок с `scrap_qty>0` по всем двигателям (из локальной SQLite: operations × engines × attrs) — № двигателя, внутр. №, марка, контракт, контрагент, деталь, № чертежа, клеймо, кол-во утиля, причина, ветка замещения, дата дефектовки.
- Строки-двигатели: engines с утильными статусами — «Двигатель целиком», причина из `scrap_reason`, дата статуса.
- Фильтры: период (дата дефектовки/статуса), марки, контракты, контрагенты, № двигателя (текст), ветка замещения (select), вид (детали/двигатели/все), колонки.
- Итоги: всего позиций утиля, штук, по веткам замещения.

### Не трогаем (уже закрыто)
Прогноз/остатки утиль не считают доступным; движение «в утиль» на складе — существующая scrap-локация. Отдельных движений не добавляем.
