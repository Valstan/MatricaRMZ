# Эпик: не сохранять пустые карточки + чистка пустых + черновики/восстановление

**Статус:** Phase 1 / 2a (двигатели) / 2b (сотрудники) / 3a отгружены (релиз v2026.629.1711). В `main` (на прод ещё не выпущены): 3b (наряд-пилот), 3c (раздел «Черновики» + «Сохранить как черновик» для нарядов), Phase 2 deferred-create нарядов **и заявок**. Дальше: deferred-create для договоров/masterdata (общий `admin:entities`, широкий радиус) + 3d (автосейв/черновики на прочие редакторы).

## Context (зачем)

Три связанные боли владельца:

1. **Авто-создание пустых карточек.** `create()` любой карточки сразу вставляет строку в локальную SQLite (`syncStatus:'pending'`) → пустые незаполненные карточки синкаются всем и копятся мусором. Нужно: персист только при наличии содержимого / по явному сохранению.
2. **Нет чистки накопившегося мусора.** В области «Поиск дублей / Обслуживание» нужна функция поиска и удаления пустых карточек / пустых документов.
3. **Потеря несохранённых данных** при закрытии программы / отключении питания / авто-обновлении. Текущий guard на грязном закрытии через 10 с **принудительно сохраняет** (`App.tsx:702`), хотя оператор мог намеренно не сохранять. Нужна система мировой практики: непрерывный автосейв-снимок, на старте — предложение восстановить/отклонить/решить позже, и явные черновики (документ создан, но ещё не в документообороте).

**Решения владельца (2026-06-29):** (1) порядок — **сначала чистка пустых**, затем отказ от авто-создания + черновики/восстановление; (2) черновики и автосейв — **синхронные между ПК** (следуют за оператором).

## Архитектура (как 3 части связаны)

- **A (отказ от авто-создания):** `create()` не вставляет строку — возвращает клиентский UUID (генерируются до вставки) + пустой payload в памяти. Реальная вставка в `operations`/`entities` — только при явном «Сохранить» (commit/upsert).
- **C (черновики/восстановление):** синкаемый слой черновиков (`card_drafts` в sync-реестре — владелец выбрал sync). Пока карточка открыта и dirty — debounced-автосейв снимка в `card_drafts`. Явное «Сохранить» = commit в операционные таблицы + удаление черновика. Краш/форс-закрытие → черновик остаётся → на старте предложение восстановить. A+C делят движок: «новая карточка = черновик до commit».
- **B (чистка):** инструмент, чистит то, что A предотвращает на будущее. Шипается первым, самостоятелен.

Граница A/C: пустая новая карточка (открыли/закрыли без ввода) **не материализуется даже как черновик** — автосейв пишет только при осмысленном содержимом.

## Phase 1 — Чистка пустых (в работе)

Операторский поток в духе дедупа: **анализ → ревью списка → soft-delete через синк** (с серверной ре-валидацией «пусто» перед удалением). Без тихого авто-удаления (data-integrity).

- `shared`: предикаты `isWorkOrderPayloadEmpty` / `isSupplyRequestPayloadEmpty` (defensive).
- `backend-api/src/services/emptyCardsService.ts`: `analyzeEmptyCards()` (скан по типам) + `deleteEmptyCards({ids,actor})` (ре-валидация → soft-delete). Сущности (двигатель/договор/сотрудник) через `softDeleteEntity` (гард входящих ссылок). Документы (наряд/заявка) на `operations` — soft-delete + `recordSyncChanges` op:'delete' (паттерн `engineDedupeService.mergeEngineGroup`).
- Правила «пусто»: двигатель — нет `engine_number` И `engine_brand`; договор — нет `number`/`internal_number`/`date`; сотрудник — нет `login`/`full_name`; наряд — пустые `freeWorks`/`works`/`crew`/`workGroups` и нет `linkedDocumentId`; заявка — пустые `items`/`attachments` и пустой `title`. Деталь/номенклатура (PG) — отдельная ветка, отложена.
- `backend-api/src/routes/maintenance.ts`: `GET /maintenance/empty-cards` (MasterDataView) + `POST /maintenance/empty-cards/delete` (MasterDataEdit, без авто-ретрая); монтаж в `app.ts`.
- Electron: обёртки в `erpService.ts` (`maintenanceEmptyCardsAnalyze/Delete`, `httpAuthed`), IPC `ipc/register/maintenance.ts` (+ `registerIpc.ts`), bridge `window.matrica.maintenance.*` в `preload/index.ts`.
- UI: `EmptyCardsCleanupPage.tsx` (группы по типам, чекбоксы, «Удалить выбранные» + подтверждение, отчёт удалено/пропущено), таб `empty_cards` в группе `control` (Tabs.tsx + App.tsx), гейт `caps.canEditMasterData`.

## Phase 2 — Отказ от авто-создания (в работе, по типам)

Паттерн (EAV-сущности): `create()` не вставляет строку (возвращает UUID), `get(id)` синтезирует пустую карточку для отсутствующего id, первая запись материализует строку. Пустую открыли/закрыли — призрака нет.

- ✅ **Двигатели** (PR в работе): `createEngine` без insert; `getEngineDetails` синтезирует; `setEngineAttribute` материализует сущность при первой записи атрибута. CDP-verify PASS (create+abandon → не в списке; setAttr → материализуется; UI новая карточка рисует пустую форму). Именно двигатели дали 15 призраков в снапшоте.
- ✅ **Сотрудники** (PR в работе): изолировано через employees-IPC — `employees:create` возвращает UUID без вставки; `getEntityDetails`/`setEntityAttribute` получили **опциональный `fallbackTypeId`** (синтез/материализация только когда передан → общий admin-путь НЕ затронут); employees:get/setAttr передают тип `employee`. CDP-verify PASS. (Контракт-cleanup `employees.delete` несинканного — отдельная преждинг-боль remote-delete, не от этой правки.)
- ⚠️ **Договоры + вся masterdata** — идут через **общий** `admin:entities:create/get/setAttr`. Делается **по типам карточек, слайсами** (каждый слайс = отдельный verified-PR), а НЕ «во всех потребителях разом». Разведка уточнила радиус (важно):
  - **Inline-lookup / copy / bulk-seed** потребители (создать единицу/склад/марку/отдел/контрагента-копию + сразу `setAttr` имени; `copyToNew`; preset-seed) **никогда не дают пустых призраков** → их НЕ трогаем (insert+setAttr как есть, нулевой риск).
  - **Только «новая полная карточка»**-потоки плодят призраков → их и конвертируем: клиентский `crypto.randomUUID()` вместо `create()`, `get`/`setAttr` с `fallbackTypeId`.
  - `MasterdataDirectoryPage` — **мёртвый код** (нигде не импортируется), не трогать. Товары/услуги идут через `NomenclatureDirectoryPage` → `createNomenclatureLineFromPreset` (**warehouse-номенклатура**, не admin-entity) → отдельная ветка позже, не через этот механизм.
  - **Ядро (backward-compatible):** `admin:entities:get(id, fallbackTypeId?)` + `setAttr(id, code, value, fallbackTypeId?)` (IPC `admin.ts` + preload + shared-тип `MatricaApi`). Существующие вызовы без аргумента ведут себя как раньше.
  - **✅ Слайс 1 (ветка `feat/masterdata-deferred-create`):** ядро + **контрагенты** (`CounterpartiesPage`→`CounterpartyDetailsPage`) + **марки** (`EngineBrandsPage`→`EngineBrandDetailsPage`). CDP-verify PASS на `PC40`: create→пустая карточка→бросить → нет призрака (count 21→21 / 39→39); create→имя→Сохранить → +1 в списке (22 / 40). summary-persist марки на несохранённой марке — тихий no-op (не материализует). Кнопка «Удалить» на новой карточке — безвредный no-op (как сотрудники/двигатели).
  - **✅ Слайс 2 (ветка `feat/contract-deferred-create`):** **договоры** (`ContractsPage`→`ContractDetailsPage`). Create → клиентский UUID; `loadContract` get с fallbackTypeId; все ~16 собственных `setAttr` (saveSections/ГОЗ-accounting/attachments/custom-field) через хелпер `setContractAttr` (резолв contract-typeId + fallback). copyToNew/inline-lookup (`created.id`) и `engines.setAttr(contract_id)` — не тронуты.   - **Осталось (следующие слайсы):** `AdminPage:1077` raw-сетка; товары/услуги (nomenclature-путь — отдельно); прочие EAV-карточки если всплывут.

- Ранний id (файлы, аудит) — через клиентский UUID. Граничный кейс (ввёл значение → стёр → закрыл) оставляет пустую строку — добивается Phase 1.
