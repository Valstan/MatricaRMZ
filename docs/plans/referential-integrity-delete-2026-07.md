# Референс-целостность при удалении объектов

**Статус:** черновик на утверждение владельца (2026-07-24)
**Инициатор:** владелец — «при удалении объекта искать связанные, показывать и предлагать: удалить связи / оставить / заменить / отложить в журнал напоминаний».
**Ветка-фундамент:** PR #322 (видимость висячих ссылок) — предпосылка варианта «оставить».

## Проблема (корень)

Удаление сущности (`entityService.softDeleteEntity`) ставит `deletedAt` без проверки входящих ссылок. Единственная существующая проверка `findIncomingLinkRows` видит **только** link-типизированные EAV-`attribute_values`. Она слепа к ссылкам в:
- `contract_sections` JSON (contracts) — `engineBrandId`, `customerId`, `partId`;
- `operations.meta_json` (наряды) — `assemblyEngineId`, `employeeId`, `serviceId`, `partId`, `engineId`, `engineBrandId`, `defectOrigin`;
- `operations.meta_json` (заявки) — `productId`, `departmentId`, `workshopId`, `sectionId`;
- junction-таблицы BOM — `erp_engine_assembly_bom_brand_links.engine_brand_id`, `..._lines.component_nomenclature_id`.

Итог на проде: 10 висячих строк марок в 5 контрактах — удалили марку, на которую ссылался контракт (см. диагностику в PR #322).

## Цель

На удаление любой masterdata-сущности:
1. Найти **все** входящие ссылки (по всем хранилищам, не только EAV-линки).
2. Показать их оператору списком (что именно ссылается: контракт №, наряд №, BOM …).
3. Предложить действие: **(a)** удалить связи, **(b)** оставить (висячие, но видимые), **(c)** заменить связи на другой объект того же типа, **(d)** удалить связи + отложить замену в «Заметки» напоминанием.

## Архитектура

### Ф1 — Универсальный сборщик входящих ссылок (фундамент)
`shared/src/domain/entityReference.ts` уже имеет `EntityReferenceTarget` и `collectWorkOrderEntityReferences`. Добавить **обратный индекс**: для заданного `entityId` найти все записи, что на него ссылаются.

- **Серверная реализация — авторитетная** (`backend-api`): сканит все хранилища из списка выше и возвращает `IncomingReference[] = { sourceKind: 'contract'|'work_order'|'supply_request'|'bom'|'eav_link'|'engine', sourceId, sourceLabel, path, targetType }`. Переиспользовать существующие коллекторы (`collectWorkOrderEntityReferences`, `entityReferenceGuard` reader'ы) — не дублировать разбор.
- **Клиентская реализация — advisory** (для offline-диалога): тот же сбор по локальной SQLite-реплике. Реплика может быть неполной → диалог помечает «проверено по локальной копии», а серверный гейт (Ф4) — страховка.
- Новый IPC `admin:entities:incomingReferences(entityId)` (расширяет нынешний `deleteInfo`, который покрывает лишь EAV-линки).

### Ф2 — Диалог намерения при удалении
Сделать текущий `AdminPage` deleteInfo-диалог **стандартным** путём удаления во всех masterdata-точках (list-меню `listContextActions`, detail `SimpleMasterdataDetailsPage`). Диалог показывает найденные ссылки и 4 кнопки. Пустой список ссылок → обычное быстрое удаление (без лишнего клика).

### Ф3 — Четыре действия
- **(a) Удалить связи:** обнулить/убрать ссылку в каждом источнике. Правки в JSON-блобах/junction — через **sync-безопасный** путь владельца записи (не raw SQL): контракт/наряд/заявка пересохраняются штатным upsert'ом с вычищенной ссылкой; BOM-линк — delete junction-строки. Затем soft-delete цели.
- **(b) Оставить:** soft-delete как есть; ссылки становятся висячими, но **видимыми** (PR #322).
- **(c) Заменить:** пикер сущности того же типа (`EntityReferenceField` target = тип цели) → перецепить каждую ссылку на новый id тем же sync-безопасным путём → soft-delete цели.
- **(d) Отложить:** удалить связи (или оставить) + создать **Заметку** (`notes.upsert`) `importance:'important'`, `dueAt`, тело = список нерешённых ссылок («Марка X удалена — заменить в контракте 04/ГОЗ-25, наряде …»), опц. `notes.share` ответственному оператору.

### Ф4 — Серверный гейт-страховка
Расширить `enforceEntityReferenceIntegrity` (или добавить delete-time проверку в `adminMasterdataService.softDeleteEntity`), чтобы удаление, проскочившее клиентский диалог (offline/старый клиент), не создавало висячих ссылок в JSON/junction-хранилищах — а не только в EAV-линках, как сейчас.

## Решения (принял сам — технические)

- **Хранилище отложенных задач — «Заметки»** (существующий журнал: importance/dueAt/бейдж/самонапоминание/шаринг). Структуру ссылки кодируем в теле заметки; отдельную типизированную таблицу заводить не будем, пока не потребуется машинная обработка.
- **Серверный сбор — авторитетный, клиентский — advisory + серверный гейт-страховка.** Не полагаемся только на возможно-неполную реплику.
- **Переиспользуем** `collectWorkOrderEntityReferences` / `entityReferenceGuard`-reader'ы, не пишем второй парсер ссылок.

## На утверждение владельца (продуктовые)

1. **Охват на старте:** все masterdata-типы сразу или начать с `engine_brand` (конкретная боль) и обобщить? Рекомендую **сборщик сразу общий**, а диалог включить сначала для `engine_brand` + `customer`, затем расширять — риск ниже.
2. **Действие по умолчанию** в диалоге — рекомендую **(c) Заменить** (сохраняет целостность), с явной **(a)** для «объект-ошибка, ссылки-мусор».
3. **Замена для не-марок** (9 висячих из диагностики — «Щиток…», «Полки…» ошибочно заведены как марки): их аналогов-марок нет → тут уместен вариант (a)/(d).

## Фазировка

- **Ф1** сборщик (server + client + IPC) — 1 PR, с тестами на каждый тип хранилища.
- **Ф2+Ф3** диалог + 4 действия — 1 PR (сначала engine_brand/customer).
- **Ф4** серверный гейт-страховка — 1 PR.
- Расширение охвата на остальные типы — по мере приёмки.
