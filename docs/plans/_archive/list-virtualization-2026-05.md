# План: замена постраничной пагинации на виртуализированную подгрузку

**Статус:** PR #119 — rollout почти завершён (2026-05-29). Sticky-шапка ПОЧИНЕНА. Виртуализированы: Двигатели, Сотрудники, Контрагенты, + Variant-1 (EngineBrands, MasterdataDirectory, Audit, EngineAssemblyBom, WorkOrders, SupplyRequests, Changes), + сломанные складские (StockBalances, StockDocuments — load-all), + Variant-3 (NomenclatureDirectory, PartTemplates, StockInventory). Остаётся: NomenclaturePage (сгруппированная — отложена), live-verify части страниц, ready→merge→release. Создан 2026-05-28.

## Цель

Убрать постраничный пейджер (`«‹ Назад N/M (X всего) Вперёд ›»`) во всех списках. Длинные списки рендерить плавно, без тормозов, с подгрузкой по мере прокрутки — но так, чтобы **сортировка и фильтрация работали по ВСЕМУ датасету**, а не только по видимым строкам. Оператор всегда видит актуальную информацию по всему списку.

## Ключевой вывод разведки (2026-05-28)

- **Данные — не узкое место.** Сортировка 1624 (и даже 50 000) строк в JS — <100 мс. Тормозит **рендер тысяч `<tr>` в DOM**.
- **Большинство списков уже грузят всё целиком** и сортируют в памяти (Variant 1: двигатели, детали, сотрудники, марки, контракты, наряды, заявки, audit, changes, BOM…). Пагинация там — лишь display-нарезка. Сортировка по полному набору **уже корректна**. → убрать пейджер + виртуализировать рендер. Backend не трогаем.
- **3 страницы реально сломаны** (серверная постраничная загрузка, клиентская сортировка сортирует только видимую страницу): **Номенклатура** (`NomenclaturePage`), **Документы склада** (`StockDocumentsPage`), **Остатки склада** (`StockBalancesPage`). Лечатся «load-all + клиентская сортировка + виртуализация».
- **Ни один backend-эндпоинт не умеет серверную сортировку и не отдаёт `total`** (только `rows` + `hasMore` через `limit+1`). Серверная сортировка — запасной Phase 2, только если замер покажет неподъёмный по объёму список.
- Виртуализации/infinite-scroll в коде **нет** (проверено: нет react-window/react-virtual/IntersectionObserver). Есть хелперы «загрузить все страницы»: `partsPagination.ts` (`listAllParts`), `warehousePagedFetch.ts` (`fetchWarehouseStockAllPages`).
- DOM всех списков: `<table class="list-table">` + sticky `<thead>` + `<tbody>` из `<tr>`. Переменная высота строк (превью-картинки, многострочные диффы в Changes). `TwoColumnList` (multi-column) усложняет windowing — интегрируем сначала в single-column путь.

## Подход (best practice)

**Виртуализированный рендер над полным отсортированным/отфильтрованным массивом в памяти** (`@tanstack/react-virtual`, dynamic measured row height, overscan). Это «правильная» форма «подгрузки по 100»: в DOM только видимое окно (~40-50 строк +буфер), при прокрутке окно едет; сортировка/фильтр — по всему набору. DOM постоянный независимо от размера → плавно даже на десятках тысяч строк.

## Открытые вопросы / follow-ups

- **✅ Sticky-шапка ПОЧИНЕНА (2026-05-29, коммит 091dc146).** Root cause: хук `useAdaptiveListTables` через MutationObserver вешает класс `.list-table-wrap`/`--single` на родителя каждой `table.list-table` (в `VirtualTable` это div-обёртка). `.list-table-wrap--single { overflow-x: hidden }` из-за CSS-коэрцинга делал `overflow-y: auto` → div становился скролл-контейнером и перехватывал sticky-контекст `<th>` у настоящего scrollElement. Фикс: `overflow-x: clip` (клипает без скролл-контейнера) + у обёртки `VirtualTable` нет inline-overflow. Проверено вживую на 1624 двигателях. Это и есть «причина выше по дереву» из прошлой записи.
- **🟡 NomenclaturePage — отложена.** Сгруппированный аккордеон (группы + серверно-пагинированный список позиций внутри развёрнутой группы), а не плоская таблица. Виртуализация вложенного-в-аккордеон списка не ложится на flat `VirtualTable`. Минимальный фикс реального бага: при разворачивании группы грузить ВСЕ её позиции (load-all) + клиентская сортировка, убрать внутренний пейджер (группы bounded ~363 → можно без виртуализации). Отдельным follow-up.
- **🟡 Live-verify не на всех страницах.** Вживую проверены: Двигатели, Марки, Заявки, Наряды, StockBalances (full-set sort), StockDocuments, Changes(пусто). Только typecheck (не открыты вживую): MasterdataDirectory, Audit, EngineAssemblyBom, NomenclatureDirectory, PartTemplates, **StockInventory**. У StockInventory особый риск: колонка «Факт» — редактируемый `<Input>`; значение хранится в state (не теряется), но при прокрутке строки за окно виртуализации инпут размонтируется и теряет фокус. Проверить сценарий ручного ввода на длинном складе.
- **🟢 ContractsPage / ToolPropertiesPage — пейджера нет** (не используют `WarehouseListPager`, рендерят весь набор). Конвертировать для удаления пейджера нечего; виртуализация — опциональный perf-follow-up если списки длинные.
- `useLiveDataRefresh` (15s silent refetch) на WorkOrders/SupplyRequests — scroll сохраняется через `usePersistedScrollTop`, pageIndex больше нет (нечего сбрасывать). OK.
