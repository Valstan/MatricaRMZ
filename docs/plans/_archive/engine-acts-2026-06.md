## 2. Фазовый план (инкременты, каждый — свой /verify)

Порядок — от дешёвого/высокоэффективного UI к сложному data-flow. brain предложил похожий порядок; уточнён картой.

- **Фаза 1 — чекбоксы выбора-в-печать + массовые операции** (GAP-1; чистый UI, без схемы). ✅ **ПОСТРОЕНА (PR #302, /verify PASS).** `__selected` meta-поле строки (`ROW_SELECTED_KEY`, как `__photos`; переживает нормализацию + brand-resync); колонка-чекбокс + master-чекбокс в `TableEditor`; тулбар «Выбрать все/Снять выделение · Все на месте/отсутствуют · Все в ремонт/утиль/заменить» (изменяющие данные операции под `confirm`); печать акта — только отмеченные, fallback на все при пустом выборе; счётчик «в печать: N». Тесты в `repairChecklistRows.test.ts`. CDP-драйвер `cdp-acts-phase1.mjs`.
  - **HOW-решения (рамка делегирована):** (a) семантика — чекбоксы = выбор *в печать*; кнопки «Все …» применяются ко *всем показанным* строкам (буквальный «все», уважает фильтр по варианту). (b) **версионирование актов отложено на Фазу 2** — Ф1 печатает эфемерное HTML-окно, персистентного акта ещё нет, версионировать нечего; носитель версий закладывается в Ф2, когда акт впервые станет сохранённой записью (prompt недостачи на save). Держит рамку Ф1 узкой («не раздувать над уже построенным»).
- **Фаза 2 — версионируемые акты + недостача** (GAP-3). ✅ **ПОСТРОЕНА (PR #303, /verify PASS).** Переинтерпретация (owner-aligned): владелец сказал «каждая **печать** = новая версия» → версионирование цепляется к **печати**, а не к автосейву (prompt на каждый автосейв = плохой UX). Что сделано:
  - **Снимок акта при печати** = новая строка `operations` (`operationType` = `completeness_act`/`defect_act`, payload `EngineActSnapshotPayload` плоско в `meta_json` — **без миграции**; новые типы синкаются дженериком). Дедуп идентичных подряд (`engineActSnapshotSignature`). Версия монотонна per (двигатель, actType).
  - **Детект недостачи** (`computeInventoryShortage` в shared, `actual_qty<quantity`) + баннер «Недостача: N позиц. (M ед.) — требует восполнения. Требование в снабжение не создано» — **без** заявки (Ф3 под `director_approved`).
  - **История версий**: `engineActVersions` (electron-main, читает локальный SQLite — снимки синкаются), сворачиваемый список в панели с повторной печатью замороженного снимка.
  - Носители: `shared/src/domain/engineActSnapshot.ts` (+тесты), `electron-app/.../checklistService.ts` (`saveEngineActSnapshot`/`listEngineActVersions`), IPC `checklists:engine:actSnapshot|actVersions`, bridge. CDP-драйвер `cdp-acts-phase2.mjs`.
  - **Объём**: только клиентский путь (web-admin не имеет engine_inventory) → backend-api роуты не нужны, снимки персистятся локально + синкаются как операции.
- **Фаза 3 — 3-веточный workflow восполнения + гейт** (GAP-2). ✅ **ПОСТРОЕНА (PR #304, /verify PASS).** Поле `replenishment_branch: 'customer'|'repair'|'purchase'|null` — **логическая колонка `EngineInventoryRow`** (нормализуется/синкается, без миграции; переживает brand-resync). Per-деталь селектор «—/Заказчик/Свой ремонт/Закупка», активен при `replace_qty>0`. `buildSupplyRequestItemsFromInventory` → только строки `rowGoesToPurchase` (replace_qty>0 И ветка `purchase`/null; явные `customer`/`repair` уводят из закупки; null=закупка для back-compat MVP-1). Сводка `summarizeReplenishment` + нота «требование уходит только после `director_approved`» (гейт = переиспользован, #025). Ветки «заказчик»/«свой ремонт» — закрытие в «есть» по приходу/готовности (закрытие — Ф4/Ф5). Тесты в `engineInventoryRow.test.ts`. CDP-драйвер `cdp-acts-phase3.mjs` (счётчик заявки 4→3→2 по мере маршрутизации).
- **Фаза 4 — дефектовка: финал статусов + претензии заказчику.** ✅ **ПОСТРОЕНА (/verify PASS).** Что сделано:
  - **Триггер «дефект»**: ветка восполнения активна при `scrap_qty>0 || replace_qty>0` (`rowDefectQty`/`rowHasDefect` в shared), не только при «заменить» — утиль тоже выводит деталь из двигателя. `summarizeReplenishment` считает по дефекту; `rowGoesToPurchase` не тронут (в закупку идут только единицы «заменить» — утиль закупкой не восполняется).
  - **Претензия заказчику** = третий версионируемый акт (`EngineActType='claim'`, operationType `claim_act`, синкается дженериком как Ф2-типы): `computeCustomerClaim` (строки `replenishment_branch='customer'` с дефектом, `claim_qty=scrap+replace`) + печатная форма `buildInventoryClaimHtml` с двумя секциями — «дефектные детали к восполнению заказчиком» и «недостача комплектности при приёмке» (reuse `computeInventoryShortage`; снимок претензии хранит shortage). Кнопка «Печать претензии (N)» disabled при пустой претензии и пустой недостаче.
  - **Финал статусов в печати**: акт дефектовки получил колонку «Восполнение» (Заказчик/Свой ремонт/Закупка/—).
  - **Сигнатура снимка** дополнена `replenishment_branch` — смена ветки = новый контент акта (корректный дедуп для претензии).
  - HOW-решение: недостача в позиции претензии НЕ входит (отдельная секция формы) — позиции претензии строго по дефекту на ветке «заказчик» (рамка handoff); некомплект приёмки идёт второй секцией как естественное основание претензии без расширения модели.
  - CDP-драйвер `cdp-acts-phase4.mjs` (дельта-ассерты по счётчикам версий — операции персистентны между прогонами).
- **Фаза 5 — ремфонд → наряд → прогноз** (GAP-4/5/6). ✅ **ПОСТРОЕНА (/verify PASS, CDP-драйвер `cdp-acts-phase5.mjs`, 18 шагов e2e: кнопка → наряд → in_repair → backend close → ready_for_assembly синком → бейдж «готова к сборке»).** Выучено при verify: (а) **серверные записи в sync-таблицы обязаны идти через `recordSyncChanges`** (ledger-подпись + `last_server_seq`), прямой `db.insert` клиентам не раздаётся; (б) пре-существующая дыра — статус `closed` самого наряда тоже пишется прямым update'ом и инкрементально не редистрибутится (заведён follow-up); бейдж Ф5 от неё не зависит (ready-событие того же наряда переводит отображение в «готова»). **HOW-решения (зафиксированы probe-сессией 2026-06-10, рамка делегирована):**
  - **Контекст probe:** с 2026-05-26 Repair-наряд при закрытии приходует детали как новые (`production_release` на склад цеха, ремфонд не трогает) — значит «готовность» после закрытия прогноз видит обычным стоком. Реальные дыры: (a) ветка «свой ремонт» обрывается на счётчике — наряд не создаётся; (b) открытые ремнаряды прогнозу невидимы (документ создаётся и проводится одним шагом при закрытии — planned_incoming-окна нет); (c) прогноз без фильтра складов считает «доступным» сток технических локаций (`repair_fund` — неотремонтированный фонд, `scrap` — утиль, `assembly_in_progress` — детали уже внутри собираемых двигателей).
  - **GAP-4 вход:** кнопка «Ремонтный наряд (свой ремонт) (N)» в панели списка деталей (рядом с заявкой в снабжение). Shared-хелпер `buildRepairOrderItemsFromInventory` — строки `replenishment_branch='repair'` с дефектом (`rowDefectQty>0`), id детали из `__brand_part_id`/`__part_id` (строки без id пропускаются с подсказкой), агрегация по partId, qty=Σdefect. IPC `workOrders:createRepairFromDefects` по образцу `createAssemblyFromForecast`: пустой наряд → patch payload (workOrderKind=Repair, freeWorks с partId/engineId/engineNumber/brandId). Цех/услуги/цены оператор заполняет в карточке наряда. После создания — локальные `part_status_event` 'in_repair' per-деталь.
  - **GAP-4 исход:** в `closeWorkOrderAndPostDocument` (backend) после закрытия Repair-наряда — `part_status_event` 'ready_for_assembly' per (engineId, partId) из work-lines с engineId (best-effort, не валит закрытие). Синкается дженериком (operation_type — свободный текст, sync-контракт не меняется).
  - **Статус «в ремонте» у строки:** НЕ хранится в строке — деривируется (open Repair-наряд с (engineId,partId) → «в ремонте», closed → «готова к сборке»). Нет stale-состояний при удалении наряда, ноль новых синкуемых полей. Electron main `listEngineRepairPartStates` + бейдж в панели.
  - **GAP-5 канал прогноза:** (a) `loadNomenclatureStockMap`/`loadNomenclatureWarehouseBins` в forecast-сервисе при пустом фильтре исключают системные UUID-локации repair_fund/scrap/assembly_in_progress (явный выбор локации фильтром — уважается); (b) новый loader: открытые Repair-наряды → incoming-строки прогноза (pure-хелпер `buildRepairIncomingFromWorkOrderPayloads`, dayOffset=1 консервативно, склад = цех наряда, фильтр складов уважается); warning-нота «учтены открытые ремнаряды» при ненулевом канале.
  - **GAP-6 история:** носитель — операции `part_status_event` (engineEntityId=двигатель, payload `shared/src/domain/partStatusEvent.ts`). Electron main `listEnginePartStatusEvents` + сворачиваемая секция «История статусов деталей» в панели (дата · деталь · статус · наряд № · кто).
  - **Не в объёме Ф5:** автодвижение дефектных деталей в ремфонд-локацию (бизнес приходует ремонт как новую деталь — движений до закрытия наряда нет); ожидаемая дата готовности ремнаряда (поля нет в payload — dayOffset=1).

Слотинг: стоит на parts-chain-нитке (источник списка) — не конфликтует, надстройка.

## 3. Вопросы владельцу (вернуть с предложенными вариантами — §8 директивы)

1. **Набор статусов дефектовки.** Сейчас в модели — *количества* (`repairable_qty/scrap_qty/replace_qty`). Предложение: оставить количественную модель (она уже печатается и питает MVP-1) + добавить per-деталь **ветку восполнения** (заказчик/свой ремонт/закупка) отдельным полем. Альтернатива — перейти на enum-статус детали. **Вопрос: количественная модель ОК, или нужен явный enum-статус?**
2. **Кто утверждает** переход в требование закупки и создание наряда. Предложение: переиспользовать существующий гейт снабжения (`signed → director_approved`) + роль. **Вопрос: достаточно director_approved, или отдельная роль/право?**
3. **Версионирование актов.** Предложение: **хранить версии** (для «паспорта ремонта» / истории), не перезаписывать. **Вопрос: версии или перезапись?**
4. **Подписи в актах** — кто подписывает. Предложение: оставить текущие плейсхолдеры, заложить место, настроить позже. **Вопрос: список подписантов сейчас или потом?**
5. **Терминология** «акт» vs «лист» в печатных формах. **Вопрос: финальные названия?**
