# План: капитальная доработка нарядов (карточка, список, шаблоны, отчёт)

> Статус: **черновик на утверждении владельца.** Создан 2026-06-27 (Claude Opus 4.8, `rmz4val`).
> Источник — устный запрос владельца (большой набор правок карточки наряда). Разведка кода: 6 агентов, см. §«Заземление».
> Язык плана — русский; идентификаторы/файлы — как в коде.

## Цель

Сделать работу с нарядами удобной для оператора: убрать «грабли» в подписях, привести шаблоны в актуальное состояние, переорганизовать список нарядов с датами/статусами/цветом, добавить отчёт по нарядам с двухпанельным конструктором фильтров.

Запрос разложен на **5 этапов**. Этапы 1 и 4 независимы (можно в любом порядке). Этапы 2→3→5 — единая цепочка (2 закладывает данные для 3 и 5).

---

## Что УЖЕ есть в коде (экономит работу — прочитать первым)

Разведка показала, что несколько «хотелок» уже частично реализованы — это снимает объём:

1. **Заморозка деталей «выдан → выполнен» УЖЕ работает для сборочных нарядов.**
   `saveAssemblyWorkOrderDraft()` (draft→`open`) резервирует детали (`erpRegStockBalance.reservedQty`), они **исключаются из прогноза, но не списываются**. `postAssemblyWorkOrder()` (`open`→`closed`) снимает резерв и **списывает** в `assembly_in_progress`. То есть `open` = «выдан/заморожен», `closed` = «выполнен/списан» — ровно поведение, которое описал владелец.
   → Этап 2 по складу = **в основном переименование/отображение статусов**, а не новая логика списания. (`workOrderClosingService.ts:767–960`, `warehouseService.ts:2803–3143`, `warehouseForecastService.ts:57–90`).

2. **Переупорядочивание подписей не требует схемы.** Подписи лежат в `payload.signatureBlocks[].slots[]`, порядок = индекс массива, и это **единый источник** для карточки И для печати (`buildPrintModel()`). Переставил элементы массива → печать меняется автоматически. Компонент стрелок `RowReorderButtons` уже есть (используется для бригады и строк работ). (`WorkOrderDetailsPage.tsx:1031–1073, 1376–1454`).

3. **Список нарядов уже умеет:** переупорядочивание колонок и их скрытие (`VirtualTable` + `ColumnSettingsButton` + `useColumnLayout`), подсветку строк, статус-бэйджи с 3 цветами (`StatusBadge`). Добавляем колонки/цвета — каркас готов. (`WorkOrdersPage.tsx:44–65, 240–259`).

4. **Двухпанельный конструктор отчёта (фильтры слева, превью справа) уже есть** — это `ReportPresetPage.tsx`, ровно как у «прогноза сборки». Новый отчёт получает этот UI бесплатно; нужно лишь добавить пресет + builder. Уже есть 3 отчёта по нарядам (`work_order_costs`, `work_order_payroll`, `work_order_payroll_summary`) как образец. (`shared/src/domain/reports.ts`, `reportPresetService.ts:1589+`, `ReportPresetPage.tsx`).

5. **Шаблоны уже имеют механизм вкл/выкл полей** (`hiddenFieldsJson` + `HIDABLE_FIELD_CATALOG`). Расширяем каталог, а не строим с нуля. (`workOrderTemplate.ts`, `WorkOrderTemplateEditorDialog.tsx:38–79`).

## Решения владельца (зафиксированы 2026-06-27)

- **D1 (Этап 1) → ГИБРИД.** Кастомные формулировки подписей — в **общей БД** (одинаковы у всех клиентов, синк); список «последних использованных» сотрудников — **локально** у оператора (`localStorage`).
- **D2 (Этап 2) → ВЫЧИСЛЯЕМЫЕ.** Хранимые поля минимальны: `dueDate` (план) + `completedAt` (факт). Статусы `выдан/выполнен/просрочен/выполнен-с-просрочкой` **вычисляются** из статуса+дат (presentation), без переделки workflow несборочных нарядов.
- **D3 (Этап 2) → принято предложение.** Даты в шапке: `orderDate` (создан, уже есть) + `startDate` (приступить, план) + `dueDate` (срок, план) + `completedAt` (факт).
- **D4 (порядок) → ВЕСЬ ПЛАН ПО ПОРЯДКУ:** 1 → 4 → 2 → 3 → 5, каждый этап отдельным PR.

## Этап 1 — Подписи в карточке наряда (независимый, ежедневная боль, без миграций)

**Цель:** аккуратные, читаемые подписи; переустановка порядка стрелками (с отражением в печати); кастомные формулировки с запоминанием; подсказка «последние использованные».

**Файлы:** `electron-app/src/renderer/src/ui/pages/WorkOrderDetailsPage.tsx` (секция подписей `1376–1454`, печать `1031–1073`), `electron-app/src/renderer/src/ui/components/SearchSelect.tsx`, `electron-app/src/renderer/src/ui/components/RowReorderButtons.tsx` (reuse), `shared/src/domain/workOrderSignatures.ts` (suggestions).

**Шаги:**
1. **1.1 Стрелки ↑↓ для подписей.** Добавить `RowReorderButtons` в строку подписи (как у бригады/работ), `moveArrayItem` по `slots[]`, сохранить через `setSignatureSlots()`. Порядок попадает в печать автоматически (единый `slots[]`).
2. **1.2 Чистка вёрстки.** Перекомпоновать 4-колоночный grid строки подписи так, чтобы выбранное ФИО и должность/цех были видны полностью; убрать обрезание (`maxWidth:240 + ellipsis` у суффикса должности `1423`); расширить `SearchSelect`. Цель — оператор видит, что выбрано/набрано.
3. **1.3 Кастомные формулировки с запоминанием.** Сейчас caption — свободный ввод с `datalist` (`WORK_ORDER_SIGNATURE_CAPTION_SUGGESTIONS`, 7 шт.). Добавить персист новых формулировок (хранилище — **D1**) и подмешивать их в `datalist`.
4. **1.4 «Последние использованные» в выпадающем меню.** Когда поле пустое и оператор открыл `SearchSelect` для выбора из персонала — показать сверху последних выбранных сотрудников для этого вида поля. Реестр «последних» (хранилище — **D1**).

**Проверка (`/verify`):** открыть наряд → добавить 3 подписи, переставить стрелками → распечатать (preview) → порядок в печати совпал; ввести новую формулировку, переоткрыть наряд → она в подсказках; очистить поле ФИО → в списке сверху последние выбранные.

**Риски:** ABI better-sqlite3 под Electron перед `/verify` (см. `docs/machines/rmz4val.md`). Печать — проверить обе колонки блоков (issue/completion).

## Этап 2 — Даты и статусы наряда (фундамент для 3 и 5)

**Цель:** хранимые `dueDate`/`completedAt`; шапка карточки с 4 датами; вычисляемые статусы `выдан/выполнен/просрочен/выполнен-с-просрочкой`. Складскую заморозку **не трогаем** (уже работает — см. §«Что уже есть» п.1), только отображаем.

**Файлы:** `shared/src/domain/workOrder.ts` (payload v3→bump + normalize), `WorkOrderDetailsPage.tsx` (шапка `1481–1494`), `shared/src/domain/workOrderSignatures.ts`/новый helper для деривации статуса.

**Шаги:**
1. **2.1 Поля payload.** Добавить `dueDate?`, `startDate?` (D3), `completedAt?`; поднять `WORK_ORDER_PAYLOAD_VERSION` (3→4), дописать `normalizeWorkOrderPayloadV*Fields` (back-compat старых нарядов). `completedAt` проставлять в момент перехода в `closed`.
2. **2.2 Шапка карточки.** Показать `orderDate` (создан), `startDate` (приступить), `dueDate` (срок), `completedAt` (факт). `dueDate`/`startDate` редактируемые при `canEditNow`. Расположение — как описал владелец (даты под номером, не на самом верху).
3. **2.3 Деривация статуса (helper).** Функция `deriveWorkOrderStatusBadge(payload, now)` → `{ code: 'issued'|'done'|'overdue'|'done_late', ... }` из статуса + `dueDate`/`completedAt`. Чисто presentation (см. D2). Подтвердить, что для сборочных `open`=«выдан», `closed`=«выполнен».

**Проверка:** наряд с `dueDate` в прошлом и не закрыт → `overdue`; закрыт после срока → `done_late`; закрыт в срок → `done`.

**Риски:** payload-версия синкается в `metaJson` на все клиенты — back-compat normalize обязателен (старые наряды без новых полей). Не путать с read-authz B2 (наряды не в фильтре приватности).

## Этап 3 — Список нарядов (зависит от Этапа 2)

**Цель:** новый состав колонок (переставляемых), цветовая подсветка статусов, колонка ответственного.

**Файлы:** `electron-app/src/renderer/src/ui/pages/WorkOrdersPage.tsx` (колонки `240–259`, `StatusBadge` `44–65`), `electron-app/src/main/services/workOrderService.ts` (`listWorkOrders` `312–401`).

**Шаги:**
1. **3.1 Состав колонок** (переставляемые, каркас уже есть): дата выдачи, № наряда, дата исполнения (`dueDate`), виды работ, марка двигателя, № двигателя, дата факт. завершения (`completedAt`), **ответственный** (ФИО). Порядок/видимость — через существующий `ColumnSettingsButton`.
2. **3.2 Расширить `listWorkOrders`.** Сейчас запрос не отдаёт подписи/марку/№двигателя/даты для списка. Добавить в результат: `dueDate`, `completedAt`, `engineBrand`, `engineNumber`, `responsibleName` (первый подписант блока `issue` → `signatureBlocks[0].slots[0].employeeId`, резолв ФИО). Без N+1 (резолв в одном проходе).
3. **3.3 Цвет строк/бэйджей:** жёлтый = выдан, зелёный = выполнен, красный = просрочен (срок прошёл, не закрыт), зелёный фон + **красная дата** = выполнен с просрочкой (`completedAt > dueDate`). Расширить `StatusBadge`/`rowProps` (подсветка строк уже поддержана `VirtualTable`).

**Проверка:** список показывает все колонки; строки красятся по `deriveWorkOrderStatusBadge`; просроченный-выполненный — зелёный с красной датой; ответственный = первый «принял».

## Этап 4 — Шаблоны нарядов: цех + синхронизация полей + тумблеры + подписи (независимый)

**Цель:** добавить «цех» в шаблон; подтянуть все поля наряда в шаблон; вкл/выкл каждого поля; **шаблонировать подписи** — в шаблоне задать конфигурацию подписей (роли + при желании конкретные ФИО), чтобы применение шаблона сразу проставляло блок подписей в наряд.

**Файлы:** `shared/src/domain/workOrderTemplate.ts` (`HIDABLE_FIELD_CATALOG`, `UNHIDABLE_FIELDS_BY_KIND` `90–96`), `electron-app/src/renderer/src/ui/components/WorkOrderTemplateEditorDialog.tsx` (`38–79`), `WorkOrderDetailsPage.tsx` (`applyWorkOrderTemplate` `759–794`), `backend-api/src/services/workOrderTemplateService.ts`.

**Шаги:**
1. **4.1 Цех в шаблоне.** Сейчас `workshopId` есть в наряде, но **не в каталоге полей шаблона** (задать можно только сырым `payloadOverrides`-JSON). Добавить структурный селектор цеха в редактор шаблона + в `HIDABLE_FIELD_CATALOG`/apply-логику.
2. **4.2 Аудит «отставших» полей.** Свести список всех полей наряда (по типам Regular/Repair/Assembly/Manufacturing) против каталога шаблона; добавить недостающие, чтобы все поля попадали в шаблон и дублировались в наряд при применении.
3. **4.3 Тумблеры вкл/выкл.** Механизм `hiddenFields` уже есть — убедиться, что каждое поле имеет тумблер в редакторе и корректно скрывается/показывается при применении (`setAppliedHiddenFields`).
4. **4.4 Шаблонирование подписей (запрос владельца 2026-06-27).** В редактор шаблона добавить конфигуратор подписей — те же блоки `issue`/`completion` со слотами (роль + опционально сотрудник), что и в карточке (переиспользовать UI/логику Этапа 1). Хранить в шаблоне (`payloadOverrides.signatureBlocks` или отдельное поле `signatureBlocks` в `work_order_templates`). При `applyWorkOrderTemplate` — проставлять `payload.signatureBlocks` из шаблона (включая ФИО, если заданы), не затирая уже введённые оператором без подтверждения. Так оператор одним выбором шаблона получает готовую конфигурацию подписей с фамилиями. ⚠️ Зависит от Этапа 1 (общий UI слотов подписи) — поэтому в очереди Этап 4 идёт после Этапа 1.

**Проверка:** создать шаблон с заданным цехом, выкл. частью полей и преднастроенными подписями (роли + ФИО) → применить к новому наряду → цех подставился, скрытые поля скрыты, остальные продублированы, **блок подписей заполнен ролями и фамилиями из шаблона**.

## Этап 5 — Отчёт по нарядам (двухпанельный, печать, кнопка у списка) (зависит от 2/3)

**Цель:** новый пресет-отчёт с богатыми фильтрами; UI «фильтры слева / превью справа» (как у прогноза); печать; кнопка рядом с созданием нарядов.

**Файлы:** `shared/src/domain/reports.ts` (`ReportPresetId` `3–27`, `REPORT_PRESET_DEFINITIONS` `194+`), `electron-app/src/main/services/reportPresetService.ts` (builder, образец `buildWorkOrderCostsReport` `1589–1650`; switch `4120+`), `WorkOrdersPage.tsx` (кнопка), `App.tsx` (`openReportPreset` `2100–2103`).

**Шаги:**
1. **5.1 Пресет `work_orders_report`.** Фильтры: период по дате выдачи, период по сроку (`dueDate`), статус (`select`: все/выдан/выполнен/просрочен/выполнен-с-просрочкой), ответственный (`multi_select` employees), марка (`multi_select` brands), виды работ (`text`/`multi_select`), флаг «только просроченные» (`checkbox`).
2. **5.2 Builder `buildWorkOrdersReport(db, filters)`** по образцу `buildWorkOrderCostsReport`; колонки = состав списка (Этап 3) + ответственный; вернуть `{ ok, presetId, title, columns, rows, totals, generatedAt }`; добавить case в switch.
3. **5.3 Двухпанельный UI** — бесплатно через `ReportPresetPage` (превью пересобирается при изменении фильтров). При необходимости интерактива — компонент по образцу `AssemblyForecastReportView`.
4. **5.4 Кнопка у списка.** На `WorkOrdersPage` рядом с «Создать наряд» — кнопка «Отчёт по нарядам» → `openReportPreset('work_orders_report')` (deep-link, проброс коллбэка).

**Проверка:** кнопка у списка открывает отчёт; фильтры по датам/ответственному/статусу меняют превью справа; печать даёт корректную форму.

## Порядок (рекомендация)

1. **Этап 1 (подписи)** — независим, без миграций, максимальная ежедневная польза → **первым**.
2. **Этап 4 (шаблоны)** — независим, небольшой → можно вторым или параллельно.
3. **Этап 2 → 3 → 5** — связная цепочка (данные → список → отчёт) → блоком после.

Каждый этап = отдельный PR (PR-only flow, ADR-0002), зелёные гейты перед мержем, `/verify` для UI-правок.
