## Зачем внешний процесс (ядро проблемы)

NSIS one-click-инсталлятор (`electron-app/installer/installer.nsh:76`) перед установкой делает `RMDir /r "$LOCALAPPDATA\Programs\MatricaRMZ"`, затем ставит заново. **Если инсталлятор падает/убит между сносом и установкой — папка приложения и ярлыки исчезают, exe нет.** Встроенный апдейтер (`updateService.ts`) тут бессилен: он живёт ВНУТРИ приложения, которого больше нет. Значит ремонт должен делать **процесс вне приложения**, который переживает его пропажу.

| Критические события | `criticalEventsService.ts` (POST в `/diagnostics/critical-events`, severity warn/error/fatal, source=client, username/clientId) | канал «не справился → доложить супер-админу» |

**Вывод:** сеть, проверка инсталлятора, канал команд, канал критсобытий, формат диагностики — всё есть. Watchdog = тонкая внешняя обёртка над этими кирпичами + детектор «приложение исчезло».

## Дизайн (черновик — выбрать с владельцем)

### Форма внешнего процесса (главный выбор)
- **Вариант A — Scheduled Task (рекомендую).** Запись в Планировщике задач Windows: триггеры «at logon» + «every 15 min». Запускает крошечный `matricarmz-watchdog.exe`, который делает один проход проверки и выходит (не резидент). _Плюсы:_ почти ноль памяти (процесс живёт секунды), переживает краш/снос приложения, проще подписать, меньше похоже на резидентного зловреда. _Минусы:_ задержка до 15 мин; задачу создаёт инсталлятор (в `installer.nsh`).
- **Вариант B — Windows Service.** Резидентный сервис. _Плюсы:_ мгновенная реакция. _Минусы:_ память, права администратора на установку (one-click = per-user → конфликт), сильнее триггерит антивирус.
- **Вариант C — лёгкий резидент в трее.** Отметается: умрёт вместе со сносом приложения, если жил в его папке.

### Из чего собрать watchdog.exe
- Минимальный Node-/Go-/Rust-бинарь (выбрать по размеру/подписи; Node-pkg ~ привычно стеку, но крупнее; Go/Rust — мелкий статичный exe, меньше AV-шума). Не тащить Electron.
- Логика прохода (ladder):
  1. **Проверка присутствия:** есть ли `…\Programs\MatricaRMZ\MatricaRMZ.exe` + ярлык (Desktop/StartMenu)? + жив ли процесс?
  2. **Если приложение на месте и здорово** → выход (ничего не делаем).
  3. **Если exe/ярлык пропали** → найти валидный инсталлятор: pending-update.json → папка обновлений → (если нет/битый) скачать с `/updates/latest-meta`+`/updates/file/…` (паттерн `emergencyUpdate`). Проверить (MZ+sha256+size).
  4. **Тихая переустановка:** запустить инсталлятор silent, дождаться, перепроверить присутствие.
  5. **Успех** → отметить в логе, heartbeat «восстановлено».
  6. **Провал (N попыток)** → POST в критические события (severity=error/fatal, source=client, с логом-хвостом и причиной) → ждать удалённой команды.
- **Удалённое восстановление:** watchdog тоже опрашивает `/settings` (или новый лёгкий `/watchdog/command`) и понимает команду `reinstall`/`repair` от владельца из админки (расширение существующего sync-request-перечня). Владелец жмёт кнопку — watchdog при следующем проходе чинит принудительно.
- **Сбор причин падений:** при ремонте приложить хвост `matricarmz.log` + `matricarmz-updater.log` + код выхода инсталлятора к критсобытию (диагностику взять из существующих логов, не изобретать).

### Антивирус / память (требование владельца)
- Подписать `watchdog.exe` тем же сертификатом, что и клиент (если есть code-signing; если нет — поднять отдельным вопросом, иначе SmartScreen/AV будут ругаться).
- Scheduled Task (вариант A) → процесс-однопроходник, не резидент → минимальный след памяти и AV-подозрений.
- Сеть только к известному update-серверу (тот же хост, что апдейтер). Никаких сторонних доменов.
- Понятное имя/издатель/манифест.

## Фазы
1. **Согласовать форму** (A/B/C) + язык бинаря + наличие code-signing — с владельцем (блокирует реализацию).
2. **Серверная команда `reinstall/repair`** в существующий sync-request-перечень (`clientSettingsService` + `adminClients` + кнопка в админке клиентов). Маленько, переиспользует канал.
3. **watchdog.exe MVP:** проход проверки присутствия + ремонт из локального инсталлятора + критсобытие при провале. Без сети-скачивания сперва (только локальный инсталлятор).
4. **Скачивание инсталлятора** (паттерн emergencyUpdate) когда локального нет/битый.
5. **Установка задачи** инсталлятором (`installer.nsh`: создать Scheduled Task при install, удалить при uninstall).
6. **Удалённый триггер** (фаза 2 + опрос в watchdog) + сбор причин падений в критсобытие.
7. **Тест на стенде:** сымитировать «ярлыки пропали» (снести `…\Programs\MatricaRMZ`) → watchdog восстанавливает; провал → критсобытие видно супер-админу.

## Риски
- **Сам watchdog может стать source of AV-ложнопозитива** (запускает инсталляторы) → подпись + Scheduled Task + минимализм обязательны; согласовать с владельцем код-сайнинг.
- **Гонка с обычным апдейтером** (оба могут запускать инсталлятор) → watchdog действует ТОЛЬКО когда приложение отсутствует/мертво; здоровое приложение он не трогает. Уважать `update.lock`.
- **Per-user установка** (one-click, не per-machine) → Scheduled Task в пользовательском контексте, без админ-прав; Windows Service (вариант B) конфликтует с этим.
- **Watchdog тоже может не запуститься** (тот же диск/ОС) → не цель «100% надёжность»; цель — закрыть частый класс «сорванное обновление». Деградация мягкая (доклад владельцу).

## Стейджинг и проверяемость (важно — ограничения dev-стенда)

**Ограничения, влияющие на план:**
- **Go не установлен на dev-машине** (`rmz4val`) → Go-бинарь собирается/проверяется **в CI** (GitHub Actions, там Go есть), не локально. Локальная build-проверка — только если поставить Go-тулчейн.
- **Нет установленного Windows-клиента на dev** → функц-сценарий «приложение пропало → watchdog переустановил» **нельзя сымитировать на стенде**; нужен **реальный Windows-клиент** (та же преграда, что у PENDING «live-аудит раскладки клиента» — машина оператора).
- **`installer.nsh` = огромный blast radius** (ст. 1 конституции): ошибка в нём встанет у ВСЕХ обновлений. Интеграцию Scheduled Task в инсталлятор тестировать особенно осторожно, на реальной установке.

**Порядок сборки (по проверяемости и риску):**
- **Фаза 2 — серверная команда `reinstall/repair`** (чистый TS): расширить sync-request-перечень (`clientSettingsService` + `adminClients` + кнопка в админке клиентов). **Проверяется гейтами здесь и сейчас** (typecheck/lint/backend-тесты). Нулевой риск для установки. Фундамент удалённого триггера.
  - **✅ Реализовано 2026-06-22.** Технические решения (приняты в реализации, не выносились владельцу — см. memory `delegate-technical-decisions`):
    1. **Один тип `reinstall`** (не `reinstall`+`repair`): действие одно — тихий запуск инсталлятора; behavioral-разницы между ними план не задавал → вторая команда = лишняя абстракция (против code-style «No abstractions beyond what the task requires»). При появлении реальной разницы — добавить тогда.
    2. **Переиспользуем существующий sync-request слот** (без новой колонки/миграции — G74 drizzle snapshot drift = лишний риск). Команда `reinstall` едет в тех же `sync_request_*` колонках.
    3. **Гонка двух потребителей** одного слота: Electron-клиент опрашивает `/client/settings` каждые 60с и потребляет+аккает sync-request; Go-watchdog — раз в ~15 мин. Чтобы клиент не «съел» watchdog-команду раньше watchdog'а → **клиент явно пропускает тип `reinstall`** (`isWatchdogCommand` guard в `clientAdminService.ts`): не аккает, не очищает, не запускает sync — оставляет в слоте. Watchdog (Фаза 3–4) читает `syncRequestType==='reinstall'` и аккает через тот же `/settings/sync-request/ack`.
    4. **Компромисс single-slot:** пока watchdog-команда висит в слоте, владелец не может одновременно поставить sync-команду (`sync_now` и т.п.) этому клиенту — слот один. Приемлемо: watchdog-команды редки и снимаются watchdog'ом за ≤15 мин. Если станет тесно — разнести в отдельные колонки (миграция).
- **Фаза 3–4 — `watchdog.exe` (Go):** написать + добавить шаг сборки в CI (build-verified в CI). Функц-тест — на реальной машине.
  - **✅ Реализовано 2026-06-22.** Решения (приняты в реализации — `delegate-technical-decisions`):
    1. **Handshake-файл вместо чтения SQLite клиента.** Go-бинарь не реверсит именование Electron и не парсит SQLite/WAL — app пишет `%APPDATA%\MatricaRMZ\watchdog.json` (`watchdogHandshakeService.ts`, хук в `applyRemoteClientSettings` каждые 60с) с `clientId/apiBaseUrl/version/appExePath/userDataDir/updatesRootDir/updaterLogPath/appLogPath`. Путь вычислим из `%APPDATA%` (через `app.getPath('appData')`, не productName) и переживает снос install-папки. **Причина:** `app.getName()` в запакованном клиенте → `MatricaRMZ`, в dev → `@matricarmz/electron-app` → userData неоднозначен; читать чужой SQLite Go-кодом — худшее «новое», чем явный JSON-шов.
    2. **Bridge-эндпойнт `/client/watchdog/report`** (неаутентифицированный, рядом с `/client/settings`) → `ingestServerCriticalEvent`. Эндпойнт критсобытий `/diagnostics/*` весь под `requireAuth`, а watchdog без сессии. Провал/успех восстановления → «Критические события» (политика `notification_channels_decision`).
    3. **Чистый stdlib** (без go.mod-зависимостей) → тривиальная CI-сборка, ~малый exe, минимум AV-шума. Уважает `update.lock` (не гонит инсталлятор поверх работающего апдейтера). Ладдер: handshake → poll `reinstall` → presence → найти инсталлятор (pending-update.json → updates-dir → download) → validate (MZ+size+sha256) → silent `/S` → recheck → report + ack.
    4. **CI build на `ubuntu-latest`** (`go vet` + cross-compile `GOOS=windows`) — артефакт unsigned. **Подпись отложена в Фазу 5** (нужны секреты CSC_* в релизном контуре + распространение бинаря клиентам через installer) — не трогаю load-bearing release workflow преждевременно.
- **Фаза 5 — Scheduled Task в `installer.nsh`:** высокий риск → отдельным осторожным шагом + тест на реальной установке.
- **Фаза 7 — функц-тест** «снести `…\Programs\MatricaRMZ` → watchdog восстановил» — на реальном Windows-клиенте.

## План релиза watchdog (2026-06-22)

Watchdog раздаётся **unsigned** (серта нет — см. статус-блок). Релиз обычный по `CLAUDE.md` §Release process — watchdog уезжает сам:

**Предусловия (уже в `main`):**
- Фаза 5a — `installer.nsh` ставит Scheduled Task + копирует бинарь; watchdog вшит как `win.extraResources`, собирается Go-шагом релизного workflow. Ничего отдельного делать не надо — попадёт в инсталлятор автоматически.
- **Разводка кнопки** (`feat/watchdog-reinstall-button-version-gate`): кнопка «Переустановить» и серверная команда `reinstall` гейтятся `clientHasWatchdog(lastVersion)` (граница `WATCHDOG_ROLLOUT_AFTER_VERSION = 2026.622.1241`). Поэтому **безопасно выпускать кнопку и watchdog одним релизом** — кнопка спит для необновлённых клиентов и зажигается per-client по мере обновления. Ручного флипа/второго релиза не нужно.

**Шаги релиза** (стандартные): `bump-version` → эпиграф + «что нового» в `releaseWelcome.ts` → PR → tag `vX.Y.Z` → GitHub Actions собирает инсталлятор (watchdog внутри) → на проде собрать серверные пакеты → скачать артефакты updater → `ledger-publish` → restart → проверить `/health` + `/updates/status`.

**Постельная проверка (после раската):**
1. Дать клиентам обновиться. В админке «Клиенты»: у обновившегося клиента (версия > `2026.622.1241`) кнопка «Переустановить» становится активной — у необновлённых остаётся серой с подписью «нет watchdog — после обновления».
2. На одном реальном обновлённом клиенте — опц. полный Фаза-7 тест «снести install-папку → watchdog восстановил» (на стенде `rmz4val` уже зелёный; на проде по желанию владельца).
3. **Мониторить AV-ложнопозитивы** unsigned-watchdog'а — критсобытия/жалобы операторов первые дни. Если карантинят — рассмотреть исключение в их AV или вернуться к вопросу подписи.

**Когда включать кнопку для всех:** отдельного «включения» нет — кнопка сама активна для каждого клиента, у кого версия с watchdog. Полезной она станет, когда заметная часть парка обновится.
