# Нижний поиск у списков + глобальный поиск «ищет везде» — 2026-07

**Задание владельца (2026-07-04, голосом):**
1. У **каждого списка внизу** — окно поиска. Ищет **внутри карточек**, отображённых в списке; результат — не открытие карточки, а **фильтрация списка** до карточек, где нашлось.
2. Верхний глобальный поиск (Ctrl+K) остаётся как есть по месту, но растёт до «ищет вообще везде»: интерфейс (надписи на кнопках/меню), карточки, база. Результаты — списком с указанием **пути/места** («где нашёл»), по клику — открыть найденное; если открыть нельзя (элемент интерфейса) — показать, где это находится.

## Что уже есть (разведка 2026-07-04)

- Глобальный Ctrl+K: `GlobalSearchOverlay.tsx` — 3 уровня: L1 (строки текущей страницы через `globalSearchScope`), L2 (справочники в памяти, `globalSearchSources.ts`), L3 (сервер `/search`: номенклатура + складские документы, `globalSearchService.ts`). Навигация по хиту — `searchHitToRoute` → `DeepLinkRoute` (App.tsx открывает карточку). Группировка по kind есть, **пути «Меню → Раздел» нет**, интерфейс не ищется, вглубь карточек (EAV) не ищет.
- Списки: у большинства есть верхний фильтр по **загруженным строкам** (`filterPreparedRecords` / `matchesQueryInRecord`, `ui/utils/search.ts` — глубокий обход полей строки, tiered-matcher с RU/EN-раскладкой). Внутрь карточки (EAV `attribute_values` локального SQLite) никто не ищет; нижней панели нет нигде.
- IPC поиска по атрибутам entity нет — надо заводить.

## Решения (техвыбор за Claude по памяти `feedback_delegate_technical_decisions`)

- **Семантика нижнего поиска:** фильтрует **отображённый** список (текущие загруженные строки; для серверно-пагинированных — загруженную страницу). Совпадение = tier-1 по полям строки (существующий matcher) **ИЛИ** tier-2 по содержимому карточки (EAV) для entity-списков.
- **Tier-2 (внутрь карточек):** новый IPC `search:cardContent` (main, локальный SQLite): `{ entityIds: string[], q: string }` → `string[]` (ids, где `attribute_values.value_json LIKE %q%`, живые строки). Без сервера — база уже на клиенте (display-filter модель). Кап 5000 ids, дебаунс 250мс, нормализация lower + компакт (как tiered).
- **UI:** переиспользуемый `ListSearchBar` (нижняя панель: инпут + счётчик «N из M» + очистка, Esc чистит). Один компонент — везде одинаковый.
- **Хук:** `useListDeepFilter(rows, getId, { entityBacked })` — объединяет tier-1 sync + tier-2 async, отдаёт `{ filtered, query, setQuery, matchInfo }`.
- **Глобальный: L0 «Интерфейс»** — курируемый реестр `uiSearchRegistry.ts`: `{ label, synonyms, path: 'Меню → Производство → Наряды', tabId }` для пунктов меню/страниц/ключевых кнопок и надписей. Хит kind `ui`: клик → переход на вкладку; в строке — путь. Реестр статический (кода), пополняется по мере жалоб «не нашёл».
- **Глобальный: путь у всех хитов** — статическая карта kind → «Меню → Раздел» (`globalSearchKindPath`), подстрока-путь под лейблом в overlay.
- **Глобальный: L2.5 «Внутри карточек»** — тот же `search:cardContent` без ограничения ids (по всем entity-типам, top-N на тип), хиты открывают карточку, в строке пометка «найдено в содержимом карточки».
