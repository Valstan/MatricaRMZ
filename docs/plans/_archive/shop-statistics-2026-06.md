## 0. Ответы владельца (2026-06-09) — финализируют модель

- **Q1 «где фиксируется выпуск цехом сегодня» → НИГДЕ.** Цех-привязка выпуска не пишется ни одним носителем (lifecycle-EAV двигателя выпуск несёт, цех — нет).
- **Q5 «один цех или несколько за ремонт» → НЕСКОЛЬКО.** Двигатель проходит цепочку цехов (разборка → ремонт узлов → сборка). Цех — **не одно поле на двигателе, а маршрут-последовательность этапов**.
- **Q2 «механизм захвата» → НАРЯД НА КАЖДЫЙ ЭТАП.** Каждый этап ремонта оформляется нарядом в своём цехе.
- **Q4 «объём учёта» → ТРУД + ПРОХОЖДЕНИЕ ДВИГАТЕЛЕЙ** (не материальный выпуск деталей/узлов).

**Следствие — задача схлопывается:** нужный носитель = **сам наряд**, который УЖЕ стампит цех (`meta_json.workshopId`) + привязку к двигателю (`engine_entity_id`) + труд (работы/начисления/бригада). При Q4=труд+прохождение **нового поля захвата не нужно** — модель уже всё пишет. Маршрут двигателя (Q5) = упорядоченная по времени последовательность его нарядов по цехам, собирается сама. **Реальный левередж — не схема, а adoption:** добиться, чтобы наряд на свой этап выписывал каждый цех (сегодня только Цех №1). `production_release`-путь материального выпуска (§2) при Q4 **не нужен** — Фаза 2 снимается.

## 2. Дизайн точки захвата — откуда брать цех-привязку выпуска

Ключевые верифицированные факты кода:

- **Наряд уже несёт ОБА конца нити:** `workOrderClosingService.ts:235` (engine link) + `:259` (жёсткий гейт `workshopId` для non-Regular) + `:387,392` (складской `production_release`-документ получает `workshopId` + `engineId`). **Модель уже умеет workshop-attributed материальный выпуск** — но срабатывает только на non-Regular нарядах, которых на проде 0. Regular-наряды (единственные используемые) гейт workshopId **пропускают** (`:240–251`) и складского документа не создают.
- **Lifecycle-переход двигателя ловит ТОЛЬКО актора, не цех:** `enginePhaseService.ts:189–205` пишет `audit_log.actor = username` + `{from,to}`, **без цеха**; то же `adminMasterdataService.recordEngineStatusAudit` для `is_scrap`/статусов. `args.actor` доступен.
- **Цепочка actor→цех неполна:** сотрудник→`department_id` (EAV link) есть; **department→workshop отсутствует** (Department несёт только `name`). Вывести цех из «кто менял статус» сегодня нельзя — не хватает одного маппинга.

**Кандидаты захвата (оценка «что замкнёт нить дешевле всего»):**

| # | Механизм | Стоимость | Покрывает | Зависит от ответа владельца |
|---|---|---|---|---|
| **A** | **Вывести цех из наряда:** при выпуске/смене фазы двигателя, инициированной нарядом, проставить цех двигателя из `workshopId` наряда (оба конца уже связаны) | ~дёшево (связь есть) | выпуск, идущий **через наряд** | Q1/Q2: идёт ли выпуск через наряд |
| **B** | **department→workshop маппинг:** добавить связь подразделение↔цех → любой lifecycle-переход выводит цех из актора (`enginePhaseService`/`recordEngineStatusAudit`) | средне (1 связь + статичный маппинг, не добэкфилл данных) | **все** статус-переходы (1460 выпусков) | Q2: считать ли цех = подразделение исполнителя |
| **C** | **Новое явное поле у источника:** дропдаун «цех» на UI смены статуса двигателя | средне + нагрузка оператора | то, что оператор заполнит | Q3: согласен ли владелец на ручной ввод |

Рекомендация (предварительно, до ответов): **A + B комбинированно** — A замыкает наряд-driven выпуск почти бесплатно (связь готова), B даёт цех для остальных 1460 lifecycle-выпусков через актора без ручного ввода. C — только если процесс требует явного выбора цеха, не выводимого из A/B. Финал — после Q1–Q3.

## 3. Фазовый план

Финализировано ответами владельца: захват = наряд-на-этап (нового поля не нужно), объём = труд + прохождение двигателей, цех-маршрут = последовательность нарядов. Фаза «материальный выпуск» **снята** (Q4).

- ✅ **Фаза 0 — агрегирующий экран на `operations`** (read-only, zero-migration) — **ПОСТРОЕНА 2026-06-09:** «Статистика цехов» — открыл цех → по нему из нарядов: **труд** (работы / сумма начислено / бригада) + **прохождение двигателей** (distinct `engine_entity_id`, число этапов-нарядов, период). **Маршрут двигателя** (Q5) — упорядоченная по времени последовательность его нарядов по цехам — собирается из связей. **Явно помечается охват** «показаны цеха, выписывающие наряды» (сегодня = Цех №1) — не выдаётся за весь завод (#023). Honest-MVP, наполняется по мере adoption **без миграций**. Файлы: `shared/src/domain/workshopStats.ts`, `backend-api/src/services/workshopStatsService.ts`, route `GET /workshops/stats`, `electron-app/.../pages/WorkshopStatsPage.tsx`.
- **Фаза 1 — adoption захвата (процесс, не схема):** добиться, чтобы наряд на свой этап выписывал каждый цех, не только Цех №1. Модель уже всё стампит — нужно облегчить операторам выписку наряда per-этап (UI-нуджи, если упрётся) + договорённость с владельцем о процессе. Дашборд Фазы 0 наполняется сам по мере роста adoption.
- **Фаза 2 — материальный выпуск/межцех — СНЯТА** (Q4: считаем труд+прохождение, не материал). Вернуть только по явному будущему запросу владельца; `production_release`+`workshop_id` FK уже готовы в модели, если понадобится.
- **Фаза 3 — обогащение дашборда** по мере adoption и фидбэка: тренды, сравнение цехов, разрез по маркам двигателей (`engine_brand` плотно заполнен).

Слотинг (§5 pool #031): Фаза 0 на `operations`/`engine_brand` — не параллелит и не конфликтует с `parts-chain-audit` (она достраивает component-пулы деталей, не цех-измерение).

## 4. Вопросы владельцу — ✅ отвечены 2026-06-09 (см. §0)

**Q1/Q2/Q4/Q5 отвечены** (см. §0 — финализировали модель). **Q3 (почему только Цех №1)** свёрнут в Фазу 1 (adoption): остальные цеха ещё не выписывают наряды; нить замкнётся по мере того, как начнут (модель уже готова). Исходные формулировки оставлены ниже как след постановки.

1. **Что считать «выпуском цеха» и где это фиксируется СЕГОДНЯ** в реальном процессе — наряд? смена статуса двигателя? приёмо-передача? (определяет носитель захвата — знание процесса, не данных).
2. **Откуда брать цех-привязку выпуска** без дубль-учёта: из наряда (A) / из подразделения исполнителя, менявшего статус (B, нужен маппинг department→workshop) / новое явное поле (C)?
3. **Почему наряды пишет только Цех №1?** Остальные 6 цехов реально не выписывают наряды в системе, пишут вне её, или просто ещё не начали? (определяет, замкнётся ли нить сама собой по мере adoption нарядов, или нужен другой носитель).
4. **Материальный выпуск и межцех-передачи per-цех — нужны ли в учёте вообще**, и где они фиксируются в реальном процессе? (сейчас `producedLines`/`stock_transfer`/движения пусты — Фаза 2 строится только если это реальный процесс).
5. **Двигатель проходит ОДИН цех за ремонт или НЕСКОЛЬКО?** Если несколько — цех-привязка = последовательность этапов (нить-маршрут), а не одно поле; это меняет модель захвата.
