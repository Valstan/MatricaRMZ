## Архитектурный вердикт (кратко)

1. **Модульность (идея 1) — да, но лениво и без рефактора ядра.** Renderer уже грузит страницы
   через `import.meta.glob` — «модуль» ложится на существующую механику как каталог
   `userData/modules/<id>/` с манифестом + динамический `import()`. Host-API фиксируем на
   минимуме только для пилота. Полноценный each-page-is-a-plugin рефактор — НЕ делаем
   (дорого, ломает рабочую структуру, ценность недоказана).
2. **Конструктор (идея 2) — пилот уровня 1 БЕЗ нейронки.** Выход конструктора = JSON-спек
   (дерево компонентов из нашей дизайн-системы: кнопка/поле/список/селект + intent-привязки
   к существующим действиям). Рендер — schema-driven из того же реестра компонентов.
   Это закрывает «хочу свою форму/список» без кодогена и без attack-surface загрузки кода.
3. **Уровень 2 (нейронка генерит модуль) — отложен до доказанной потребности.** Гейт:
   typecheck/lint/тесты + рантайм-прогон verifier-electron (pool #047 verified≠exercised)
   + human-in-loop ревью. Не раньше, чем уровень 1 покажет реальные запросы операторов,
   которые в JSON-рендер не помещаются. Также упирается в гео-блок Anthropic (PENDING 🔴).

## Пилот (одна нитка, ~3 PR)

- **PR1 — реестр + рендерер:** `shared/src/domain/uiSpec.ts` (типы спека + sanitize),
  renderer `ui/uiBuilder/SpecRenderer.tsx` (JSON → наши компоненты), реестр допустимых
  intent'ов (навигация на таб, открытие карточки, запуск отчёта). Без persistence.
- **PR2 — конструктор:** страница-редактор (палитра = реестр, drag-drop через @dnd-kit,
  живой превью через SpecRenderer), сохранение спеков per-user (sysDb KV, как UiShellPrefs;
  шаринг на весь завод — отдельным решением владельца).
- **PR3 — «мои экраны»:** пункт меню, список сохранённых спеков, открытие как вкладки.
  CDP-verify всей петли (собрал → сохранил → переоткрыл → intent сработал).

Оценка: пилот целиком ~1 длинная сессия. Puck/GrapesJS не берём (лишняя зависимость,
чужая модель данных); dnd-kit уже в проекте.
