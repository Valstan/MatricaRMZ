# Учёт ремонтного фонда (ремфонд) — план

> Статус: **активный**. Заведён 2026-06-28 (Claude Opus 4.8, `rmz4val`). Владелец заказал «продумать учёт ремфонда» + сделать инструмент массового ввода.

## Проблема (словами владельца)

Ремфонд в программе как регистр **не ведётся**. Дефектовка/комплектность есть, но после них детали «уходят в пустоту» (обезличивание). Наряды создают уже **готовые** (отремонтированные) детали для сборки. В ремфонд ничего не добавляется → «бездонная яма». Нужен учёт деталей, **ожидающих ремонта**: ещё не отремонтированы, но пригодны (не утиль, редактируемо).

## Что выяснила разведка (2026-06-28)

- **Локация `repair_fund` («Ремонтный фонд») уже существует** как системная: `shared/src/domain/warehouseLocations.ts` (`WAREHOUSE_LOCATION_REPAIR_FUND='repair_fund'`), сидируется `seedSystemLocations()` в `backend-api/src/services/warehouseLocationsService.ts`. Остатки `erp_reg_stock_balance` держат qty по `(nomenclatureId, warehouseLocationId)` для любой локации; движения/проводка документов работают.
- **Поток наполнения заморожен 2026-05-26** (`FEATURE_ENGINE_DISMANTLE=false`, `EngineDetailsPage.tsx`): «разборка→repair_fund» отключили из-за призраков (списки деталей по маркам неактуальны). Ремонт сейчас приходует детали как новые на склад цеха через `production_release`. Поэтому ремфонд пуст.
- **Акты комплектности/дефектовки** (хранятся в `operations`, payload `engine_inventory`, модель строки `EngineInventoryRow` в `shared/src/domain/repairChecklist.ts`) несут всё для авто-наполнения: `present` (на месте), `scrap_qty` (утиль), `repairable_qty`, `replenishment_branch='repair'`, **`stamped_number`** (личный номер). Правило «годна к ремонту» = `present && scrap_qty===0` (точнее — ветка `replenishment_branch==='repair'`).
- **Жизненный цикл ремонта** размечен частично: создание ремонтного наряда из дефектовки (`workOrders:createRepairFromDefects`) пишет `part_status_event='in_repair'`; закрытие (`workOrderClosingService.closeWorkOrderAndPostDocument`, ветка Repair) → `production_release` + `part_status_event='ready_for_assembly'`. Сюда вешается **расход из ремфонда** (Фаза 2).
- **Личные номера деталей поэкземплярно НЕ учитываются**: регистр экземпляров `erp_part_cards` удалён (миграция 0060, мёртвый). Serialized только двигатели (`erp_engine_instances.serialNumber`). Но `stamped_number` в актах есть — образец для Фазы 3.
- UI остатков/документов/инвентаризации: `StockBalancesPage.tsx`, `StockDocumentsPage.tsx`, `StockDocumentDetailsPage.tsx`, `StockInventoryPage.tsx`. Отдельного UI ремфонда нет.

## Модель статуса детали

«В ремфонде / ожидает ремонта» = деталь числится на регистре `repair_fund` (qty>0). **Отдельно** от: утиль (`scrap`), отремонтирована (склад цеха / `ready_for_assembly`). Редактируемо (ревизией). Не новый статус-флаг на детали, а **остаток на локации ремфонда** (количественно) — Фаза 0/1; поэкземплярно по личному номеру — Фаза 3.

## Решения владельца (2026-06-28)

1. **Фаза 0 — только количество** (личные номера — позже, Фаза 3).
2. **Регистр — на готовой локации `repair_fund`** (бесплатные складские остатки/отчёты, без миграции).

## Дорожная карта

- **Фаза 0 — «Ревизия ремфонда» (✅ сделано 2026-06-28, #638):** инструмент массового прямого ввода списка «деталь + кол-во» в ремфонд, минуя дефектовку. Семантика **ревизии** (set-абсолют): оператор видит текущее содержимое ремфонда, вносит фактические количества (добавляет строки для новых деталей), проводит → остатки `repair_fund` приводятся к внесённым. Опора на существующую механику документов склада (инвентаризация на локации `repair_fund`). Страница `RepairFundAuditPage`, вкладка в меню «Склад».
- **Фаза 1 — приход из дефектовки (✅ сделано 2026-06-28, #639; решения владельца 2026-06-28):** кнопка на карточке двигателя «Занести дефектовку в ремфонд». Заносятся **годные к ремонту** детали = `repairable_qty` по строкам с `present===true` (НЕ `scrap+replace` — то дефектные). **Идемпотентность по двигателю:** бэкенд считает целевой набор {nomenclatureId→qty}, сравнивает с сохранённым прошлым заносом этого двигателя (операция `repair_fund_intake`), проводит **дельту** (+приход inventory_opening / −расход writeoff) в `repair_fund`, перезаписывает запись заноса. Повтор без изменений → ноль. **Защита от призраков:** только реальные данные дефектовки (не теоретический BOM марки — он давал призраков), дельта-сверка на двигатель, «Ревизия ремфонда» (Ф0) — страховка. partId строк берётся из мета `__brand_part_id`/`__part_id` (как `buildRepairOrderItemsFromInventory`).
- **Фаза 2 — авто-расход при ремонте (✅ сделано 2026-06-28):** закрытие ремонтного наряда списывает отремонтированное из ремфонда (qty из строк наряда). Хук — `workOrderClosingService` ветка Repair (после `writeRepairReadyPartStatusEvents`, динамический import → нет цикла), best-effort + clamp по остатку (`stock_writeoff` из `repair_fund`, не уходит в минус, не валит закрытие). `repairFundService.releaseRepairFundForWorkOrder`. Верифицировано CDP: закрытие наряда (qty 2) → фонд −2.
- **Фаза 3 — личные номера (✅ сделано 2026-06-28; решения владельца 2026-06-28):** поэкземплярный учёт деталей с `stamped_number` **с провенансом** (с какого двигателя снята) — для требования к заказчику по конкретному двигателю. **Хранилище — строки `operations` (`operationType='repair_fund_instance'`), НЕ новая синхро-таблица** (полноценная `erp_*` = ~20 файлов sync-обвязки + миграция + риск cold-start; `operations` уже синкается, образец — `part_status_event`). Запись = `{engineEntityId, nomenclatureId, partId, partLabel, stampedNumber, classification: repairable|scrap|replace, status, capturedAt/By}`, ключ `(engine, nomenclature, stampedNumber)`, идемпотентный серверный upsert. Захват — кнопка «📌 Зафиксировать личные №» в `RepairChecklistPanel` рядом с Ф1 (хелпер `buildStampedInstancesFromInventory`); просмотр — секция там же. Backend: `repairFundInstanceService` + route `/warehouse/repair-fund/instances/capture-from-engine`. Тип/парсер — `shared/src/domain/repairFundInstance.ts`.
  - **Ф3.1 (отложено):** переход `in_fund→repaired` на закрытии ремнаряда. Причина отсрочки — наряды количественные (partId+qty, без личного номера), сопоставить *какой* экземпляр отремонтирован можно только эвристически; ценность маргинальна (претензии держатся на классификации утиль/замена с дефектовки). См. `PENDING_FOLLOWUPS.md`.
  - **⚠️ Sync-фикс Ф3 (в составе Ф4):** server-написанный захват делал голый `db.insert(operations)` без `recordSyncChanges` → строки не получали `last_server_seq` и **не доезжали клиентам** (incremental pull фильтрует по seq; запись жила только в ответе capture). Поймано на Ф4-смоуке (свежий reload → реестр пуст). Фикс — провести захват через `recordSyncChanges` (ledger-подпись + seq + проекция), in-place upsert по opId. Образец — `workOrderClosingService` part_status_event. Тот же класс — GOTCHAS **M6/M8** (server-запись без seq не доезжает инкрементально).
- **Фаза 4 — печатное «требование к заказчику» по двигателю (✅ сделано 2026-06-28; решения владельца 2026-06-28):** печать с карточки двигателя (контур актов, НЕ reports-пресет) — **документ-доказательство без сумм**: личный №, деталь, классификация утиль/замена, дата фиксации, статус. Источник — реестр Ф3 (`selectRequirementInstances` — только утиль/замена). Версионируемый снимок (`operations` type `repair_fund_requirement`, dedup по сигнатуре) с повторной печатью — как акты. Билдер `buildEngineRequirementHtml` (`engineInventoryPrintHtml.ts`), снимок `saveRepairFundRequirementSnapshot` (`checklistService.ts`), кнопка «🧾 Печать требования» в `RepairChecklistPanel`. **Эпик готов к релизу.**
