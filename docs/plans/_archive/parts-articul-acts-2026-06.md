## Контекст модели (разведка 2026-06-12)

- **Артикул** = EAV `article` ↔ `directory_parts.code` (зеркало `partFieldMirror.ts`). Это «сборочный номер» из списка владельца.
- Отдельные поля, НЕ артикул: `assembly_unit_number` (узел сборки, в строках актов из brand-link) и `part_number` (чертёжный/набитый номер, столбец акта).
- Строки актов: сигнатура `part_name|assembly_unit_number|part_number` (`engineInventoryRowSignature`, shared/repairChecklist.ts:250); идентичность строк по `__part_id`/`__brand_part_id`.
- Детали ↔ марки: `directory_parts.brand_links_json` = `PartSpecBrandLink[]{id, engineBrandId, assemblyUnitNumber, quantity}` — **галочек актов в линке нет** (появятся в Т4).
- Гейт дублей: `createDirectoryPart` (warehouseService.ts:2045), контракт `duplicate part exists:<uuid>` (вызывающие реюзают карточку — менять только семантику ключа, не форму).
- Engine-dedup машинерия для Т2: `engineDedupeService.ts` (normalizeLookupCompact-группы, tombstone `merged_into`, перевес операций, hourly job).
- Готовый опций-хелпер: `mapPartRowsToSearchOptions` (selectOptions.ts:92) — label=имя, hint «Арт. X», searchText с артикулом.

## Т1 — артикул как полноправный различитель ✅ (#327, 2026-06-12)

Было: code-приоритет (имя игнорировалось при заданном code) → легальный кейс владельца «Картер верхний 3301-15-30» + «Картер нижний 3301-15-30» отвергался; нормализатор расходился с поисковым. Тесты обновить + кейс картеров.
2. **Выпадашки актов с артикулом**: три option-builder'а RepairChecklistPanel (defect/completeness/inventory) → `mapPartRowsToSearchOptions` (hint «Арт.», поиск матчит артикул); meta-карты ключевать по option id, не по label (label-коллизия одноимённых деталей); резолв текущего значения строки через `__part_id` прежде label.
3. Прочие места выбора деталей уже на `mapPartRowsToSearchOptions` (Contract, EngineBrand) — проверить остальные точечно (BOM, шаблоны).

## Т2 — модуль поиска/слияния дублей номенклатуры ✅ (2026-06-12)

Построено: shared `partsDedup.ts` (группировка: exact = Т1-пара; fuzzy = похожие имена union-find'ом при совместимых артикулах — одинаковое имя с РАЗНЫМИ артикулами легально и кандидатом не считается); backend `directoryPartsDedupeService.ts` (analyze с usage-статистикой + merge: дозаполнение полей survivor, объединение brand-links по марке, перевес остатков с слиянием одинаковых локаций, движений, BOM-строк (дубль в той же BOM — гасится с конфликт-репортом), строк документов, перезапись `__part_id`/`__brand_part_id` в meta операций через recordSyncChanges, tombstone `mergedInto` в metadata + soft-delete, ledger-emits по всем sync-таблицам); роуты `GET /warehouse/parts-dedupe` + `POST /warehouse/parts-dedupe/merge`; UI `PartsDedupePage` (вкладка «Дубли номенклатуры» в группе Склад): жёсткие группы предотмечены, fuzzy решает оператор, survivor по умолчанию — самая используемая, отчёт слияния с конфликтами. Ключевой факт модели: парная номенклатура живёт под тем же UUID, что и деталь → перевес одного id покрывает всех потребителей. Auto-merge job деталям НЕ делался (осознанно: нет офлайн-генератора дублей масштаба двигателей; write-гейт Т1 + операторский экран достаточны). Гейт: survivor обязан иметь складскую карточку, если она есть у поглощаемой (клонировать нельзя — `erp_nomenclature.code` глобально уникален включая soft-deleted).

## Т4 — наследование марка → двигатель ✅ полностью (2026-06-12)

Галочки актов в brand-link (расширить `PartSpecBrandLink`: `inCompletenessAct?`, `inDefectAct?`); при создании двигателя подтягиваются с деталями. Default-наличие: акт-детали — наличие ПУСТО, прочие — проставлено. Акт претензии питается от акт-деталей без галочки наличия — проверить связку.

## Т5 — per-engine override галочек

Копия флагов в данные двигателя при создании, правка локальной копии в карточке двигателя (не трогает шаблон марки). Глобальные правки — в марке (существующий путь).

## Т6 — формы актов

Дефектовка: поле «дефектовщик» (выбор из работников) + дата; новый столбец «Номер детали» (набитый, отдельно от чертёжного). Комплектность: комиссия из трёх (нач. цеха / мастер / нач. ОТК — автоподстановка по цеху + ручной выбор, ФИО в шапку И в подписи); шапка с необязательным договором и текстом «Комиссия в составе…»; дата внизу. Оба акта: пустые поля вместо нулей при печати.
