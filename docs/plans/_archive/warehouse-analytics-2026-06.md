> Источник: brain-директива `2026-06-07-warehouse-analytics-per-shop-growth-charts` (compliance=recommend). Подход — «сперва карта, потом proposal» (probe-before-build, brain #020). Этап A (карта) выполнен на прод-данных v1.44.0 2026-06-08. Этап B (proposal + решение) — этот файл. Этап C (стройка) — по go.

## Context (зачем)

Владелец хочет аналитику движения деталей/выпуска по заводу с упором на per-цех динамику: каждый цех видит «сколько выпустил / осталось / отдал», график роста выпуска по изделиям с чекбоксами позиций и пулами-группами (все «гильзы» → один суммарный ряд). Плюс ревизия зрелости складского учёта (1С-фишки).

## Этап A — карта (на прод-данных, 2026-06-08, v1.44.0)

**Движения склада почти пусты** — 43 строки (март–май 2026), 39 простых `receipt`, 0 с `engine_id`, ноль `production_release`/`assembly`/`dismantling`. Склад как регистр производственного оборота не наполняется.

**Реальный «выпуск» живёт в жизненном цикле двигателя (EAV, тип `engine`, 1611 сущностей):**

→ time-bucket `date_trunc` по дате-атрибуту; **снапшоты не нужны** (история уже есть, brain 2b).

**Пулы:** `group_id` номенклатуры заполнен (636/637), но резолвится в 4 грубых каталожных бакета (Услуги 363 / Детали 182 / Товары 84 / Готовая продукция 4) — не «Гильза/Головка/Картер». Тонкая группа — `component_type_id` (заполнен 1/637). Пул деталей опирается на нитку parts-chain-audit (brain 2c). Пул двигателей по выпуску = `engine_brand` (готов).

**1С-фишки:** есть резерв (`reserved_qty`, 6 строк), межцех `stock_transfer` (5 док, first-class), инвентаризация (5 док), колонки `min/max_stock`. Слабо/нет: low-stock-алёрт (данных 2), неликвид/оборачиваемость, ABC. Фантом подтверждён: `erp_part_cards`=0, `erp_reg_part_usage`=0 — мёртвая схема.

## Этап B — решение (выбрано Claude по делегации владельца 2026-06-08)

**Строим MVP-1 сейчас + параллельно начинаем захват цеха (НЕ дашборд цеха).** Обоснование:
1. Строй то, что данные поддерживают; не строй дашборд над пустотой (per-цех `workshop_id` = 2/1611 → анти-паттерн 2d).
2. Захват цеха нельзя добэкфилить — начинать рано (дёшево, одно поле), дашборд над ним — потом, когда история накопится. Откладываем дашборд, не захват.
3. Форма графика = переносимое ядро (brain 2a, 3-й потребитель после setka/KARMAN) — окупается на любых «метрика во времени» (детали, payroll, снабжение).

Не конфликтует с parts-chain-audit: MVP-1 на `engine_brand`, не на component-пулах (они отложены, ждут ту нитку — слотинг brain §5).

## Re-probe 2026-06-08 (v1.46.0) — поправка к Этапу A

**Важно: Этап A считал СТРОКИ EAV-атрибутов, а не НЕПУСТЫЕ значения — наполненность дат завышена.** Пересчёт по непустым (`value_json` не в `''`/`null`/`"null"`):
- `arrival_date`: карта 1585 → реально **331** (1254 строки пусты); `shipping_date`: 1460 → **637**; `status_repaired_date`: 96 → **1**; `status_repair_started_date`: 97 → **3**.
- Полные оси (без изменений): `engine_brand_id` 1634, `is_scrap` 1460.
- **`workshop_id` (захват цеха) = 0 / 1612**; `operations.workshop_transfer` = **0** за всё время.

**Причина «0 захвата» — свежесть, не баг.** v1.45.0 (C1–C4 + reorder point) выпущен **2026-06-08 03:50** — инструменты захвата цеха в клиентах < суток. 7 цехов активны, путь сохранения (`EngineDetailsPage` → attr `workshop_id`, при смене → `workshop_transfer`) корректен. Adoption требует времени; диагностического фикса не нужно.

**Вывод probe-before-build:** узкое место — захват данных, не фичи. Дашборды по цехам / складу / low-stock / оборачиваемости строить ещё не на чем (поля пусты, инструменты захвата только что отгружены). Обеспечена данными только ось `engine_brand` + lifecycle-даты → C1 (выпуск) и C5 (брак). Остальное ждёт накопления после adoption.
