# План: единый источник истины для «цеха» (SSOT) — устранить двойную бухгалтерию

**Статус:** ✅ **ВЫПОЛНЕНО (2026-06-22)** — миграция применена и верифицирована на проде. Сценарий **B**, **Опция 1 (ссылка на справочник)**. PR #532 + прод-исполнение под гейтом. См. §Результат внизу.
**Источник:** директива brain `from-brain/2026-06-21-workshop-single-source-of-truth.md` (SHOULD/normal).
**Боль владельца:** Табель и «добавить сотрудников» берут цех из разных осей; «Цех №1» ощущается как разные объекты.

## Что показал прод-аудит (2026-06-21, read-only) — реальность ≠ теор. модель директивы

- **`directory_workshops`** — 7 канонических: «Цех №1»…«Цех №7» (codes 1–7, активны). Это SSOT-кандидат, его не трогаем.
- **Подразделения (EAV type=department)** — 12. Из них **4 = дубли цехов** (двойная бухгалтерия):

  | дубль (department-сущность) | UUID | сотр. | = канонич. цех (UUID) |
  |---|---|---:|---|
  | «Цех № 1» (пробел) | `4f9eaa2c-3e0d-4d2a-ab99-b488a768aa81` | 68 | Цех №1 `0fc8a6ff-3cb0-4d58-9fd7-759f1abfe97d` |
  | «Цех № 2» | `4757c91d-6753-4fd0-b4bd-0bfafc8c036b` | 11 | Цех №2 `6d4633fa-678d-4361-b942-d70710613728` |
  | «Цех № 3» | `d032daca-a09f-4457-ac1c-0478ac5ec393` | 1 | Цех №3 `118e28d5-e017-48b7-901f-df1dcad33444` |
  | «Цех № 4» | `65146db0-bfbb-4ece-950e-484b0407a168` | 46 | Цех №4 `60b9c1d5-a5a7-406c-9f39-6cffc15c445b` |

  Остальные 8 — настоящие оргединицы (Бухгалтерия, ИТР 12, Кадры, ОТК, ПЭО 2, Производство, Склад 1, Снабжение 2) — остаются на `department_id`.
- **Чем сотрудник привязан к цеху (135 сотрудников):** живая ось — **`department_id`** (133/135); 126 из них висят на 4 дубль-цехах. `ceh` (number) — мёртвая (1 чел.), `section_id` — 0, **`workshop_id` (UUID) на проде не существует даже как attribute_def** (UI создаёт его лениво на карточке — никто не наполнял).

### Следствия
1. Кнопка «**добавить весь цех**» на проде по факту мертва — фильтрует по `employee.workshop_id`, которого нет → 0. Операторы добавляют по подразделению → так и завелись дубль-департаменты «Цех № N». **Корень двойной бухгалтерии.**
2. UI-часть уже готова: карточка сотрудника **уже рисует `workshop_id` дропдауном** из `workshops.list()` (`EmployeeDetailsPage.tsx:1284-1293`, save `:546`). Т.е. валидируемый выбор — есть; нужен только бэкафилл истории. **Отдельная UI-правка Табеля не требуется** для корректности (после бэкафилла кнопка «весь цех» оживает; дроп-группировка по подразделению остаётся фолбэком). _(Опц. полировка: для workshop-scoped Табеля группировать пикер по цеху — не в этой нитке.)_

## Решение (Опция 1, подтверждено владельцем)

Цех живёт только в `directory_workshops`. Ось employee→цех = **`workshop_id` (text, UUID цеха) + валидируемый дропдаун** (уже в UI). Перевесить 126 сотрудников с дубль-департаментов на `workshop_id`, дубли soft-delete; office-8 остаются на `department_id`. Две чистые оси: **цех (производство)** / **подразделение (офис)**.

## Миграция — скрипт `backend-api/src/scripts/migrateWorkshopDepartmentsToCanonical.ts`

Через штатный mutation-путь (`recordSyncChanges`/change_log → синк клиентам), **не сырой SQL**. Dry-run по умолчанию, `--apply` для записи, `--actor=<username>` (по умолч. — первый superadmin). Идемпотентно.

**Шаги (expand-contract):**
1. **Expand:** `upsertAttributeDef(actor, {entityTypeId: employee, code:'workshop_id', name:'Цех', dataType:'text', sortOrder:105})` — создать деф (на проде отсутствует). Идемпотентно (resurrect-by-key).
2. Авто-карта дублей: department-сущности с именем `~ 'Цех'`, парс номера → `directory_workshops` по `code`. Abort если у дубля нет активного цеха.
3. Per дубль D→цех W:
   - сотрудники с `department_id=D` (активные): `setEntityAttribute(actor, emp, 'workshop_id', W.id)` (skip если уже =W).
   - **`detachIncomingLinksAndSoftDeleteEntity(actor, D)`** — обнуляет `department_id` у привязанных (EAV-link) + soft-delete D. (dry-run заранее печатает `getIncomingLinksForEntity(D)`.)
4. **Edge-case Табелей (⚠️):** `timesheets.department_id` — реляц. FK, **не** EAV-link → detach его не зацепит. Скрипт **abort'ит удаление того дубля, на который ссылается активный Табель**, и репортит для ручной сверки. На проде это **только Цех № 4** (`65146db0`): на нём dept-scoped Табель `eae0b224` (2026-06) **И** на каноническом Цех №4 есть workshop-scoped Табель `5cbf2134` (2026-06) — **дубль одного цеха за один месяц**. Пере-скоуп упрётся в `timesheets_workshop_period_uq`. **Решение владельца:** какой из двух июньских Табелей Цех №4 оставить (другой soft-delete), затем домигрировать Цех № 4. Цеха № 1/2/3 — без Табелей, проходят чисто.

**Verify после apply:** `workshop_id` у 126 проставлен; `department_id` у них null; 4 дубля soft-deleted (минус Цех№4 до ручной сверки); ни одного осиротевшего FK; «весь цех» наполняет (живой прогон). Цеха 5/6/7 становятся назначаемыми.

## Результат (выполнено 2026-06-22)

Применено на проде (`workshop:merge-dup-departments:apply`, actor=valstan, `MATRICA_LEDGER_DIR` явно):
- **expand:** создан `employee.workshop_id` attribute_def (на проде отсутствовал);
- **бэкафилл:** 116 активных сотрудников → канонич. цеха (Цех№1=66 / №2=11 / №3=1 / №4=38);
- **contract:** 4 дубля «Цех № N» — `detachIncomingLinksAndSoftDeleteEntity` обнулил `department_id` + soft-delete (отвязано 40+11+1+66 ссылок);
- **дубль-Табель Цех №4** (Jun-21, черновик 40 ячеек) soft-deleted; рабочий workshop-Табель Jun-18 (425 ячеек) оставлен.

**Verify (прод, read-only):** 0 активных «Цех»-департаментов; 0 orphan `department_id`-FK; `workshop_id` резолвится у 116; 8 офисных подразделений нетронуты; оба backend-сервиса `active`, `/health` 200. Конкурентная запись в живой ledger прошла безопасно — `loadIndex/loadState` читают с диска под кросс-процессным файловым advisory-локом (`ledger/src/store.ts`), стоп сервиса не понадобился (мина M15 = неверный каталог, не гонка).

### Хвосты (низкий приоритет)
- **Клиенты:** изменения (`workshop_id` + удалённые департаменты) досинкаются ledger-pull'ом при перелогине/обновлении; «Добавить весь цех» оживёт у операторов после синка. Живой UI-прогон кнопки — при случае.
- **Опц. полировка (не делалось):** для workshop-scoped Табеля группировать пикер «добавить сотрудников» по цеху (сейчас по подразделению — фолбэк; кнопка «весь цех» покрывает массовый кейс).
- **Обратимость:** дубль-Табель Цех№4 (`eae0b224`) soft-deleted — восстановим сбросом `deleted_at`, если понадобится.
