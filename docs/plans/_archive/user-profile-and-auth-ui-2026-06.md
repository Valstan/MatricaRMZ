# Персонализация и вход: серверный профиль настроек, новый экран входа, меню аккаунта

Источник: запрос владельца 2026-06-12 (редизайн окна авторизации; меню аккаунта Настройки/Смена аккаунта/Выйти; настройки интерфейса пользователя должны храниться «рядом с аккаунтом» и подгружаться на любом клиенте).

## Контекст (разведка 2026-06-12)

- Визуальный тюнинг (`UiControlSettings`: шрифты/кнопки/пресеты) **уже** хранится per-user на сервере: `/auth/ui-settings` GET/PATCH → EAV `ui_settings_json` на employee (`employeeAuthService.getEmployeeUiSettings/set`). Прецедент self-записи EAV: `PATCH /auth/profile`.
- **Не** синхронизируются (только локально): раскладка вкладок `TabsLayoutPrefs` (локальный SQLite, `settings.uiSet`), закреплённые ярлыки (`shortcuts.set`), «Мой Круг» — недавние страницы и рейтинги быстрого старта (localStorage `matrica:recent-visits:<userId>`, `matrica:history:quick-start-ratings:<userId>`).
- Экран входа `AuthPage.tsx`: MRU-логины (5, локальный json) + публичный `/auth/login-options` (login, fullName, role — должности нет).
- Кнопки аккаунта в шапке нет: клик по имени = переход в «Настройки». Компонент контекстного меню есть (`ListContextMenu`).
- Сессия одна (AuthSession в SQLite, шифруется safeStorage); мульти-сессии не держим осознанно (общие цеховые машины).

## Фаза 1 — серверный workspace-профиль `ui_profile` (эта нитка)

1. **Shared**: `userUiProfile.ts` — тип `UserUiProfile { updatedAt, tabsLayout?, shortcuts?, recentVisits?, quickStartRatings? }` + `sanitizeUserUiProfile` (ограничение длин массивов/строк, отбрасывание мусора).
2. **Backend**: AUTH_CODES `ui_profile_json` (ensure в `ensureEmployeeAuthDefs`, `serverOnly`), get/set-хелперы по образцу `ui_settings_json`; роуты GET/PATCH `/auth/ui-profile` (requireAuth, только свой); LWW-гейт по `updatedAt` (PATCH со старым штампом возвращает серверный профиль с `stale: true`).
3. **Клиент main**: `authUiProfileGet/Set` в authService (по образцу authProfileGet) + IPC + preload `auth.uiProfileGet/uiProfileSet`.
4. **Renderer App.tsx**: при логине — fetch профиля и применение в state (tabsLayout/shortcuts/recentVisits/quickStartScores; локальные хранилища дописываются существующими каналами); далее debounced (1.5 c) push изменений этих state с сигнатурой против эхо-записей. Офлайн: ошибка push глушится, локальные хранилища остаются источником до следующего изменения.
5. Распространение между клиентами: атрибут синкается стандартным EAV-sync; применяем на логине (live-подхват чужих правок mid-session — не нужен).

## Фаза 2 — новый экран входа

Двухпанельный AuthPage (макет в сессии 2026-06-12): слева вход/регистрация (Caps Lock-индикатор, глаз пароля, «запомнить меня», статус сервера), справа список пользователей с аватарами-инициалами: «Недавние на этом компьютере» (MRU + «был N дней назад») и «Все пользователи» с поиском по ФИО/должности. `/auth/login-options` дополнить должностью (`position` из EAV role). Клик по человеку → подстановка логина + фокус в пароль.

## Фаза 3 — меню аккаунта и смена аккаунта

Кнопка-имя в шапке → контекстное меню (аватар+ФИО+роль; Настройки / Смена аккаунта / Выйти). «Смена аккаунта» — модалка со списком пользователей (как панель Фазы 2) → пароль → штатный login-цикл (новая сессия, права, профиль Фазы 1 подгружается автоматически). Несколько одновременных сессий НЕ делаем (общие цеховые машины).

## Отложено / не делаем сейчас

- PIN-вход для пересменки (отдельное решение по безопасности).
- Live-применение профиля посреди сессии при правке с другой машины.
- Миграция legacy localStorage-ключей: не нужна — первый push после логина наполнит профиль текущими локальными значениями.
