## Контекст

Владельцу нужна гибкость «добавить/убрать человека в узел программы», а не грид роль×область. Легаси-роль `user` (полный доступ) — **оставить**, никого вручную не перенастраивать.

## Модель

Программа делится на **разделы**. У раздела два списка людей:
- **Наблюдатели** — видят всё в разделе, не меняют ничего.
- **Редакторы** — полный CRUD в разделе.
- **Не в списке** — раздел не существует для человека: нет плитки меню (клиент), данные не отдаются pull'ом (сервер, механика B2 per entity-type/table), запись режется write-гейтом.

Роли остаются **шаблонами для новичков** (кнопка «разложить по типовым разделам»); финальное слово — списки разделов. Личное (чат, заметки, «Мой круг») — вне модели, как сейчас.

## Хранение

EAV-атрибут на сотруднике: `section_access` = JSON `{ "<sectionId>": "editor" | "viewer" }` (нет ключа = нет доступа). Без DDL, синкается штатно с employee, зеркалится в карточку пользователя автоматически. Страница «Доступы по разделам» — агрегированное представление того же атрибута (правишь с любой стороны — источник один).

## Автозасев (день включения — поведение не меняется ни у кого)

Backfill-скрипт: каждому существующему пользователю проставить membership из его **текущей** роли: `user`/admin-тир → editor во все разделы кроме №3 и №9 (admin — включая №9); операторские роли → по их сегодняшним фактическим `*.View`/`*.Edit` (`operatorRolePermissions`). №3 засевается из текущего хардкода (ramzia=editor, glavbux=viewer). Dry-run → таблица владельцу → `--apply`.

## Фазы (каждая = свой PR под гейтами)

- **Ф0 — модель + засев (без энфорса):** `sectionAccess.ts` (defs: раздел → табы/entity-types/permissions; resolve-функции), атрибут `section_access`, backfill-скрипт, unit-тесты. Поведение 0 изменений.
- **Ф1 — UI:** страница «Доступы по разделам» (строка=раздел, две колонки фишек-людей, picker сотрудников) + зеркальная секция в карточке пользователя + группировка существующих переключателей прав по разделам. Меню-гейтинг клиентом по membership (пока только UI).
- **Ф2 — read-энфорс НА ПРИЛОЖЕНИИ** _(решение владельца 2026-07-02: серверную фильтрацию sync НЕ делаем — сохраняем модель «полная база на каждом клиенте» из разворота #063/M21; конфиденциальность прикладного уровня, не at-rest)_: section-гейт в main-процессе клиента ([`sectionGate.ts`](../../electron-app/src/main/ipc/sectionGate.ts)) оборачивает все IPC-регистрации — чтение данных раздела без membership отклоняется (channel-prefix map + entity-type map для generic `admin:*`; membership-кэш 15с). Fail-open: superadmin / незасеянные / незамапленные каналы. **Известные щели (осознанно):** `search:global` не фильтруется по разделам; `tools:` не гейтится (namespace общий для производства и снабжения); lookup-чтения `employees:list`/`workshops:list` открыты (ими живут наряды/чат/табель); сквозные связки — договоры→производство (`engine:list` в карточке договора), склад→договоры (`warehouse:lookups:get`) — при ручном сужении матрицы сужать согласованно. M2b-wipe кэша НЕ нужен (данные на клиенте остаются by design).
- **Ф3 — write-гейт + Рамзия** _(реализована)_: (а) **клиент** — явный список мутирующих IPC-каналов в `sectionGate.ts` требует editor (наблюдателю — «раздел доступен только для просмотра»); generic `admin:entities:create/setAttr/softDelete/detachLinksAndDelete` — тоже editor; (б) **сервер** — `ledgerAuthzGuard` режет ledger-write наблюдателя по shared-карте `sectionForLedgerWrite` (entity-type/operation-type/таблица → раздел; tools/структурные справочники/social-таблицы сознательно не мапятся — fail-open), для ВСЕХ ролей кроме superadmin (legacy `user` не обходит); своя карточка сотрудника пишется при любом уровне (паритет own_employee); незасеянный membership → fail-open; (в) **Рамзия** — `workOrderAccess` принимает policy из membership раздела `restricted_work_orders` (editor=владелец, viewer=читатель), собираемую из EAV на сервере (`getRestrictedWorkOrderPolicy`, кэш 15с) и на клиенте (`getRestrictedWorkOrderPolicyLocal`); пока ни у кого не засеяно — действует легаси-хардкод ramzia/glavbux.

## Границы / решения

- «Видит договор, но без сумм» — НЕ в scope (возможная будущая галка; разделная модель сознательно грубее и предсказуемее).
- Viewer-логины бухгалтерии (glavbux, tatreal, gala, alvina, ur491) автозасевом попадают наблюдателями/редакторами в Договоры/Отчёты — финансы у них не отнимаются (инвариант из H1-анализа).
- legacy-роль `user` живёт; матрицу не обходит после Ф2 (энфорс по membership, не по роли).
