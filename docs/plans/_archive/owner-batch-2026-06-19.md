## A. Детали-«инкогнито» + дубли номенклатуры — `[код готов, ждёт деплоя]`

**Прод-диагностика (2026-06-19):** «инкогнито»-деталей всего **1** — «Гильза стальная» `fa096ecf` код `303-07-22` (зеркала не было никогда). Оказалась **половиной дубля по коду**: тот же код `303-07-22` держит **другая активная** номенклатура `dc2554af` «Гильза». Старый best-effort путь молча пропустил зеркало при занятом коде → «инкогнито». Именно на этой паре владелец ловил HTTP-400 (если выживший — деталь без карточки). Расхождение 200 деталей / 83 зеркала `directory_kind='part'` — просто непротегированный kind, на ошибку не влияет.

**Сделано (PR fix/dedupe-heal-missing-nomenclature):** `mergeDirectoryParts` теперь **лечит вместо ошибки** — если у выжившего нет карточки, а у поглощаемой есть: soft-delete карточки донора (освобождает код под 0066) → вставка зеркала выжившему (клон классификации донора) → штатный репойнт. Heal-тест добавлен; гейты зелёные (284/284).
**Открыто:** деплой бэкенда (heal — серверный код, на проде заработает после релиза); затем владелец сливает пару «Гильза стальная»/«Гильза» `303-07-22` в UI — пройдёт без ошибки. Бэкофилл-скрипт сирот не нужен (их 0 после слияния).

**Симптом:** в модуле «Дубли номенклатуры» — `HTTP 400: у главной детали нет складской карточки (номенклатуры), а у поглощаемой есть`.

**Модель (подтверждено recon):** источник истины — `directory_parts`; зеркало `erp_nomenclature` (тот же UUID, `directory_kind='part'`). С #458 `createDirectoryPart` создаёт зеркало всегда. Легаси `/parts/*` → 410.

**Причины «инкогнито»:** (1) легаси-сироты до Phase 2 (directory_parts без зеркала); (2) старый best-effort путь создавал деталь, но не зеркало при занятом коде; (3) зеркало soft-deleted после прошлого мерджа → проверка дублей видит «нет карточки» (ложно). Источник ошибки: `directoryPartsDedupeService.ts:366–382`.

**Развязка от 0066:** прежний коммент «клонировать номенклатуру выжившему нельзя — `code` глобально-уникален» больше **неверен** — после 0066 индекс частичный (`WHERE deleted_at IS NULL`), soft-deleted loser не держит код → выжившему можно создать/перенести зеркало.

**План фикса:**
- A1. Прод-диагностика масштаба сирот (read-only).
- A2. `mergeDirectoryParts`: вместо ошибки — если у survivor нет активного зеркала, а у loser есть → **перенести/создать** зеркало survivor'у (id остаётся directory_parts.id; перенос ссылок и так делается). Учесть soft-deleted зеркало survivor'а (reuse/hard-delete).
- A3. Бэкофилл существующих сирот (`warehouse:backfill-orphan-part-nomenclature` + audit) — задокументировать row counts в PR.
- A4. (опц.) Чётче текст/диагностика, если останется честный кейс.
