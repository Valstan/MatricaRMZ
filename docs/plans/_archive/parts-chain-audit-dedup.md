## Источник данных
Снимок прода 2026-06-06. «partish» = nomenclature-строки, являющиеся деталями (directory_kind='part' ИЛИ есть directory_parts-twin по id/ref ИЛИ есть в BOM). Дубли = одноимённые (case-insensitive) внутри partish.
**70 строк / 29 групп.** Происхождение пустышек: импорт 2026-04-16 (код `DET-<id8>`, 0 нагрузки). Реальная нагрузка только у `Головка` (17 BOM, 100 склад) и `Гильза` (1 BOM).

## ⚠️ Gap инструмента (блокер исполнения)
`warehouse:merge-duplicate-part` (`mergeDuplicatePart.ts`) переносит **только** `contract_sections`/`contract_execution_parts` и **бросает** на прочих ссылках (`unhandled reference type`). Наши merge-строки держат **26 brand-links** (`part_engine_brand.part_id`). **Перед исполнением скрипт нужно расширить:**
1. Репойнт `part_engine_brand.part_id` `from→into` через подписанный `setEntityAttribute` (синк клиентам), с **дедупом**: если `into` уже связан с той же маркой — не плодить вторую связь, а ретайрнуть дубль-связь.
2. BOM (`erp_engine_assembly_bom_lines.component_nomenclature_id`) и склад — в нашем датасете на merge-строках = 0 (нагрузка только на keep-строках), репойнт не требуется; но добавить guard на всякий.
3. Зеркало `directory_parts` + `brand_links_json` каноничной — обновить после переноса.

## Правило выбора каноничной (keep)
Приоритет: (1) есть нагрузка (BOM/склад/движения) → (2) реальный код (не `DET-`) → (3) больше brand-links → (4) старшая. Прочие строки группы → merge (soft-delete с переносом связей).
