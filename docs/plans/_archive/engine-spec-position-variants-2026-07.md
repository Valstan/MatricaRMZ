> Задание владельца (2026-07-06): текущая «глобальная схема» BOM слишком жёсткая и костыльная. Нужна гибкая, наглядная, легко обслуживаемая модель: у каждой марки свой набор деталей (2 картера / плита / блок — по-разному), а там где к марке подходит несколько взаимозаменяемых деталей (напр. несколько поршней) — собирать нужно из **одного**, но учитывать все допустимые. ## Context — что сейчас (проверено по коду)

**Данные (PostgreSQL):**
- `erp_engine_assembly_bom` — заголовок (name, version, status `draft|active|archived`, isDefault) — [schema.ts:1217](../../backend-api/src/database/schema.ts).
- `erp_engine_assembly_bom_brand_links` — M:N BOM ↔ марки — [schema.ts:1240](../../backend-api/src/database/schema.ts).
- `erp_engine_assembly_bom_lines` — строки: `componentNomenclatureId`, `componentType` (default `other`), `qtyPerUnit`, `variantGroup`, `isRequired`, `priority`, `notes` (JSON-мета `lineKey`/`parentLineKey`). UNIQUE `(bomId, variantGroup, componentNomenclatureId, componentType)` — [schema.ts:1266](../../backend-api/src/database/schema.ts).

**Два корня жёсткости:**

1. **Глобальная схема — один скелет на все марки.** `DEFAULT_WAREHOUSE_BOM_RELATION_SCHEMA` (дерево 7 типов: engine→sleeve→{piston,jacket}→ring/head + carter + other) в [shared/warehouse.ts:822](../../shared/src/domain/warehouse.ts), хранится в `client_settings.bom_relation_schema_json` — **одно на всю программу**. Backend требует наличия **всех** активных типов схемы в каждом базовом BOM, иначе «BOM не сохранён: отсутствуют обязательные типы» — [warehouseBomService.ts:517-562](../../backend-api/src/services/warehouseBomService.ts). Это и мешает: у марки может быть 2 картера / плита / блок вместо фиксированного набора.

2. **Взаимозаменяемость выражается не тем механизмом.** «Несколько поршней, собрать из одного» изобразимо только через `variantGroup` = **вариант сборки всего двигателя** (фикстура [`seedVariantBomVerifyFixture.ts`](../../backend-api/src/scripts/seedVariantBomVerifyFixture.ts): varA→группа A, varB→группа B). Прогноз строит по одному киту на пару `марка × variantGroup` — [warehouseForecastService.ts:474-513](../../backend-api/src/services/warehouseForecastService.ts). Два независимых выбора (3 поршня × 2 гильзы) = 6 вариантов с дублированием общих строк → комбинаторный взрыв. Понятия «у позиции список запасных деталей, один основной» в модели **нет**.

**Потребители BOM (что заденет переделка):**
- **Прогноз сборки** — единственный «умный» потребитель. Читает active+default BOM, строит киты, симулирует сборку ([warehouseForecastService.ts](../../backend-api/src/services/warehouseForecastService.ts), [shared/assemblyForecast.ts](../../shared/src/domain/assemblyForecast.ts)). Дефицит агрегирует по номенклатуре.
- **Наряды на сборку BOM НЕ читают** — оператор набивает детали руками ([workOrderClosingService.ts](../../backend-api/src/services/workOrderClosingService.ts)). → малый blast radius сейчас + возможность улучшить позже.
- **Список деталей марки** ([EngineBrandDetailsPage.tsx](../../electron-app/src/renderer/src/ui/pages/EngineBrandDetailsPage.tsx)) — плоский (деталь+узел+кол-во). Основной вариант позиции уже зеркалится в деталь марки через [`bomBrandPartSync.ts`](../../backend-api/src/services/bomBrandPartSync.ts).
- Импорт-скрипты, RepairChecklistPanel, sync/ledger (`erpDto.ts`, `pullChangesSince`).

**Объём данных (оценка, уточнить read-only на проде до backfill):** ≈5–10 активных BOM, сотни строк, 0–3 variantGroup на BOM. Миграция дешёвая.

## Фаза 0 — наблюдения на живом клиенте (2026-07-06, verifier-electron, Electron 43)

Поднят локальный стенд, снята карточка BOM и живая модель через bridge. Подтвердило диагноз и добавило факты:

- **Карточка BOM = блоки-варианты как главная ось.** У TEST-BRAND два блока «A» и «B», каждый — полноценный блок со своей строкой (Тип / Компонент / Кол-во/двиг. / Быстрые связи / Обяз. / Приоритет). Чтобы выразить «VAR-A **или** VAR-B» (взаимозаменяемость), заведены ДВА целых блока-варианта. Это ровно тот механизм, который плодит комбинаторику.
- **Взрыв в проде реален:** BOM В-59 — **8 вариантов**, BOM В-84 — 5. Каждый вариант = дубль всего набора строк.
- **Один BOM навязан нескольким маркам:** BOM В-84 покрывает **9 марок** (В-84, В-84Б, В-84 ДТ, …) одним набором — различия между этими марками сейчас выразить нельзя (все обязаны быть идентичны). ⚠️ Миграционное решение: часть таких «широких» BOM, возможно, придётся расщепить по маркам, если наборы деталей реально различаются.
- **Глобальная схема — запутанный ГРАФ с циклами.** Кнопка «Глобальная схема» открывает редактор типов-связей; на dev-снапшоте узлы образуют циклы (engine→sleeve→piston→ring→piston, sleeve→engine, добавлен ad-hoc `carter`→sleeve). Строки вида «Кольцо ✕ +Поршень / Нет правил». Это конкретное доказательство костыльности схемы-графа — её и надо ретайрить в пользу свободного списка позиций.
- **Колонка «Быстрые связи» = parentLineKey-правила** («Нет правил») — механизм дерева, вынесенный в строку; сбивает с толку.
- **Трение с типом:** у деталей без «Тип компонента BOM» save отдаёт warning «не задан Тип компонента BOM» (VAR-A/B в фикстуре). Тип завязан на глобальную схему.
- Кнопки, которые в редизайне уходят/меняются: «Добавить вариант сборки (из схемы)», «Пересортировать по схеме», «Глобальная схема», «Дерево», «Техполя», «Режим: Один столбец».

**Дополнительное решение (из наблюдений):** редактор графа-схемы (`Глобальная схема` + колонка «Быстрые связи») — ретайрить; «широкие» мульти-марочные BOM — при миграции проверить на расщепление по маркам.

## Целевая модель

**Спецификация марки = свободный список позиций. Позиция = роль(метка) + количество + один-или-несколько взаимозаменяемых вариантов детали, один основной.** (Индустрия: configurable / 150% BOM + alternate/interchangeable parts; резолв 150%→100% выбором основного варианта.)

- **Набор позиций свой для каждой марки.** Добавляй/убирай свободно. Никакого обязательного глобального набора.
- **`componentType` → открытая необязательная метка-роль** из редактируемого справочника (группировка/печать/цвет), НЕ обязательный слот. Глобальная схема демонтируется из «жёсткого валидатора» в необязательный **шаблон-подсказку** («вставить типовой набор позиций»).
- **Взаимозаменяемость — внутри позиции** (список вариантов, один `isDefaultOption`). Независимые выборы независимы → нет взрыва. `variantGroup` (полный альтернативный BOM) остаётся редким верхним уровнем.
- **Обратная совместимость:** существующий BOM = каждая строка становится позицией с одним (основным) вариантом = собой. Поведение прогноза для текущих данных **не меняется**.

### Хранение — эволюция существующей таблицы (решение владельца #1)

Добавить реальные колонки в `erp_engine_assembly_bom_lines` (не JSON-в-notes — уходим от костыля #1 старого плана):
- `position_key text` — группирует строки-варианты в одну позицию (в рамках `bomId + variantGroup`).
- `position_label text` — человекочитаемое имя позиции (напр. «Картер верхний»), отдельно от имени детали (две позиции могут делить роль `carter`).
- `is_default_option boolean not null default true` — основной вариант позиции (ровно один true на позицию; forecast/сборка берут его).

`qtyPerUnit` — свойство позиции; UI держит одинаковым у всех вариантов позиции, forecast читает у основного. UNIQUE-индекс не трогаем (варианты позиции различаются `componentNomenclatureId`).

Синхронизация: обновить DTO ([shared/src/sync/erpDto.ts](../../shared/src/sync/erpDto.ts)) + зеркало SQLite-клиента ([electron-app/drizzle](../../electron-app/drizzle)). Колонки аддитивные, nullable/с дефолтом → старые клиенты в период раскатки читают строки как позиции-одиночки (forward-compat).

## Риски
- **Sync-совместимость в период раскатки:** старые клиенты без новых колонок. Аддитивность + дефолты + forward-compat в DTO (строка без position_key = позиция-одиночка).
- **exactOptionalPropertyTypes** — новые опциональные поля через conditional spread.
- **Данные вариантов на проде** могут быть нетривиальнее оценки — измерить read-only SQL до backfill (data-integrity > convenience: блокирующая проверка, не авто-фикс).
- **Ledger-подпись** всех записей BOM — как сейчас (`signAndAppendDetailed`).
- **Миграция необратимого не делает** — 1 строка→1 позиция, слияние вариантов ручное.

## Rollback
- Колонки аддитивные — при откате оставить (drop ломает клиент). Откат = client+backend к пред-версии.
- Backfill идемпотентен и не удаляет данные.
