## A. Баг: наряды выпадают в «несохранённые черновики», теряют порядковый номер

**Симптомы (со слов владельца):**
- Создал наряд → номер присвоен → распечатал → сохранил → позже наряд «пропадает» и всплывает как восстановленный черновик.
- У восстановленного наряда вместо номера — «Новый»; вручную вписать нельзя (запрещено), автоматически не пересоздаётся → наряд приходится удалять и делать заново.
- Сценарий: создаёшь наряд 1, 2, 3 — первые два вдруг выскакивают как несохранённые черновики.

**Гипотезы для проверки:**
1. Recovery-драфт (локальный черновик карточки) не инвалидируется после успешного сохранения в БД → при следующем старте всплывает stale-копия и затирает сохранённую версию (без номера, т.к. драфт снят ДО присвоения номера).
2. Deferred-create нарядов (create не пишет в БД, материализация первым update — известная особенность из профиля PC40): гонка между recovery-драфтом «до материализации» и сохранённой записью.
3. Присвоение номера идёт при update; restore-ветка открывает карточку в режиме «новый» и путь пересвоения номера заблокирован гейтом «номер руками нельзя».

**Шаги:**
1. Разбор жизненного цикла recovery-драфтов: `WorkOrderDetailsPage.tsx` + `App.tsx` (модалка «Восстановление несохранённых карточек») + `workOrderService.ts` — когда драфт пишется, когда удаляется, ключ драфта (id? temp-id?).
2. Воспроизвести в verifier-electron: создать 2–3 наряда подряд, сохранить, перезапустить клиент, посмотреть, всплывают ли драфты.
3. Fix (по результату), ожидаемо:
   - удалять recovery-драфт транзакционно после успешного сохранения;
   - restore существующего наряда — открывать как существующий (по id из БД), не как «новый»;
   - если наряд без номера восстановлен — разрешить авто-присвоение номера при сохранении (не «Новый» навечно).
4. Grep GOTCHAS по симптому перед долгой отладкой.
5. E2E-смоук: create → save → restart → нет призрачных драфтов; принудительный драфт → restore → номер сохраняется/присваивается.

## B. Чистое закрытие программы: сохранить и досинкать перед выходом

**Требование владельца:**
- При закрытии программы: несохранённые карточки (не намеренные черновики со статусом «черновик» — те не трогаем) — заставить сохранить/проверить.
- После сохранения — **дождаться push-синка на прод** и только потом закрывать окно.
- При аварийном выключении (свет/насильное закрытие) — текущее поведение с recovery-драфтами остаётся как страховка.

**Шаги:**
1. Разбор текущего on-close флоу: `before-quit`/`close` в main (`App.tsx` диалог «сохранить изменения?»), `syncService.ts` — есть ли ожидание push.
2. Реализация: on-close → сохранить открытые карточки → `sync.run()` (push pending) с прогресс-индикатором «Синхронизация…» и таймаутом (например 15–30 с; по таймауту — закрыть, драфты остаются) → очистить recovery-драфты успешно сохранённых → выход.
3. Смоук: закрытие с несинканными изменениями → окно ждёт push → на проде строка есть → при рестарте драфтов нет.

## D. Кнопки-подсказки в AI-чате (быстрые дополнения к запросу)

**Состав (предложение):**
1. **Чипы формата ответа** (добавляют хвост к промпту): «📄 в DOCX», «📊 в Excel», «📕 в PDF», «📃 текстом в чат». Ряд кнопок над полем ввода; клик — добавить/убрать (toggle), текст дописывается при отправке.
2. **Чипы-уточнения:** «за сегодня / за неделю / за месяц» (период), «кратко / подробно», «таблицей».
3. **Шаблоны запросов:** после успешно отвеченного вопроса — кнопка «Сохранить как шаблон» (название + текст). Шаблоны — в выпадашке «Мои шаблоны» у поля ввода; подстановка текста в поле (можно править перед отправкой).
4. **Популярные запросы:** список последних N **успешно отвеченных** (status=answered) вопросов — свои + общезаводские топ-повторы. Клик — подставить текст.

**Хранение:** шаблоны и счётчики — через существующий ledger/EAV-контур ai_chat (без DDL, если возможно; иначе таблица `ai_chat_templates`). Популярные — агрегат по `ai_chat_questions` (answered) на сервере, endpoint в `/ai-chat/meta`.

**Шаги:** дизайн чипов → серверная часть (шаблоны CRUD + топ-запросы) → клиент (ряд чипов, «Мои шаблоны», «Сохранить как шаблон») → смоук CDP → релиз.
