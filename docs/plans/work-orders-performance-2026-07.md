## Корень

`listWorkOrders` ради одного (дотянуть номер/марку двигателя для **сборочных** нарядов) строит **весь каталог двигателей** — `listEngines`, а тот вызывает `getEngineInventoryFlagsMap`, который читает **все 2086 актов дефектовки вместе с `meta_json` (79 МБ) и парсит каждый**. Это ~1.2 с синхронной работы main-процесса.

И вызывается это:
- на **каждую букву** в поиске списка нарядов (`refresh` пересоздаётся на каждый `patchState({query})`, дебаунса нет);
- каждые 15 с (`useLiveDataRefresh`);
- на каждый sync-пульс с `pulled > 0`;
- при каждом возврате в список.

`better-sqlite3` синхронный → пока это крутится, **любой** другой IPC (открыть карточку, сохранить, синхронизация) стоит в очереди. Отсюда «всё тормозит», а не только список.

Второй очаг — карточка: `loadRefs()` тянет 368 услуг **поштучно** (`admin.entities.get` на каждую), то есть 368 IPC round-trip'ов, и у каждого внутри — чтение сессии для проверки прав + запрос defs + 9 запросов значений. И так **при каждом открытии карточки** (справочники между карточками не переиспользуются).

## Этапы

- [x] **Ф1. Список нарядов не строит каталог двигателей.** Точечный резолв `resolveEngineLabels(db, ids)` только по тем engineId, что реально нужны (обычно единицы) вместо `listEngines`.
- [x] **Ф2. `getEngineInventoryFlagsMap` не читает 79 МБ.** Две фазы (головы без blob → `meta_json` только последних актов) + кэш разбора по `(id, updated_at)`. Чинит заодно страницу двигателей и всех прочих потребителей `listEngines`.
- [x] **Ф3. Кэш summary нарядов в main** по `(id, updatedAt)`: парсим и `recalcPayload` только изменившиеся строки; haystack поиска предпосчитан (сейчас `JSON.stringify(payload)` на каждую букву).
- [x] **Ф4. Индексы клиентской БД:** `operations(operation_type, deleted_at, updated_at)` и `operations(engine_entity_id, operation_type)`.
- [x] **Ф5. Дебаунс поиска** в списке нарядов (~250 мс).
- [x] **Ф6. Карточка:** bulk-IPC `admin:entities:listByEntityTypeWithAttrs` (1 вызов вместо 368) + кэш справочников между карточками с инвалидацией по sync-пульсу.
- [x] **Ф7. `PartDetailsPage`:** 81 `workOrders.get` на открытие карточки детали → один серверный вызов «где используется деталь».
- [x] **Ф8. Синхронизация:** проверена. Дублирующих прогонов нет: `syncManager` держит `inFlight`-гейт (параллельные синки отбиваются), интервал ≥10 с с backoff на ошибках, лишних full-pull не нашлось. Дублировалась не синхронизация, а ЧТЕНИЕ — оно и переделано выше.

## Границы

- Схема **серверной** БД не трогается: индексы добавляются в клиентскую (`electron-app/drizzle`).
- Поведение списка и карточки не меняется — только скорость. Исключение: haystack поиска строится из полей payload'а вместо `JSON.stringify` всего объекта; набор искомых полей подобран так, чтобы покрыть то же самое.
