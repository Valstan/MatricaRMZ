## Часть 1 — Переключатель «Комп/Планшет»

### Требование владельца
Крупная кнопка вверху интерфейса, нажимается пальцем, переключает Комп ↔ Планшет. Иногда к планшету подключают мышь и работают «как на компе» → **ручной** переключатель, не жёсткий автомат. Кнопка **появляется только когда клиент запущен на планшете**.

### Ключевой вывод проверки: инстинкт владельца верный, но «только на планшете» ≠ live-автодетект
Адверсариальная проверка (`verify:detect-reliability`, вердикт **PARTLY**, confidence high): **надёжного рантайм-детекта «это планшет» не существует.**
- `navigator.maxTouchPoints > 0`, `any-pointer: coarse` = «на устройстве **есть** тач» — сенсорный ноутбук рапортует так же. Это свойство наличия, не класс устройства.
- Единственный сигнал, отличающий планшет-без-мыши от ноутбука-с-мышью — `pointer: coarse` (**основной** указатель) — **ломается ровно на кейсе владельца**: воткнули мышь в планшет → основной pointer становится `fine` → live-гейт **спрятал бы кнопку**, хотя это планшет.
- Диагональ экрана в дюймах из `screen.width/DPR` не вычислить (нет EDID). Планшет 10″ 1920×1200 и масштабированный ноутбук отдают одинаковое CSS-разрешение.
- Windows-нативные сигналы класса шасси (WMI `Win32_SystemEnclosure.ChassisTypes`, `SM_CONVERTIBLESLATEMODE`, WinRT `DeviceForm`) ближе к истине, но флаки (OEM оставляют SMBIOS generic/Unknown; поза трансформера динамична) и тянут native-зависимость, которой в проекте нет.

**Решение (проверенное):** видимость кнопки гейтить **явной машинно-локальной настройкой**, а не живыми сигналами. Эвристика лишь **предлагает дефолт один раз**, дальше значение зафиксировано и не флипается при подключении мыши / смене разрешения / складывании трансформера.

### Два независимых понятия (не путать)
1. **«Это планшет» — `deviceMode`** (гейт **видимости** большой кнопки).
   Значения `auto | tablet | desktop`. Свойство **рабочего места**, задаётся один раз при развёртывании. Хранить машинно-локально (не per-user, не синкать) — новый ключ `SettingsKey.UiDeviceMode='ui.deviceMode'` в локальной sysDb, по образцу `UiTheme` (`electron-app/src/main/services/settingsStore.ts`). Плюс явная галочка в SettingsPage «Это планшет (цеховой режим)».
2. **«Текущий режим Комп/Планшет» — `uiMode`** (что **переключает** сама кнопка).
   Значения `comp | tablet`. Живой touch-layout. Тоже машинно-локальный. Дешевле всего — хук-клон `useListColumnsMode.ts` (localStorage `matrica:uiMode` + custom-event для реактивности на всех страницах без prop-drilling), либо тот же sysDb-канал `ui:prefs:get/set`. Применяется в UI через `document.documentElement.dataset.uiMode` (зеркалит существующий `dataset.theme`).

Кнопка показывается, когда **эффективный deviceMode = tablet** (`tablet` — явно; `auto` — если эвристика дала tablet). Клик по кнопке флипает `uiMode` → `dataset.uiMode` → CSS. Когда deviceMode = desktop, кнопки нет вовсе (обычный ПК бухгалтера её не видит — как и просил владелец).

### Эвристика дефолта для `deviceMode='auto'` (считается один раз, потом заморожена)
Консервативная конъюнкция (чтобы подключение мыши НЕ сбрасывало детект — берём `any-*`, а не основной указатель):
```
maxTouchPoints > 0
  AND matchMedia('(any-pointer: coarse)').matches
  AND matchMedia('(any-hover: none)').matches
  AND primaryDisplay небольшой (порог калибруется по пилотному планшету)
```
`primaryDisplay` берётся из нового IPC `window.matrica.device.info()` (main читает Electron-модуль `screen`: `{width,height,scaleFactor,rotation,touchSupport}` — дёшево, без native-зависимостей). Результат при первом запуске **фиксируется** явным `tablet|desktop` в sysDb; дальше правит только настройка. Порог «маленького экрана» откалибровать на реальном пилотном планшете (10–12″ обычно 1920×1200 при scaleFactor 1.5–2.0).

### Где живёт кнопка
Верхняя тёмно-синяя (#0b2d63) полоса рендерится компонентом [`Page`](../../electron-app/src/renderer/src/ui/layout/Page.tsx) — она на **каждой** странице и в **обеих** оболочках (v1 и v2 рендерятся внутри `Page`). Кластер кнопок справа собирается инлайном в [`App.tsx`](../../electron-app/src/renderer/src/ui/App.tsx) (`right={…}`, ~5045–5229). Прямой прецедент «кнопки-режима» уже там — переключатель оболочки `isV2` («Резиновый»/«Старый интерфейс», ~5071–5081) с персистом `switchShellVersion()`→`persistShellPrefs()`.

**Куда монтировать:** первым элементом правого кластера (перед переключателями темы), гейт `authStatus.loggedIn && deviceMode==='tablet'`. Даёт кнопку на всех страницах обеих оболочек без правки общего `Page`. Все кнопки топ-бара — общий `Button` с инлайн-оверрайдом размера (прецедент — sync ↻ 28px); большой палец-таргет делаем таким же инлайном: `minHeight ~48px, padding ~10px 18px, fontSize ~16px`, лейбл «💻 Комп» / «📱 Планшет».

## Часть 2 — Touch-адаптация интерфейса

Проверка (`verify:toggle-scope`, вердикт **PARTLY**, confidence high): один корневой класс + переопределение токенов закрывает **~70–75%** без правки компонентов; остальное — точечно в 1–2 экранах.

### Механизм (проверен, работает в проде для темы)
`document.documentElement.dataset.uiMode = 'tablet'` (зеркало существующего `dataset.theme`) + один CSS-блок в [`global.css`](../../electron-app/src/renderer/src/ui/global.css) рядом с блоком темы `warm` (который уже поднимает размеры до 30px — почти готовый шаблон планшет-режима).

### Что делает один CSS-блок (без правки компонентов)
`Input.tsx` и `Button.tsx` **на 100%** token-driven (`var(--ui-input-*)`, `var(--ui-button-*)`) → переопределение `:root`-токенов масштабирует все инпуты/кнопки/строки списков/карточки/таблицы разом:
- `--ui-input-height ≥44px`, `--ui-input-padding`, `--ui-input-font-size`;
- `--ui-button-{md,lg}-{height,font-size}`;
- `--list-row-padding-y/-x`, `--card-row-padding-*`, шрифты `--ui-*-size`.

Списки **не ломаются** при масштабировании (проверено): `.list-table` — `table-layout:auto` + колонки в `ch` (растут со шрифтом); `VirtualTable` (`@tanstack/react-virtual`) меряет строки через `measureElement`/ResizeObserver → перемеряет и рефлоует; широкие 13-колоночные списки уходят в горизонтальный скролл, а не в поломку.

Плюс глобальные правила на «сырые» контролы (тоже без JSX):
- `[data-uiMode=tablet] input[type=checkbox]{width/height 24–28px; accent-color}` — сейчас в global.css **0 правил** на чекбоксы, чистое поле. Это критично: галочка «на месте» в дефектовке ~13px — основное действие оператора.
- `[data-uiMode=tablet] .v2-menu-btn{min-height:44px; font-size; padding}` (в `.v2-menu-btn` размеры захардкожены, не через токены).

### Что требует `!important` (инлайн-стили компонентов, но всё ещё без правки кода)
- `.matrica-datepicker-input` (UnifiedDateInput — **общий** date-инпут приложения, инлайн `fontSize:13/minHeight:28`) → `min-height:44px; font-size !important`;
- ячейки календаря react-datepicker (свой size-switcher, клик-режим — отдельная touch-обвязка не нужна, только увеличить клетки).

### Что требует реальной правки кода (те самые «пара операторских экранов»)
Всё в [`RepairChecklistPanel.tsx`](../../electron-app/src/renderer/src/ui/components/RepairChecklistPanel.tsx) + shellV2:
1. **qty-степперы +/-** захардкожены инлайном `width:30 height:28` (~3600/3629) → дать `className` (напр. `.qty-stepper-btn`), чтобы CSS поднял до ~44px (сейчас инлайн-размер прячет их от CSS).
2. **`numberFitMode`** (~3548) в списке деталей двигателя в НЕ-компактном режиме **прячет степперы** — оператор печатает в узкий инпут. Завязать на планшет-флаг: в планшете форсировать видимые степперы (или numpad).
3. (Позже) настоящий numpad-оверлей для qty; упрощённое операторское меню (React-фильтр пунктов — CSS только увеличит существующие).

### Грабля координации токенов
`displayPrefs` (пользовательские размеры шрифтов из БД) применяются инлайном через `root.style.setProperty(--ui-list-font-size / --ui-card-font-size…)` в App.tsx — **инлайн бьёт CSS-класс**. Планшет-блок для этих конкретных переменных вешать либо через тот же `setProperty`-канал при включении режима, либо `!important`. Иначе планшет-масштаб не переборет пользовательские prefs.

## Часть 3 — Установка на Windows-планшет и архитектура CPU (ответ на вопрос владельца)

Проверка (`verify:arch-emulation`, вердикт **CONFIRMED**, confidence high). Текущая сборка: `build.win.target = nsis, arch:["x64"]` **только**; нативные модули `better-sqlite3` + `better-sqlite3-multiple-ciphers` 12.11.1 (`buildDependenciesFromSource:true` → **компилируются**, не prebuild); Go-watchdog `GOARCH=amd64`; клиент собирается **только** на GitHub Actions; автообновление — один канал `latest.yml`.

### Вопрос владельца по пунктам
**«Другой процессор — как это работает?»** У Windows-планшетов два класса CPU:

**① x64-планшет (Intel N100/N150/N200 — ровно то, что рекомендует родительский аппаратный план).**
Текущий инсталлятор — **под эту архитектуру**. Ставится и работает **нативно, бесшовно, с нулём изменений** в упаковке: та же nsis/x64-цель, те же нативные модули и watchdog, тот же канал автообновления. **Ничего адаптировать под установку не нужно** — как на обычном ПК.

**② ARM64-планшет (Snapdragon / Copilot+).**
Текущий x64-инсталлятор **запустится через встроенную в Windows 11 эмуляцию x64-on-ARM** — это ровно то, что владелец описал словами «ОС сама на лету переконвертирует команды процессора» (движок Prism в Win11 24H2). Работает; NSIS-инсталлятор ставится без блокеров (user-mode, без драйверов); автообновление остаётся консистентным (`process.arch='x64'` → тот же `latest.yml`, **инфраструктуру менять не нужно**). **Цена:** накладные расходы по скорости и батарее; нативный C++ `better-sqlite3` тоже идёт эмулированным (самое чувствительное к эмуляции место — крупные локальные выборки / шифрование ledger).
> Требуется именно **Windows 11** (Win10-on-ARM эмулировал только 32-бит x86). Идеально 24H2 (Prism заметно быстрее).

**«Можно ли самим написать, чтобы нативно на мобильном CPU?»**
Важное уточнение: **«нативность к процессору» — это цель компиляции, а не код в приложении.** Исходник один и тот же; меняется тулчейн сборки. Нативная win-arm64 сборка технически возможна, но это **вторая матрица сборки**, не однострочник:
- `electron-builder --arm64`;
- arm64-компиляция **обоих** `better-sqlite3`-модулей под arm64-ABI. `buildDependenciesFromSource:true` → это **кросс-компиляция** arm64-MSVC-тулчейном на x64-раннере **или** переход на native `windows-11-arm` GitHub-раннер (уже доступен);
- watchdog `GOARCH=arm64` (в коде watchdog арх не зашита — правится только env в workflow);
- отдельный `latest.yml`-канал под arm64 (electron-updater умеет и один манифест с per-arch записями, но у Windows-мультиарча есть история edge-баг → отдельный канал **рекомендуется как страховка**, не строго обязателен);
- операционная стоимость: два канала обновлений + вторая CI-матрица навсегда в поддержке.

### Итог и рекомендация
**Брать планшет на x64 (Intel N100/N150/N200)** — тогда весь вопрос архитектуры/установки **растворяется**: нативно, бесшовно, нулём работы. Это и так рекомендация родительского аппаратного плана. ARM брать только если навязан конкретной моделью/ценой/автономностью; тогда: **(1)** сначала просто **эмулировать** текущий x64-билд под Win11, проверить приемлемость скорости/батареи в цеху; **(2)** нативную arm64-сборку делать позже, **только если** эмуляция не устроит по SQLite/ledger-crypto. Touch-режим UI (Части 1–2) от выбора CPU **не зависит** — делается на любой архитектуре.

## План работ (UI-адаптация)

### Ф1a — переключатель + CSS-масштаб (основной эффект, низкий риск) — ~1–2 сессии
1. `SettingsKey.UiDeviceMode='ui.deviceMode'` в sysDb (машинно-локально, не синк) + чтение/запись в `ui:prefs:get/set`; галочка «Это планшет (цеховой режим)» в SettingsPage.
2. IPC `window.matrica.device.info()` (main `screen` → `{width,height,scaleFactor,rotation,touchSupport}`) + эвристика дефолта `auto` (заморозка на первом запуске).
3. Хук `useUiMode` (клон `useListColumnsMode`) + применение `dataset.uiMode`.
4. Большая кнопка «💻 Комп / 📱 Планшет» в правом кластере `Page` (гейт `loggedIn && deviceMode==='tablet'`).
5. CSS-блок `[data-uiMode=tablet]` в `global.css`: токены (хит-таргеты ≥44px, шрифты, паддинги) + чекбоксы + `.v2-menu-btn` + `!important` для `.matrica-datepicker-input` и календаря. Учесть конфликт с `displayPrefs`.

### Ф1b — точечные правки операторских экранов — ~1 сессия
6. qty-степперы `RepairChecklistPanel` → `className` + размер ≥44px в планшет-блоке.
7. `numberFitMode`/compact завязать на планшет-флаг (видимые степперы в списке деталей).

### Ф-later (бэклог, после обкатки пилота)
- ✅ **Настоящий numpad-оверлей для qty** — сделано 2026-07-21 (`NumpadOverlay.tsx` + тап по числовому полю в планшете открывает крупную клавиатуру; клампы совпадают с онлайн-вводом, presence-блокировки уважаются). Степперы +/- остаются для ±1.
- Упрощённое операторское меню в планшет-режиме (React-фильтр разделов под роль) — **нужен продуктовый выбор владельца** (какие разделы оставить оператору) + пилот.
- (Только если выбран ARM и эмуляция не устроит) нативная win-arm64 матрица сборки.

### Гейты (по CLAUDE.md §Autonomy)
`shared`+`ledger` build → `typecheck` + `lint` → `backend-api test` → **CDP e2e-smoke планшет-режима** (окно с touch-эмуляцией, проверить: кнопка видна при `deviceMode=tablet`, клик флипает `dataset.uiMode`, хит-таргеты выросли, чекбокс дефектовки крупный) → CI зелёный.

## Открытые вопросы (мои дефолты — по «delegate technical decisions»; правьте, если не так)
1. **Хит-таргет:** беру **≥44px** (iOS HIG) — на 48px список деталей почти наверняка уходит в горизонтальный скролл. *Дефолт: 44px.*
2. **qty-ввод в Ф1b:** крупные степперы +/- (правка размера существующих), numpad — отдельно в Ф-later. *Дефолт: степперы.*
3. **Упрощение операторского меню** (прятать бухгалтерские разделы в цеху) — **продуктовое** решение владельца, влияет на роль носителя планшета. *Дефолт: в Ф1 меню НЕ фильтруем (только увеличиваем), фильтрацию — отдельной фазой, когда решим, кто носит планшет.*
4. **Персист:** оба флага (`deviceMode`, `uiMode`) — машинно-локальные, не per-user, не синк (планшет — свойство рабочего места). *Дефолт принят.*
