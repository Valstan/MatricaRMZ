# Движение запчастей: ревизия модуля и предложения

**Status:** DRAFT — на утверждении владельца
**Created:** 2026-07-17
**Owner ask:** «рефакторинг модуля движения запчастей»: улучшения по мировой практике, удобство операторов, усиление контроля учёта деталей; найти пропущенные связи/отчёты/этапы для контроля ремонта двигателей.

## Ревизия 2026-07-17: что уже сильно (не трогаем)

- Документная модель с проведением: `draft → planned → posted → cancelled`, проведённый документ иммутабелен, оптимистичная блокировка (`warehouseService.ts:3049`, `:2649`).
- Регистры: остатки + журнал движений с hash-chain (SHA-256, env `MATRICA_STOCK_MOVEMENT_HASHCHAIN_ENABLED`) + подписанный ledger (`schema.ts:1353,1373`); журнал событий документа `erp_journal_documents`.
- min/max/«К заказу», резервы, `qty−reserved`, кнопка «Создать заявку на закупку» из остатков; прогноз сборки с BOM-китами, planned incoming, ремфонд-рекомендациями.
- Авто-документы из нарядов: `production_release` (Repair/Manufacturing), `assembly_consumption` + резерв (Assembly), `assembly_return` (rework/scrap), списание ремфонда при закрытии ремнаряда.

По мировой практике (документ-ориентированный inventory с posting, item ledger, reservations, reorder points) фундамент **уже соответствует**. Разрывы — в связках между модулями и в операторском контуре.

## Предложения (фазы, по убыванию ROI)

### Ф1. «Комплектование двигателя» (kitting) — главный затор ⭐ — ✅ отчёт отгружен (PR #251)
Отчёт-пресет `engine_kitting`: BOM марки (REST `/warehouse/assembly-bom`, вариантные группы схлопнуты) × выдано в сборку (движения по engineId, с учётом сторно) × доступно (минус резерв, без техлокаций, топ-3 «где лежит») × ремфонд × чистый дефицит; печать = пикинг-лист. Закрывает G1 в отчётной части; интерактивный экран комплектования — кандидат Ф1.2 по отзывам операторов.
**Находка гейта (G13):** клиентские таблицы `erp_engine_assembly_bom*` НЕ входят в sync-пайплайн (клиент тянет только `erp_nomenclature`/`erp_reg_stock_balance`/`erp_reg_stock_movements`, `syncService.ts:2330–2364`) → существующий отчёт «Готовность двигателей к сборке» на клиентах был слеп (`totalComponents=0` у всех). ✅ Исправлен PR #252: общий `loadBomKitForBrand` (REST + офлайн-fallback) + BOM по марке двигателя + имена деталей в дефицитах.

### Ф2. Замкнуть снабжение (G2+G4) — ✅ отгружено полностью (ядро PR #253, хвост 2026-07-17)
Кнопка **«Оформить приход на склад»** на карточке заявки (статусы accepted/fulfilled_*) → предзаполненный черновик `purchase_receipt`: строки из позиций (привезённое либо заказанное), номенклатура из `productId`, связь через `sourceRef` = «Заявка <номер> · <uuid>»; открывается карточка документа. Отчёт-пресет **`supply_receipt_gap`** «Заявки снабжения без прихода» (документы по REST — см. G14).
**Ф2-хвост ✅ (владелец одобрил поле 2026-07-17):** опц. поле **«Ожидаемая дата поставки»** в шапке заявки (`expectedDeliveryAt`) + канал «принятые заявки → будущий приход прогноза» (`loadSupplyRequestIncomingLines`: статусы accepted/fulfilled_partial, остаток = заказано − max(привезено по deliveries, оформлено приходами по sourceRef) — дедуп с Ф2-ядром; мост productId→номенклатура как в ремканале; склад default, фильтр складов уважается; просроченные даты не считаются). Нота в прогнозе «Учтены принятые заявки снабжения: N позиц.». CDP-смоук зелёный (поле → сохранение → sync → прогноз). Контрагент-поставщик в приход по-прежнему не подставляется (в заявке его нет — отдельный кандидат по отзыву).
**Находка (G14, класс G13):** локальный кэш `erp_document_headers`/`erp_document_lines` на клиентах **никем не наполняется** (документы вне sync-пайплайна, REST-only; офлайн-fallback читает пустую таблицу). Побочно: в `part_movement_journal` колонки «№ документа/Тип документа» на клиентах пусты (label-degrade, не критично). Новые отчёты по документам строить REST-first.

### Ф3. Выдача под двигатель/наряд + печать (G3+G11) — ✅ отгружено 2026-07-18
Поля «Двигатель (адресно)»/«Наряд (адресно)» в шапке `stock_issue`/`stock_writeoff` (опциональные, анти-бюрократия): `engineId` уходит в регистр движений, `workOrderId`+`workOrderNo` — в payload шапки (колонки наряда в регистре нет — для отчётов достаточно движений по engineId). Печать «Требование-накладная» (расход) / «Акт списания» из карточки (`openPrintPreview`, печатается текущее состояние экрана — черновик работает как пикинг-лист). Селекты грузятся лениво (нет прав на разделы — просто пустые).
**Находка:** «регистр уже готов» было неточно — ветки постинга `stock_issue`/`stock_writeoff` НЕ прокидывали `lineEngineId` в движение (только assembly_*-ветки). Починено в этой же фазе: `engineId` из шапки/строки теперь штампуется в `erp_reg_stock_movements`. CDP+REST смоук зелёный (раунд-трип полей → UI-селекты → печать → post → движение с engineId).

### Ф4. Сторно (G5) — ✅ отгружено 2026-07-18
`reverseWarehouseDocument`: кнопка «Сторнировать» на проведённом документе → авто-документ того же типа с docNo «СТОРНО-<номер>» и зеркальными движениями (`reversal_<type>`, direction наоборот) **по фактическим строкам регистра** (не по строкам документа). Балансы двигаются обратно с запретом минуса (сторно израсходованного прихода блокируется с подсказкой). Исходный остаётся posted + помечается `reversedBy*` (бейдж на карточке); повторное сторно и сторно-сторно запрещены. Право — существующее `movements.revert` (admin-only), REST `POST /documents/:id/reverse`, hash-chain и ledger — как при обычном проведении; журнал: событие `reversed` на исходном + `posted` на сторно. Сторно только онлайн (без offline-очереди — осознанно). CDP+REST смоук зелёный (кнопка → confirm → бейдж → reversal_issue → баланс вернулся → двойное сторно заблокировано).

### Ф5. Отчёты учёта (G7+G8)
**Оборотная ведомость ✅ отгружена 2026-07-18:** пресет `stock_turnover` по (локация × номенклатура), сальдо привязаны к текущим остаткам регистра (closing = current − движения после конца периода, opening = closing − нетто за период → легаси-остатки без истории не искажают). Фильтры: период, склады, поиск детали, «только с движением». Осталось: типизировать норму (`normPercent` в BOM-строке, миграция notes → поле) → отчёты «расход план/факт по нормам» и «план закупок по нормам» (уже в PENDING §нормы, для экономистов). ~2 сессии.

### Ф6. Контроль физики (G6+G9+G10)
Кнопка «Списать в утиль-локацию» из дефектовки (полуавто, по строкам scrap_qty; анти-бюрократия — не принудительно); переключатель «слепая инвентаризация» (факт пуст, учёт скрыт) + частичная инвентаризация по фильтру (cycle counting); отчёт-сверка «экземпляры ремфонда vs агрегатный сток». ~2 сессии.

### Quick wins (в любую фазу)
Дедуп строк документа; клиентская проверка остатка при вводе расхода (по загруженным балансам, мягкое предупреждение); ед. измерения в данные, а не placeholder.

### Отложено (не сейчас)
- QR-сканер движений — уже в PENDING (блокер: склад REST-only, offline-скан = L–XL сдвиг + железо).
- Партионный учёт (batch как регистр) — на текущем объёме не окупается.
- Слияние `SupplyToolMovementsPage` в документную модель — при явной боли.
- Выделенная роль «кладовщик» (сейчас склад под `operations.edit` у engineer/master) — вопрос владельцу при внедрении МОЛ.
