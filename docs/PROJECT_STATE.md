# Project State

Короткая оперативная память проекта для ИИ-сессий.

Цель файла:
- дать агенту быстрый актуальный контекст после синхронизации с GitHub;
- фиксировать только важные изменения, влияющие на архитектуру, эксплуатацию, release-flow, setup и правила работы;
- не раздувать историю мелочами.

## Как поддерживать этот файл
- Обновлять только при изменениях высокого уровня.
- Не писать сюда мелкие правки UI, рефакторинг без смены поведения, переименования локальных переменных, косметические исправления и одиночные багфиксы без архитектурного эффекта.
- Если старый пункт больше не актуален, не копить мусор:
  - либо переписать его в актуальном виде;
  - либо перенести в `Устаревшие решения` с короткой пометкой, что именно больше не используется.
- Держать файл коротким и практичным:
  - `Текущее состояние` — только действующие важные факты;
  - `Последние важные изменения` — обычно 5-10 пунктов максимум;
  - `Устаревшие решения` — только то, что еще может ввести в заблуждение в новой сессии.

## Порог важности для записи
Записывать изменение сюда нужно, если выполнено хотя бы одно условие:
- изменилась архитектура или основной контур работы системы;
- изменился способ синхронизации, обновления, релиза, деплоя или доступа к серверу;
- изменились обязательные правила разработки или эксплуатации;
- добавился или исчез важный модуль, сервис, документ или источник истины;
- изменилось то, из-за чего новый агент может принять неверное решение, если будет опираться на старое знание.

Не записывать, если:
- изменение локальное и очевидно из кода;
- изменение не влияет на следующие сессии;
- достаточно обновить профильный документ без добавления в общую память проекта.

## Текущее состояние
- Основная точка входа для новой ИИ-сессии: `docs/README.md`.
- Локальная разработка на Windows 11 считается штатной; рабочие шаги и ENV описаны в `docs/WINDOWS_DEVELOPMENT.md`.
- Управление VPS из Cursor идет через MCP `ssh-mcp`; рабочая конфигурация и типичные ошибки описаны в `docs/MCP_SETUP_WINDOWS.md`.
- Прод-контур backend работает в режиме dual-instance: `matricarmz-backend-primary.service` (`:3001`) + `matricarmz-backend-secondary.service` (`:3002`) за nginx upstream.
- Singleton background jobs (sync-pipeline supervisor, critical notifier, schedulers) должны запускаться только на `primary`; `secondary` обслуживает только API.
- Завершение ИИ-сессии теперь должно доводить не только git-репозиторий, но и прод-контур до актуального состояния: push в GitHub, выравнивание VPS checkout, deploy при необходимости, и выпуск нового Windows-клиентского релиза при риске несовместимости с продом.
- Рабочий контур синхронизации клиента использует ledger-эндпоинты; legacy `sync/*` не считается актуальным контуром.
- Клиентский release для автообновлений становится видимым клиенту только после публикации installer-метаданных в ledger с корректными `version`, `fileName`, `size`, `sha256`.

## Последние важные изменения
- Клиентский sync recovery усилен: если сервер требует rebuild локальной SQLite-базы из-за `schema_mismatch`, клиент теперь планирует полный relaunch вместо продолжения работы с уже закрытым DB connection.
- Раздел отчетов переведен на новый UX-контур: каталог с `Избранным` и историей последних отчетов, отдельные страницы шаблонов, а также новый экспорт отчетов в 1С XML.
- Release welcome-flow уточнен: новый приветственный текст релиза теперь должен суммировать 3 последних приветственных описания и добавлять главное описание текущей закрываемой сессии; welcome-окно клиента переведено на спокойный teleprompter-style сценарий чтения.
- Модуль нарядов усилен как базовый учетный контур: карточка наряда переведена на общий `EntityCardShell`/section-layout, quick-create сотрудников и услуг в форме доведен до дедупликации и более предсказуемой автоподстановки.
- В отчетах по нарядам добавлен бухгалтерский preset `work_order_payroll_summary`; детали фильтров и состава колонок зафиксированы в `docs/REPORTS.md`.
- Протокол завершения сессии расширен до полного production closure: GitHub push, sync VPS-репозитория, deploy, пост-проверки и decision gate по клиентскому релизу.
- Уточнен прод operational-контур: dual-backend сохранен как штатный режим, restart/deploy делается последовательно (primary -> health -> secondary).
- Telegram poll-сбои `sync pipeline` больше не должны эскалироваться как серьёзная авария по умолчанию: они демотированы, а их тренд отслеживается через тихие метрики и diagnostics.
- Поиск в `SearchSelect` / `MultiSearchSelect` стал общим UX-паттерном для отчетов и ключевых карточек: поддерживаются вторичная строка-подсказка, поиск по нескольким полям и более релевантное ранжирование.
- Шум в критических событиях снижен: plain client offline и transient Telegram bot poll ошибки больше не должны выглядеть как серьёзная sync-авария.
- Клиент-серверное взаимодействие переведено на HTTPS; использовать HTTP как актуальную норму больше нельзя.
- Release-flow уточнен для Windows-разработки: локальный релиз может создать commit/tag/push, но финальная доступность обновления для клиентов зависит от наличия installer на сервере и публикации релиза в ledger.
- Добавлена отдельная краткая инструкция по настройке MCP на Windows 11: `docs/MCP_SETUP_WINDOWS.md`.
- В `docs/README.md` введен текстовый протокол старта и завершения ИИ-сессии вместо отдельного автоматического скрипта.
- Добавлена система дедупликации карточек справочников: при вводе данных (название, артикул, цена) фоновая проверка ищет похожие сущности; при сохранении карточки с дубликатами показывается диалог с выбором — отменить сохранение, объединить данные в существующую карточку, заменить старую карточку новой или сохранить как новую. Алгоритм скоринга использует нормализацию текста (ё→е, удаление пунктуации) и многоуровневое сравнение (точное совпадение, префикс, подстрока, subsequence).
- Контекстное меню `SearchSelect` переработано: кнопка «+Создать и Вставить» закреплена внизу и всегда видна; список подсказок ограничен 5 строками без прокрутки; каждый вариант помечен бейджем источника (БД / Текущее / Авто); попап автоматически выбирает направление (вверх/вниз) чтобы не перекрывать поле ввода и не выходить за край экрана; при пустом вводе кнопка показывает подсказку «Введите название и нажмите Ctrl+Enter».
- Кнопка создания шаблона детали переименована из «Создать деталь» в «Создать новый шаблон детали» для ясности. Создание детали «на месте» из привязки в марке двигателя уже работает через `SearchSelectWithCreate` → `SearchSelect` с обновлённым контекстным меню.
- Цветовая подсветка прогресса контрактов исправлена: неисполненные контракты — белый фон с чёрными (или красными если просрочен) буквами вместо розового фона с белым текстом; частично исполненные — лёгкий прозрачный градиент поверх белого фона; полностью исполненные — синий фон с белым текстом.
- Раздел «Работы по изделиям» в карточке наряда удалён; «Работы без привязки к изделию» переименован в «Виды работ». Строка вида работ расширена новыми полями: «Номер изделия» (ручной ввод), «Номер двигателя» (выбор из списка двигателей), «Марка двигателя» (автозаполнение из выбранного двигателя). Поле «Вид работ» расширено для лучшей читаемости.
- Кнопки печати в карточке заявки («Распечатать (кратко)», «Распечатать (полно)») перемещены в `CardActionBar` между «Сохранить и выйти» и «Сброс». Компонент `CardActionBar` расширен пропами `extraActionsLeft` / `extraActionsCenter` для кастомных действий.
- Исправлена печать через предварительный просмотр: inline `onclick="window.print()"` заменён на `addEventListener('click', ...)` после `document.write()` — Chrome/Electron блокирует inline event handlers. Исправлено в `printPreview.ts` и `RepairChecklistPanel.tsx`.
- UI карточек уплотнён: вертикальные отступы уменьшены (`--ui-space-2: 4px`, `--ui-space-3: 8px`, `--ui-space-4: 10px`); `EntityCardShell` скрывает заголовок при `title=""`; `CardActionBar` получил проп `cardLabel` для отображения типа карточки («Наряд», «Заявка», «Контракт») в единой строке управления; кнопка «Сохранить» убрана из наряда (дублирует «Сохранить и выйти»); ширина карточки растянута на 100%; таблица видов работ расширена — «Вид работ» 480px, «№ изделия»/«№ двигателя» вместо полных слов.
- Статистика онлайн-времени в «Сводный отчёт по клиентам» ограничена рабочим окном 08:00–18:00 МСК (макс 10 ч/день). Сессии за пределами рабочего времени не учитываются, пересчёт происходит автоматически при каждом тике шедулера (~3 мин).

## Устаревшие решения
- Не считать актуальным предположение, что отдельный session-start скрипт обязателен для новой ИИ-сессии. Источник истины для старта и завершения сессии теперь текстовый протокол в `docs/README.md`.
- Не считать актуальным предположение, что клиент работает по HTTP. Текущий целевой контур — HTTPS.
