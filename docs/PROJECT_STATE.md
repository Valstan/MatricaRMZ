# Project State

Короткая оперативная память проекта для ИИ-сессий.

Цель файла:
- дать агенту быстрый актуальный контекст после синхронизации с GitHub;
- фиксировать только важные изменения, влияющие на архитектуру, эксплуатацию, release-flow, setup и правила работы;
- не раздувать историю мелочами.

## Как поддерживать этот файл
- Обновлять только при изменениях высокого уровня.
- Не писать сюда мелкие правки UI, рефакторинг без смены поведения, переименования локальных переменных, косметические исправления и одиночные багфиксы без архитектурного эффекта.
- Если старый пункт больше не актуален, не копить мусор:
  - либо переписать его в актуальном виде;
  - либо перенести в `Устаревшие решения` с короткой пометкой, что именно больше не используется.
- Держать файл коротким и практичным:
  - `Текущее состояние` — только действующие важные факты;
  - `Последние важные изменения` — обычно 5-10 пунктов максимум;
  - `Устаревшие решения` — только то, что еще может ввести в заблуждение в новой сессии.

## Порог важности для записи
Записывать изменение сюда нужно, если выполнено хотя бы одно условие:
- изменилась архитектура или основной контур работы системы;
- изменился способ синхронизации, обновления, релиза, деплоя или доступа к серверу;
- изменились обязательные правила разработки или эксплуатации;
- добавился или исчез важный модуль, сервис, документ или источник истины;
- изменилось то, из-за чего новый агент может принять неверное решение, если будет опираться на старое знание.

Не записывать, если:
- изменение локальное и очевидно из кода;
- изменение не влияет на следующие сессии;
- достаточно обновить профильный документ без добавления в общую память проекта.

## Текущее состояние
- Основная точка входа для новой ИИ-сессии: `CLAUDE.md` (правила проекта) + slash-команды `/start` / `/close_session` / `/reliz`. `docs/README.md` — справочный hub профильных документов, не протокол сессии.
- Локальная разработка на Windows 11 считается штатной; рабочие шаги и ENV описаны в `docs/WINDOWS_DEVELOPMENT.md`.
- Управление прод-VPS для ИИ-агента и разработчиков: **только SSH в терминале** по Host-алиасу из `%USERPROFILE%\.ssh\config` на Windows или `~/.ssh/config` (например `matricarmz`). Это штатный способ деплоя, просмотра логов и git на сервере; **отдельный MCP ssh-mcp для прода не используется**. Подробнее: `docs/README.md` (блок «Напоминание для ИИ-агентов»), `docs/WINDOWS_DEVELOPMENT.md` §8, `docs/OPERATIONS.md`, правило `.cursor/rules/production-ssh.mdc`.
- Прод-контур backend работает в режиме dual-instance: `matricarmz-backend-primary.service` (`:3001`) + `matricarmz-backend-secondary.service` (`:3002`) за nginx upstream.
- Singleton background jobs (sync-pipeline supervisor, critical notifier, schedulers) должны запускаться только на `primary`; `secondary` обслуживает только API.
- Закрытие ИИ-сессии (`/close_session`) = синхронизация всей работы на GitHub через PR-flow (ADR-0002); сессия не закрыта, пока sync-гейт не зелёный (brain #010). GitHub — источник истины между машинами. Деплой на прод и выпуск клиентского релиза — **отдельный осознанный шаг** (`/reliz` + ручной деплой по CLAUDE.md §Release process), а НЕ автоматическая часть закрытия сессии.
- Рабочий контур синхронизации клиента использует ledger-эндпоинты; legacy `sync/*` не считается актуальным контуром.
- Клиентский release для автообновлений становится видимым клиенту только после публикации installer-метаданных в ledger с корректными `version`, `fileName`, `size`, `sha256`.
- **«Цех» — единый источник истины = таблица `directory_workshops` (зафиксировано 2026-06-08).** 7 строк (коды `1`–`7`, имена «Цех №1»…«Цех №7», все активны). Канон используют: наряды (`workOrderClosingService` валидирует `workshopId`), складские локации (`warehouse_locations.workshop_id` FK), `erp_document_headers.workshop_id` FK, `workshop_repair_templates`, route `/workshops` (`workshopsService`), карточка двигателя (поле «Цех», с warehouse-analytics C2) и аналитика (`analyticsService`). Два EAV-entity-type — `workshop_ref` («Цеха») и `workshop` («Цех») — **легаси-дубли, НЕ источник истины** (см. «Устаревшие решения»). Любой новый захват/чтение «цеха» обязан идти через `directory_workshops`.
- **Прод-секреты/env (brain #008, 2026-05-30):** канонический файл — `/etc/matricarmz/matricarmz.env` (`root:valstan` 0640, вне дерева репо). Оба backend-юнита грузят его через `EnvironmentFile=`. In-tree `backend-api/.env` на проде — **симлинк** на него (для `dotenv/config` в `db:migrate` и tsx-скриптах). В репо коммитится только `backend-api/.env.example`; `.env` и `.env.bak*` в `.gitignore`. ⚠️ После `git clean -fdx` на проде пересоздать симлинк: `ln -sfn /etc/matricarmz/matricarmz.env backend-api/.env`.
- **Прод-тюнинг лимитера запросов (не секрет, но только в gitignored env; зафиксировано 2026-06-23, фон #555):** в `/etc/matricarmz/matricarmz.env` заданы `MATRICA_TRUST_PROXY_HOPS=2` (myjino edge + nginx = 2 прокси-хопа → Express считает rate-limit по реальному IP клиента, а не по IP nginx — иначе один внешний IP клал общий лимит), `MATRICA_RATE_LIMIT_AUTH=300` (окно `/auth/login`+`/register`; строгий лимитер сужен только на эти пути — #555), `MATRICA_RATE_LIMIT_GLOBAL=3000`. Поднято после 429-шторма. Значения живут только в gitignored env → записаны здесь, чтобы пережить реклон/`git clean` и были воспроизводимы будущими сессиями. Дефолты кода (если переменные не заданы) ниже — на проде обязательно выставить эти три.

## Устаревшие решения
- **EAV entity-types `workshop_ref` и `workshop` — легаси-дубли `directory_workshops`, не источник истины (ревизия 2026-06-08).** Канон — таблица `directory_workshops` (см. «Текущее состояние»). Состояние на проде (v1.44.0, read-only probe):
  - `workshop_ref` (id `0d685f89`, «Цеха») — 7 живых сущностей с **расходящимися** именами («Цех механической обработки», «Сборочный цех», «Ремонтный цех», «ОГМ», «Цех №1/2/4»). До PR #267 сидился клиентом в `masterdataClassicPreset.ts` и читался дженерик-админкой `AdminPage.tsx` (text-lookup/link-normalize `'workshop'`→`'workshop_ref'`); сидинг и редиректы сняты. Backend этот тип не знает вообще.
  - `workshop` (id `a491ca5d`, «Цех») — целевой тип link-поля `metaJson={"linkTargetTypeCode":"workshop"}` у атрибутов `engine.workshop_id` и `section.workshop_id`. **На 2026-06-08 пуст** (hard-purge, см. ниже); сам entity-type оставлен как inert-маркер кода `'workshop'`.
  - **Орфан `f5c4b717` — миграции не требовал, hard-purged 2026-06-08.** Это было единственное непустое значение `engine.workshop_id` на всём проде, и носитель — удалённый тестовый двигатель №456456; второй (и последний) носитель — тоже удалённый тест №1111 с литералом `null`. Живых данных на EAV-пути «цеха» не было. Удалены (под pg_dump-бэкап `backup-workshop-purge-20260608-*.dump`, G29-подтверждение): 2 stray `engine.workshop_id` значения + 2 soft-deleted `workshop`-сущности (`f5c4b717`, `a5f405a8`) с их `attribute_values`. Все строки были `synced`-тумбстонами → sync-безопасно; guard подтвердил 0 живых двигателей-ссылок.
  - **Резолв «цеха» переведён на канон в коде (PR #267):** снят сидинг `workshop_ref` (`masterdataClassicPreset.ts`); `AdminPage.refreshLinkOptions` резолвит link-поля с target `workshop`/`workshop_ref` на `directory_workshops` (чинит ранее пустой dropdown `section.workshop_id`); убраны text-lookup редиректы на `workshop_ref`. `linkTargetTypeCode:"workshop"` оставлен намеренно — теперь это канонический маркер, маршрутизирующий на `directory_workshops` (карточка двигателя и так спецкейсит канон с C2).
  - Остаточный inert-легаси (низкий приоритет, не источник истины): 7 живых сущностей типа `workshop_ref` на проде (расходящиеся имена «Цех механической обработки»/«ОГМ»/…) и пустые entity-types `workshop`/`workshop_ref` — можно удалить отдельной нитью. Связано с G36 (доменная группировка/дедуп, нитка `parts-chain-audit`).
