# Project State

Короткая оперативная память проекта для ИИ-сессий. **Принципы** (из чего исходить) — [`CONSTITUTION.md`](CONSTITUTION.md), этот файл — **действующие факты** архитектуры/эксплуатации (ст. 9 конституции: принцип ≠ процедура ≠ карта ≠ история).

## Как поддерживать этот файл
- Обновлять только при изменениях высокого уровня: архитектура, контур синхронизации/релиза/деплоя/доступа, обязательные правила, появление/исчезновение важного модуля или источника истины.
- Не писать сюда UI-мелочи, рефакторинги без смены поведения и одиночные багфиксы — это история `git log` / `COMPLETED.md` (ст. 10 конституции: рабочие файлы держат только открытое/действующее).
- `Текущее состояние` — только действующие важные факты; `Последние важные изменения` — **5–10 пунктов максимум**, старое выпалывается (история остаётся в git); `Устаревшие решения` — только то, что может ввести новую сессию в заблуждение.

## Текущее состояние
- Основная точка входа для новой ИИ-сессии: `CLAUDE.md` (правила проекта) + slash-команды `/start` / `/close_session` / `/reliz`. `docs/README.md` — справочный hub профильных документов, не протокол сессии.
- Локальная разработка на Windows 11 считается штатной; рабочие шаги и ENV описаны в `docs/WINDOWS_DEVELOPMENT.md`. Специфика конкретного компа (порты, пути, скиллы) — `docs/machines/<hostname>.md`.
- Управление прод-VPS: **только SSH в терминале** по Host-алиасу `matricarmz` (внешний порт `49217`, myjino-форвард на внутренний `49412`; изолированный per-machine ed25519-ключ + `IdentitiesOnly yes` — иначе fail2ban; детали/типовые грабли — `CLAUDE.md` §Prod server и §SSH tips). MCP ssh-mcp не используется. Ротация ключей — 90-дневный цикл, следующая **2026-08-21** (в `authorized_keys` прода: ключи PC40 2026-05-23, RMZ4VAL 2026-06-02, машина-2 2026-05-22 + 1 строка без комментария неустановленного происхождения — не трогаем).
- Прод-контур backend — dual-instance: `matricarmz-backend-primary.service` (`:3001`) + `matricarmz-backend-secondary.service` (`:3002`) за nginx upstream. Singleton background jobs (sync-pipeline supervisor, critical notifier, schedulers) — только на `primary`; `secondary` — только API. Restart/deploy — последовательно (primary → health → secondary).
- Закрытие ИИ-сессии (`/close_session`) = синхронизация всей работы на GitHub через PR-flow (ADR-0002); сессия не закрыта, пока sync-гейт не зелёный (brain #010). Деплой на прод и клиентский релиз — **отдельный осознанный шаг** (`/reliz` + CLAUDE.md §Release process), НЕ автоматическая часть закрытия.
- Клиент-серверное взаимодействие — только HTTPS. Рабочий контур синхронизации клиента — ledger-эндпоинты; legacy `sync/*` не актуален. Клиентский релиз становится видимым автообновлению только после публикации installer-метаданных в ledger (`version`, `fileName`, `size`, `sha256`).
- **«Цех» — единый источник истины = таблица `directory_workshops` (2026-06-08).** 7 строк (коды `1`–`7`). Канон используют: наряды (`workOrderClosingService`), складские локации (FK), `erp_document_headers`, `workshop_repair_templates`, route `/workshops`, карточка двигателя, аналитика. EAV entity-types `workshop_ref`/`workshop` — легаси-дубли (см. «Устаревшие решения»). Любой новый захват/чтение «цеха» обязан идти через `directory_workshops`.
- **Прод-секреты/env (brain #008):** канонический файл — `/etc/matricarmz/matricarmz.env` (`root:valstan` 0640, вне дерева репо), оба юнита грузят через `EnvironmentFile=`. In-tree `backend-api/.env` на проде — **симлинк** на него. В репо только `.env.example`. ⚠️ После `git clean -fdx` на проде пересоздать симлинк: `ln -sfn /etc/matricarmz/matricarmz.env backend-api/.env`.
- **Прод-тюнинг лимитера (только в gitignored env, 2026-06-23, фон #555):** `MATRICA_TRUST_PROXY_HOPS=2` (myjino edge + nginx → rate-limit по реальному IP), `MATRICA_RATE_LIMIT_AUTH=300`, `MATRICA_RATE_LIMIT_GLOBAL=3000`. Зафиксировано здесь, чтобы пережить реклон: на проде эти три переменные обязательны.

## Устаревшие решения
- **EAV entity-types `workshop_ref` и `workshop` — легаси-дубли `directory_workshops`, не источник истины (2026-06-08).** `workshop_ref` («Цеха», 7 живых сущностей с расходящимися именами) — сидинг и text-lookup редиректы сняты (PR #267), backend тип не знает. `workshop` («Цех») — пуст (hard-purge единственного орфана под бэкап, G29-подтверждение); оставлен как inert-маркер: `linkTargetTypeCode:"workshop"` теперь канонический маркер, маршрутизирующий на `directory_workshops`. Остаточный inert-легаси (7 сущностей `workshop_ref` + пустые типы) — можно снести отдельной нитью.
- Не считать актуальным **MCP ssh-mcp** / `MCP_SETUP_WINDOWS.md` для прод-VPS — контур снят; только SSH.
- Не считать актуальным «текстовый протокол старта/завершения сессии в `docs/README.md`» — источник истины теперь slash-команды `/start` / `/close_session` / `/reliz` (+ `CLAUDE.md`); README — справочный hub.
- Не считать актуальным предположение, что клиент работает по HTTP — только HTTPS.
- Не считать актуальным, что деталь отображается «зеркальной строкой» `erp_nomenclature` с серверной синхронизацией — зеркало удалено (Phase 2 Stage F); деталь = сама строка `erp_nomenclature` (тот же UUID, `spec_json.source === "part"`), spec-поля в `directory_parts`.
