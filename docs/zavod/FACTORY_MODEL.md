## 1. Метазадача программы ✅ (подтверждено владельцем 2026-06-12)

Завод (РМЗ, ремонт дизельных двигателей) реорганизует работу бригад: сейчас каждая работает разрозненно, отчётность не консолидируется. Программа MatricaRMZ строит **единую нитку ремонта двигателя** — от приёмки через разборку/дефектовку/изготовление/сборку до отгрузки и начисления зарплат бригадам. Отдел плановой проектной деятельности (ППО) ведёт оптимизацию процессов.

Цели реорганизации (словами владельца): прозрачность для управления, скорость ремонта, честная зарплата, меньше потерь деталей, повышенный контроль — чтобы управлять производством и **быстро/гибко менять объём выпуска и направление движения**. Внедрение — постоянный процесс: программа написана, теперь внедряется «на земле» с корректировкой замыслов под реальные процессы цехов (доклад ППО, 10 июня 2026).

## 2. Модули программы 📝 (черновик из кода, 2026-06-12)

Что уже есть в программе и что каждый модуль делает:

| Модуль | Что делает для производства |
|---|---|
| **Двигатели** | Картотека двигателей в ремонте: номер, марка, заказчик, фазы ремонта (приёмка→разборка→…→отгрузка). Защита от дублей (один физический двигатель = одна карточка, даже при офлайн-работе двух операторов). |
| **Акты двигателя** | Акт комплектности (приёмка: что пришло/чего не хватает), акт дефектовки (что годно/ремонт/утиль/заменить), акт претензии заказчику (недостача + дефекты ветки «заказчик»). Версионирование: каждая печать сохраняет снимок, перепечатка любой версии. |
| **Ветки восполнения** | У каждой дефектной детали — кто восполняет: Заказчик / Свой ремонт / Закупка. «Свой ремонт» собирается в ремонтный наряд, «Закупка» — в заявку снабжению, «Заказчик» — в акт претензии. |
| **Наряды** | 4 типа: обычный, ремонтный, сборочный, изготовление. Закрытие наряда с подписями, ценообразование услуг, связь с зарплатой исполнителей. |
| **Прогноз сборки** | На 7 дней: хватит ли деталей собрать двигатели; видит открытые ремнаряды как ожидаемый приход, исключает утиль/ремфонд/«в сборке». |
| **Склад** | Остатки, документы прихода/расхода/перемещения, инвентаризация, локации (склады/цеха). Номенклатура — единая картотека деталей/материалов. |
| **Снабжение** | Заявки в снабжение (черновик → одобрение директора → требование), услуги сторонних организаций с фильтром по маркам двигателей. |
| **BOM (спецификации)** | Состав двигателя по маркам: какие детали в какой сборке, варианты комплектации. Основа прогноза и актов. |
| **Работники** | Картотека сотрудников, логины, роли/права, подразделения (цеха). Подписанты в актах и нарядах — отсюда. |
| **Отчёты** | Печатные/экранные отчёты: прогноз, зарплатные ведомости, ревизия склада и др. |
| **Синхронизация** | Каждый рабочий компьютер работает офлайн со своей копией данных; сервер сводит изменения. Обновления программы раздаются по заводской сети (LAN), без интернета на каждом месте. |
| **Чат и заметки** | Внутренние сообщения и заметки между операторами. |
| **Аудит/диагностика** | Кто что менял, критические события (с уведомлением в Telegram), здоровье всех установленных клиентов. |
| **Веб-админка** | Управление пользователями/правами и просмотр данных через браузер (без установки программы). |

## 3. Процессы «на земле» ✅ (интервью с владельцем, 2026-06-12)

**Структура.** Работают цеха 1, 2, 3, 4; в них ~8–10 разрозненных бригад, которые перекидываются между работами. Конвейера нет — **обезличенный ремонт**: приёмка → комплектовка → разборка → мойка → дефектовка, после разборки детали обезличиваются и уходят в общий ремфонд; из «кучи» берут в ремонт, готовое складывают в фонд готовой продукции, оттуда комплектуют сборку.

**Передача между бригадами** — устная; каждая бригада ведёт свою документацию (тетради, листочки, таблички), сдаёт её в разные места, друг другу не передаёт; сквозной документации нет. Запущены бумажные маршрутные листы (идея 002) как первый сквозной документ.

**Главный затор** — начало сборки: подбор совместимых деталей из обезличенного фонда вручную, поиск «что есть» по цехам. Дефектовка тоже плавает (3 дня … месяц), но это меньшая боль. Целевое решение — автокомплектование по BOM (идея 007).

**Отчётность и зарплата.** Наряды пишутся на бумажках свободной формы → оператор перебивает в Excel → бухгалтерия вручную собирает зарплату. Оплата: 1 цех — сдельщина, 4 цех — повременка, 3 цех — смешанно. ПЭО отчитывается по часовым тарифам, цеха работают по сдельным расценкам — конвертация вручную (идея 006). Виды работ называются в цехах и ПЭО по-разному — нужна унификация по образцу номенклатуры.

Покрытие программой: акты приёмки/комплектности/дефектовки ✅; наряды есть, но нужна универсализация под зарплату и учёт ремонта ⚠️; маршрутный лист сборки ⛔ (бумага); конвертация сдельщина↔нормо-часы ⛔; автокомплектование сборки ⛔ (фундамент — BOM+прогноз — есть).

**Снимок потока (со слов владельца, обход цехов 2026-06-24):** вход ~1 двигатель/день (было 4); ~70 движков лежат полуразобранными (по картерам); 2 линии сборки (было 3); обкатка — 4 бокса, 3 человека (не узкое место). Бригады дефектовки исторически нет — дефектуют «на горячую». **Узкое место — начало сборки** (ручной подбор из обезличенного фонда), усугублено тем, что мастера прячут готовые детали «про запас» из-за сдельщины (делают 3 вместо 6) → программа видит ложный ноль; второе — реактивное снабжение материалов. Разбор надиктовки → идеи 007 (MVP «наряд→наличие→прогноз»), 008–010. См. [INDEX](INDEX.md).
