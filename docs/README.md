# MatricaRMZ Docs

Документация приведена к рабочему минимуму, чтобы новая сессия разработки (в том числе с ИИ-агентом) быстро входила в контекст без поиска по лишним файлам.

## Протокол ИИ-сессии
Этот файл предназначен как единая точка входа для ИИ-агента в начале и в конце каждой сессии разработки.

Если пользователь пишет:
- `Начни сессию разработки с помощью файла docs/README.md`
- `Заверши сессию разработки с помощью файла docs/README.md`

агент должен выполнить инструкции ниже.

## Старт Сессии
Порядок действий для ИИ-агента в начале новой сессии:

1. Сначала прочитать этот файл полностью.
2. До любого изучения кода проверить состояние git:
   - `git status --branch --short`
   - `git fetch origin --prune`
3. Если локальное дерево грязное (`modified`, `untracked`, `staged`) или ветка разошлась с `origin/main`, не делать destructive-действий и не затирать изменения.
   - Сначала кратко сообщить пользователю, что найдено.
   - Если нужны commit / stash / rebase / merge с риском конфликта, действовать осторожно и не терять локальные наработки.
4. Если безопасно синхронизироваться, привести проект к актуальному состоянию относительно `origin/main`.
   - Предпочтительно: `git pull --rebase origin main`, если это не уничтожает локальные изменения.
   - Если есть сомнение или конфликт, остановиться и объяснить пользователю ситуацию.
5. После синхронизации заново прочитать `docs/README.md`, потому что инструкция могла измениться на GitHub.
6. Затем обязательно прочитать `docs/PROJECT_STATE.md` как краткую актуальную память проекта.
   - Это источник важных изменений высокого уровня.
   - Если файл противоречит старым знаниям агента, приоритет у текущего содержимого файла.
7. Если после sync пришли новые коммиты, быстро обновить контекст:
   - посмотреть последние коммиты;
   - посмотреть список изменённых файлов;
   - прочитать релевантные docs, если они менялись;
   - если менялся `docs/PROJECT_STATE.md`, опираться на него как на краткую актуальную сводку.
8. Затем перейти к изучению проекта по ссылкам из раздела ниже.
9. После этого явно сообщить пользователю, что сессия синхронизирована и агент готов к работе.

Ожидаемая финальная фраза на старте:
- `Проект синхронизирован с GitHub, документация перечитана, к новой сессии разработки готов.`

## Завершение Сессии
Порядок действий для ИИ-агента в конце сессии, если пользователь просит завершить сессию через этот файл:

1. Проверить, что изменилось в рабочем дереве:
   - `git status --short`
   - `git diff --stat`
2. Убедиться, что важные изменения сохранены:
   - код;
   - документация;
   - инструкции по настройке и эксплуатации;
   - release/update-related изменения, если они были в задаче.
3. Если в сессии менялись архитектура, ENV, релизный поток, MCP, деплой, sync, UI-правила или эксплуатационные шаги, обновить документацию в этой же сессии.
4. Отдельно проверить `docs/PROJECT_STATE.md`:
   - если изменение высокого уровня, добавить или обновить короткую запись;
   - если старое знание больше не актуально, переписать его или пометить в `Устаревшие решения`;
   - не записывать туда мелкие локальные правки без долгосрочного эффекта.
5. Перед push осторожно синхронизироваться с GitHub ещё раз:
   - `git fetch origin --prune`
   - если нужно, аккуратно подтянуть `origin/main`, не потеряв локальные изменения.
6. Если пользователь явно завершает сессию, это считается разрешением не только сохранить результат в GitHub, но и довести прод до актуального состояния:
   - сделать осмысленный commit;
   - запушить изменения в `origin/main`, если нет отдельной инструкции работать через branch/PR.
7. После push обязательно синхронизировать VPS-репозиторий с текущим `origin/main` / только что запушенным commit:
   - `git fetch origin --prune` на сервере;
   - `git pull --ff-only origin main` или другой безопасный способ вывести серверный checkout ровно на актуальный commit;
   - убедиться, что серверный `HEAD` совпадает с запушенным commit и серверный checkout не отстаёт от GitHub.
8. Затем принять решение по деплою:
   - если менялись `backend-api`, `shared`, `web-admin`, runtime-скрипты, lockfile, миграции, nginx/systemd/env-related файлы или что-либо ещё, влияющее на прод-контур, выполнить полный deploy на сервере;
   - если менялись зависимости, выполнить `pnpm install`/аналогичную установку на сервере;
   - если менялись schema/DDL/migrations, выполнить миграции перед перезапуском сервисов;
   - если менялись backend/shared/web-admin, выполнить сборку нужных пакетов;
   - если используются dual-backend инстансы, перезапускать последовательно: `primary -> health -> secondary`, не рестартовать оба одновременно по умолчанию;
   - если менялись nginx/systemd/env/операционные конфиги, применить их и проверить, что сервисы поднялись корректно.
9. Затем отдельно принять решение по клиентскому релизу:
   - если изменения могут нарушить совместимость текущего Windows-клиента с продом, меняют клиентский runtime/update-flow/sync/auth/IPC/shared-контракты или должны попасть пользователям как новая поставка, выпустить новый клиентский релиз по инструкции из `docs/RELEASE.md`;
   - если есть сомнение в обратной совместимости, считать это основанием для нового релиза;
   - после релиза проверить наличие installer-артефакта, корректность `updates/status` и публикацию релиза в ledger с валидными `version`, `fileName`, `size`, `sha256`.
10. После sync/deploy/release обязательно проверить прод:
   - рабочее дерево на сервере в актуальном commit;
   - backend-сервисы `active`, health endpoints отвечают;
   - нужные порты слушаются, nginx/systemd не в аварии;
   - нет свежего потока критичных ошибок в логах/critical events после деплоя;
   - если менялся update/release-контур, проверить `updates/status`, доступность релизных метаданных и, при необходимости, latest release в ledger;
   - если менялись миграции/БД, убедиться, что backend стартовал без migration/schema ошибок.
11. После push / deploy / release проверить:
   - рабочее дерево чистое;
   - локальная ветка синхронизирована с `origin/main`.
12. В финале кратко сообщить:
   - что сохранено;
   - какой commit/push сделан;
   - синхронизирован ли сервер и что именно было задеплоено;
   - выпускался ли клиентский релиз;
   - есть ли что-то несделанное или рискованное.

Ожидаемая финальная фраза при завершении:
- `Сессия завершена: изменения сохранены в GitHub, прод-сервер синхронизирован, проект и документация приведены к актуальному состоянию.`

## Правило поддержки актуальности docs
- Обязательная политика обновления документации при изменениях в коде: [`DOCUMENTATION_POLICY.md`](DOCUMENTATION_POLICY.md)
- Коротко: любой значимый change в архитектуре, update-flow, отчетах, ENV, скриптах или эксплуатационных шагах должен сопровождаться обновлением документации в той же сессии/PR.
- Для межсессионной памяти агента использовать краткий файл [`PROJECT_STATE.md`](PROJECT_STATE.md), а не раздувать `README.md` историческим шумом.

## Базовые инварианты системы
- Синхронизация клиента только через ledger-эндпоинты: `POST /ledger/tx/submit`, `GET /ledger/state/changes`.
- Legacy `sync/*` не используется в рабочем контуре.
- Релиз клиента для автообновлений должен быть опубликован в ledger с корректными `version`, `fileName`, `size`, `sha256`.
- Секреты (`.env`, токены, ключи) не храним в репозитории.
